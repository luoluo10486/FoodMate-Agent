# FoodMate 双运行时内部契约 V1

版本：v1.0 目标契约

维护基线：2026-07-11

对应架构：[架构总览](../架构/架构总览.md)、[Python 智能体运行时设计](../架构/Python智能体运行时设计.md)

对应数据设计：[数据库设计](../数据/数据库设计.md)、[V2 双运行时迁移设计](../数据/V2双运行时迁移设计.md)

文档定位：本文是 Java 业务控制面与 Python Agent Runtime 的内部 HTTP/JSON 契约目标。当前 Java Runtime Client、Python Runtime、内部 DTO 和 V2 持久化均尚未实现；本文不能作为“已经联通”的证明。

## 1. 权威、命名与传输

- Java 是业务数据、`AgentRun`、审计、工具注册与执行、SQL Guard 与 SQL 执行的唯一权威。
- Python 只负责 Agent 推理、编排、RAG、模型调用、结果质量校验和回答组装，不持有业务库写凭据。
- 所有跨运行时消息统一使用 `run_id`。数据库映射固定为 `run_id -> agent_runs.agent_run_id`；本文不把 `agent_run_id` 作为消息字段。
- V1 使用内部 HTTPS + JSON；请求 `Content-Type` 为 `application/json`，字符集为 UTF-8。
- 所有 ID 在 JSON 中使用字符串；时间使用带 `Z` 的 RFC 3339 UTC；字段名使用 `snake_case`。
- Header 固定包含 `Authorization: Bearer <service-jwt>`、`X-Contract-Version: v1`、`X-Request-Id` 和 W3C `traceparent`。

## 2. 通用字段

| 字段 | 类型 | 必填 | 规则 |
|---|---|---:|---|
| `schema_version` | string | 是 | 固定 `v1`；与 `X-Contract-Version` 一致 |
| `run_id` | string | 是 | Java 创建的运行 ID；映射 V1 `agent_runs.agent_run_id` |
| `request_id` | string | 是 | 单次内部 HTTP 请求标识，不作为业务幂等键 |
| `trace_id` | string | 是 | 跨运行时 Trace 关联标识，不作为幂等键 |
| `dispatch_id` | string | 按类型 | 一次 Run 派发标识；同一派发的网络重试保持不变 |
| `attempt` | integer | 按类型 | 同一 `run_id` 的派发代次，从 1 开始；重新派发才递增 |
| `deadline_at` | string | 按类型 | 请求绝对截止时间；过期后不得创建新副作用 |
| `invocation_id` | string | Tool/SQL | 一次逻辑 Tool 或 SQL invocation 的稳定 ID |
| `idempotency_key` | string | Tool/SQL | Java 执行面的幂等键；重试保持不变 |

`request_id` 可随 HTTP 重试变化；`dispatch_id`、`invocation_id`、`idempotency_key` 不得随同一逻辑操作的网络重试变化。

```json
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "deadline_at": "2026-07-11T12:00:00Z",
  "message": {
    "message_id": "1912345678901234568",
    "content": "分析我最近一周蛋白质摄入",
    "attachments": []
  },
  "authorized_context": {
    "session_id": "1912345678901234561",
    "timezone": "Asia/Shanghai",
    "locale": "zh-CN",
    "knowledge_scope_ref": "ks_01J2Y8QB",
    "tool_contract_version": "v1",
    "schema_catalog_version": "catalog_v3"
  },
  "runtime_options": {
    "prompt_set_version": "foodmate-2026-07-11.1",
    "max_steps": 12,
    "stream_answer": true
  }
}
```

必填字段：`schema_version`、`run_id`、`dispatch_id`、`attempt`、`request_id`、`trace_id`、`deadline_at`、`message.message_id`、`message.content`、`authorized_context.session_id`、`authorized_context.timezone`、`authorized_context.tool_contract_version`、`runtime_options.prompt_set_version`、`runtime_options.max_steps`、`runtime_options.stream_answer`。

幂等键：`dispatch_id`。Java 对同一 `run_id + attempt` 必须复用同一 `dispatch_id`；Python 收到相同 `dispatch_id` 且规范化请求摘要一致时返回原接受结果或继续原执行，请求摘要不一致时返回 `RUNTIME_IDEMPOTENCY_CONFLICT`。重新派发使用新 `dispatch_id` 且 `attempt = previous + 1`。

## 4. RunEvent

方向：Python -> Java。用途：回传单次 dispatch 的有序运行事实。

```json
{
  "schema_version": "v1",
  "run_id": "1912345678901234569",
  "dispatch_id": "dsp_01J2Y8Q9V7J8D2M4K5N6P7R8S9",
  "attempt": 1,
  "event_id": "evt_01J2Y8QF0M1N2P3R4S5T6V7W8X",
  "event_seq": 4,
  "request_id": "req_evt_01J2Y8QG",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "occurred_at": "2026-07-11T11:59:10Z",
  "event_type": "run.planned",
  "payload": {
    "status": "planning",
    "plan_version": "plan_01",
    "step_count": 3,
    "prompt_versions": {
      "router": "router-3",
      "planner": "planner-5"
    }
  }
}
```

必填字段：`schema_version`、`run_id`、`dispatch_id`、`attempt`、`event_id`、`event_seq`、`request_id`、`trace_id`、`occurred_at`、`event_type`、`payload`。`payload` 可为空对象，但不能省略。RunEvent 不携带 `deadline_at`。

幂等键：数据库/消费端使用 `(run_id,event_id)`；`event_id` 全局唯一更佳，但 V1 保证至少在 `run_id` 内唯一。事件重放必须保持原 `event_id`、`event_seq`、`occurred_at` 和业务 payload。

### 4.1 event_seq

- 作用域是 `dispatch_id`，从 1 开始严格递增 1。
- 一个 dispatch 只能有一个序号分配者；并发节点先汇入有序事件出口。
- Java 先按 `(run_id,event_id)` 去重，再比较 `event_seq`。
- `event_seq = last_event_seq + 1` 时接受、落 inbox、推进状态并更新 `last_event_seq`。
- `event_seq <= last_event_seq` 且不是已知重复 `event_id` 时返回 `RUNTIME_EVENT_OUT_OF_ORDER`，不得回退状态。
- `event_seq > last_event_seq + 1` 时返回 `RUNTIME_EVENT_GAP`，暂停该 dispatch 的后续状态推进并请求从缺口重放。
- 终态事件一旦被 Java 接受，后续事件只可用于审计拒绝，不得改变终态。

### 4.2 event_type 与状态

| `event_type` | 建议 payload `status` | 说明 |
|---|---|---|
| `run.accepted` | `queued` | Python 接受 dispatch |
| `run.routed` | `routed` | 路由完成 |
| `run.clarification_requested` | `waiting_user` | 等待用户补充 |
| `run.planned` | `planning` | 计划已形成 |
| `run.retrieval_started` | `retrieving` | 开始检索 |
| `run.retrieval_finished` | `retrieving` | 检索完成，状态可暂不迁移 |
| `run.tool_started` | `executing` | Java 已接受 Tool invocation 后报告 |
| `run.tool_finished` | `executing` | Tool 结果已观察 |
| `run.answer_stream` | `validating` 或省略 | 增量回答；不得作为业务终态 |
| `run.cancel_acknowledged` | 省略 | Python 已停止继续调度；Java 决定最终取消 |
| `run.completed` | `completed` | 终态 |
| `run.failed` | `failed` | 终态 |
| `run.cancelled` | `cancelled` | 可由 Python建议；Java 首次合法终态裁决 |

数据库 V1 状态仅允许 `queued/routed/waiting_user/planning/retrieving/executing/validating/completed/failed/cancelled`，没有 `cancelling`。内部事件不得要求 Java 写入 V1 不存在的状态。

## 5. CancelCommand

方向：Java -> Python。用途：请求停止指定 dispatch 的后续调度。

```json
{
  "schema_version": "v1",
  "run_id": "1912345678901234569",
  "dispatch_id": "dsp_01J2Y8Q9V7J8D2M4K5N6P7R8S9",
  "attempt": 1,
  "cancel_id": "can_01J2Y8QK1M2N3P4R5S6T7V8W9X",
  "request_id": "req_cancel_01J2Y8QM",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "deadline_at": "2026-07-11T12:00:00Z",
  "reason": "user_requested",
  "requested_at": "2026-07-11T11:59:20Z"
}
```

必填字段：全部字段。`reason` 允许 `user_requested`、`deadline_exceeded`、`operator_requested`、`superseded`、`system_shutdown`。

幂等键：`cancel_id`。同一取消意图重试保持 `cancel_id` 不变；Python 重复接收时不得重复创建终态或跳号，返回已有接受结果，并按正常 `event_seq` 规则重放原 `run.cancel_acknowledged`。

方向：Python -> Java。用途：提出工具调用，不代表已获授权或已执行。

```json
{
  "schema_version": "v1",
  "run_id": "1912345678901234569",
  "dispatch_id": "dsp_01J2Y8Q9V7J8D2M4K5N6P7R8S9",
  "attempt": 1,
  "invocation_id": "tool_inv_01J2Y8QR1M2N3P4R5S6T7V8W9X",
  "request_id": "req_tool_01J2Y8QS",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "idempotency_key": "tool:1912345678901234569:food-log:01",
  "deadline_at": "2026-07-11T12:00:00Z",
  "tool_name": "food_log_writer",
  "tool_version": "1.0.0",
  "input": {
    "meal_time": "2026-07-11T12:30:00+08:00",
    "items": [
      { "name": "鸡胸肉", "amount": 200, "unit": "g" }
    ]
  },
  "context": {
    "session_id": "1912345678901234561",
    "confirmation_ref": "confirm_01J2Y8QT"
  }
}
```

必填字段：`schema_version`、`run_id`、`dispatch_id`、`attempt`、`invocation_id`、`request_id`、`trace_id`、`idempotency_key`、`deadline_at`、`tool_name`、`tool_version`、`input`、`context.session_id`。需要确认的工具还必须有 `context.confirmation_ref`。

幂等键：`invocation_id` 标识逻辑调用，`idempotency_key` 保护 Java 副作用。相同 invocation 重试必须保持二者和规范化 input 摘要不变；不一致返回 `TOOL_IDEMPOTENCY_CONFLICT`。Java 必须从 `run_id` 派生用户、租户、角色和 scope，不信任 proposal 中的身份字段。

## 7. ToolResult

方向：Java -> Python。用途：返回策略裁决和确定性执行结果。

```json
{
  "schema_version": "v1",
  "run_id": "1912345678901234569",
  "dispatch_id": "dsp_01J2Y8Q9V7J8D2M4K5N6P7R8S9",
  "attempt": 1,
  "invocation_id": "tool_inv_01J2Y8QR1M2N3P4R5S6T7V8W9X",
  "request_id": "req_tool_result_01J2Y8QV",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "status": "success",
  "data": {
    "food_log_id": "1912345678901234573",
    "status": "saved"
  },
  "error": null,
  "meta": {
    "tool_call_id": "1912345678901234564",
    "latency_ms": 182,
    "replayed": false
  }
}
```

必填字段：除 `data` 与 `error` 按状态互斥外，其余字段全部必填。`status` 允许 `success`、`failed`、`denied`、`confirmation_required`、`timeout`、`cancelled`。`success` 时 `data` 必填且 `error=null`；其他状态 `error` 必填，`data` 可为空对象或 null。

结果关联键：`invocation_id`。Java 重复收到同一幂等请求时返回已持久化的等价结果，并设置 `meta.replayed=true`，不得重复执行工具。

## 8. SqlProposal

方向：Python -> Java。用途：提交只读 SQL 候选；Java 必须重新授权和校验。

```json
{
  "schema_version": "v1",
  "run_id": "1912345678901234569",
  "dispatch_id": "dsp_01J2Y8Q9V7J8D2M4K5N6P7R8S9",
  "attempt": 1,
  "invocation_id": "sql_inv_01J2Y8QX1M2N3P4R5S6T7V8W9X",
  "request_id": "req_sql_01J2Y8QY",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "idempotency_key": "sql:1912345678901234569:protein-7d:01",
  "deadline_at": "2026-07-11T12:00:00Z",
  "datasource_id": "1912345678901234601",
  "catalog_version": "catalog_v3",
  "question": "最近 7 天蛋白质摄入总量",
  "sql": "SELECT SUM(protein) AS total_protein FROM food_logs WHERE meal_time >= ? LIMIT 100",
  "parameters": ["2026-07-04T00:00:00+08:00"],
  "expected_shape": {
    "columns": ["total_protein"],
    "max_rows": 1
  }
}
```

必填字段：全部字段。`parameters` 可为空数组但不能省略。SQL 只允许单条 `SELECT` 或 `WITH ... SELECT` 候选；即使 proposal 看似安全，Java 仍必须执行 Schema 授权、AST Guard、敏感字段、租户/用户过滤、参数绑定、LIMIT、超时、执行和审计。

幂等键：`invocation_id` + `idempotency_key`，规则与 ToolProposal 相同。相同 invocation 的 SQL、参数、catalog 或 datasource 摘要变化时返回 `SQL_IDEMPOTENCY_CONFLICT`。

## 10. RuntimeError

方向：双向。用途：表示 HTTP 请求级或协议级失败；运行中的业务失败优先使用有序 `run.failed` 事件。

```json
{
  "schema_version": "v1",
  "run_id": "1912345678901234569",
  "request_id": "req_01J2Y8R2",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "error": {
    "code": "RUNTIME_EVENT_GAP",
    "message": "event_seq gap detected",
    "retryable": true,
    "category": "contract",
    "details": {
      "dispatch_id": "dsp_01J2Y8Q9V7J8D2M4K5N6P7R8S9",
      "expected_seq": 5,
      "actual_seq": 7
    }
  }
}
```

必填字段：`schema_version`、`run_id`、`request_id`、`trace_id`、`error.code`、`error.message`、`error.retryable`、`error.category`、`error.details`。无法解析 `run_id` 的认证或 JSON 级错误可使用 `run_id="unknown"`，但日志仍必须带 request/trace。

## 11. 错误码

| 错误码 | HTTP | 可重试 | 语义 |
|---|---:|---:|---|
| `RUNTIME_AUTH_INVALID` | 401 | 否 | Service JWT 签名或 token 无效 |
| `RUNTIME_AUTH_FORBIDDEN` | 403 | 否 | issuer/audience/scope 不匹配 |
| `RUNTIME_VERSION_UNSUPPORTED` | 426 | 否 | 契约主版本不支持 |
| `RUNTIME_CONTRACT_INVALID` | 400 | 否 | JSON、必填字段或枚举不合法 |
| `RUNTIME_DEADLINE_EXCEEDED` | 408 | 视调用方 | 请求到达时已过 deadline |
| `RUNTIME_IDEMPOTENCY_CONFLICT` | 409 | 否 | 相同 dispatch 幂等键对应不同请求摘要 |
| `RUNTIME_EVENT_DUPLICATE` | 200 | 否 | 已接受的事件重放；按成功幂等响应处理 |
| `RUNTIME_EVENT_GAP` | 409 | 是 | `event_seq` 存在缺口 |
| `RUNTIME_EVENT_OUT_OF_ORDER` | 409 | 视重放 | 非重复旧序号或非法乱序 |
| `RUNTIME_STATE_CONFLICT` | 409 | 否 | 事件要求非法状态回退或终态冲突 |
| `RUNTIME_UNAVAILABLE` | 503 | 是 | Runtime 或 Java 控制面暂不可用 |
| `RUNTIME_INTERNAL_ERROR` | 500 | 视详情 | 未分类内部错误 |
| `TOOL_SCHEMA_INVALID` | 400 | 否 | Tool input 不符合注册 schema |
| `TOOL_POLICY_DENIED` | 403 | 否 | Java 策略拒绝 |
| `TOOL_CONFIRMATION_REQUIRED` | 409 | 否 | 缺少有效用户确认 |
| `TOOL_IDEMPOTENCY_CONFLICT` | 409 | 否 | 相同 invocation 对应不同请求摘要 |
| `TOOL_TIMEOUT` | 504 | 仅幂等工具 | 工具超时 |
| `TOOL_FAILED` | 502 | 视工具 | 工具执行失败 |
| `SQL_READONLY_VIOLATION` | 403 | 否 | 非只读 SQL |
| `SQL_SCHEMA_DENIED` | 403 | 否 | 表、字段或数据源未授权 |
| `SQL_SENSITIVE_FIELD_DENIED` | 403 | 否 | 敏感字段无权访问 |
| `SQL_IDEMPOTENCY_CONFLICT` | 409 | 否 | 相同 SQL invocation 摘要冲突 |
| `SQL_TIMEOUT` | 504 | 受 Guard 约束 | SQL 超时 |
| `SQL_EXECUTION_FAILED` | 502 | 受 Guard 约束 | SQL 执行失败 |
| `MODEL_PROVIDER_UNAVAILABLE` | 503 | 是 | 模型供应商不可用 |
| `MODEL_TIMEOUT` | 504 | 按模型策略 | 模型调用超时 |
| `RUN_CANCELLED` | 409 | 否 | Run 已被 Java 接受取消 |

错误信息必须脱敏，不返回 Service JWT、数据库凭据、完整 SQL 结果、完整 Prompt 或供应商密钥。

## 12. Service JWT

### 12.1 Java 调 Python

```json
{
  "iss": "foodmate-control-plane",
  "sub": "foodmate-control-plane",
  "aud": "foodmate-agent-runtime",
  "scope": "agent:run agent:cancel",
  "iat": 1783756500,
  "nbf": 1783756500,
  "exp": 1783756800,
  "jti": "jwt_01J2Y8R4"
}
```

### 12.2 Python 调 Java

```json
{
  "iss": "foodmate-agent-runtime",
  "sub": "foodmate-agent-runtime",
  "aud": "foodmate-control-plane",
  "scope": "agent:event tool:propose sql:propose model:usage",
  "iat": 1783756500,
  "nbf": 1783756500,
  "exp": 1783756800,
  "jti": "jwt_01J2Y8R5"
}
```

固定规则：

- token 有效期不超过 5 分钟；校验签名、issuer、audience、expiry、not-before、scope 和允许的算法。
- 使用 `kid` 支持密钥轮换；兼容窗口允许新旧公钥短时并存，但禁止无 `kid` 的长期共享密钥模式。
- 生产环境可叠加 mTLS，但 mTLS 不能替代 JWT scope、消息契约或 Java 业务授权。
- Service JWT 只证明服务身份，不携带或授权最终用户权限。

## 13. 版本兼容

- `schema_version` 采用主版本字符串。V1 消费者只接受 `v1`；不支持的主版本返回 `RUNTIME_VERSION_UNSUPPORTED`。
- V1 内允许新增可选字段、可选 payload 字段和新非终态 `event_type`；不得删除必填字段、改变字段类型或复用既有枚举语义。
- 由于 Pydantic/DTO 边界默认拒绝未知顶层字段，新增字段必须先发布可识别但可忽略的消费者，再发布生产者。
- 发布顺序固定为：Java/Python 消费者兼容 -> 契约测试通过 -> 生产者发送新字段 -> 观察兼容窗口 -> 才允许清理旧兼容代码。
- 兼容窗口至少覆盖一个完整发布周期和最大 Run/checkpoint 保留期；窗口内旧生产者与新消费者、新生产者与已升级消费者都必须通过契约测试。
- 破坏性变更使用 `v2` 新模型和新路由，不在 `v1` 上静默改义。数据库 Flyway `V2` 与消息契约 `v1` 是不同版本域，不得混淆。

## 14. 契约验收

1. 八类 JSON 示例必须能被 Java DTO/Jackson 与 Python Pydantic v2 双向解析、再序列化并保持语义一致。
2. 必填、未知字段、枚举、UTC 时间、ID 字符串、deadline 和 Service JWT 均有正反例。
3. 重复 Run、Cancel、Tool、SQL 和 Event 均验证不重复产生副作用。
4. 事件覆盖正常顺序、重复、旧序号、缺口、终态竞争和断线重放。
5. Java/Python 版本矩阵覆盖兼容窗口；不支持版本必须明确失败，不能降级为未校验 JSON。
