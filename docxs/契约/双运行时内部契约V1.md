# FoodMate 双运行时内部契约 V1

版本：v1.0 目标契约

维护基线：2026-07-11

对应架构：[架构总览](../架构/架构总览.md)、[Python 智能体运行时设计](../架构/Python智能体运行时设计.md)

对应数据设计：[数据库设计](../数据/数据库设计.md)、[V2 双运行时迁移设计](../数据/V2双运行时迁移设计.md)

文档定位：本文是 Java 业务控制面与 Python Agent Runtime 的内部 HTTP/JSON 契约目标。当前 Java Runtime Client、Python Runtime、内部 DTO 和 V2 持久化均尚未实现；本文不能作为“已经联通”的证明。

## 1. 权威、命名与传输

- Java 是业务数据、`AgentRun`、审计、工具注册与执行、SQL Guard 与 SQL 执行的唯一权威。
- Python 只负责 Agent 推理、编排、RAG、模型调用、结果质量校验和回答组装，不持有业务库写凭据。
- 所有跨运行时消息统一使用 `run_id`。数据库映射固定为 `run_id -> agent_runs.agent_run_id`；本文不把 `agent_run_id` 作为消息字段。
- V1 使用内部 HTTPS + JSON；请求 `Content-Type` 为 `application/json`，字符集为 UTF-8。
- 所有 ID 在 JSON 中使用字符串；时间使用带 `Z` 的 RFC 3339 UTC；字段名使用 `snake_case`。
- Header 固定包含 `Authorization: Bearer <service-jwt>`、`X-Contract-Version: v1`、`X-Request-Id` 和 W3C `traceparent`。

## 2. 通用字段

| 字段 | 类型 | 必填 | 规则 |
|---|---|---:|---|
| `schema_version` | string | 是 | 固定 `v1`；与 `X-Contract-Version` 一致 |
| `run_id` | string | 是 | Java 创建的运行 ID；映射 V1 `agent_runs.agent_run_id` |
| `request_id` | string | 是 | 单次内部 HTTP 请求标识，不作为业务幂等键 |
| `trace_id` | string | 是 | 跨运行时 Trace 关联标识，不作为幂等键 |
| `dispatch_id` | string | 按类型 | 一次 Run 派发标识；同一派发的网络重试保持不变 |
| `attempt` | integer | 按类型 | 同一 `run_id` 的派发代次，从 1 开始；重新派发才递增 |
| `deadline_at` | string | 按类型 | 请求绝对截止时间；过期后不得创建新副作用 |
| `invocation_id` | string | Tool/SQL | 一次逻辑 Tool 或 SQL invocation 的稳定 ID |
| `idempotency_key` | string | Tool/SQL | Java 执行面的幂等键；重试保持不变 |
| `request_hash` | string | 命令/事件/Proposal/Error | canonical request digest，格式为 `sha256:<64 lowercase hex>` |
| `result_hash` | string | Tool/SQL Result | canonical result digest，格式同上 |

`request_id` 可随 HTTP 重试变化；`dispatch_id`、`invocation_id`、`idempotency_key` 不得随同一逻辑操作的网络重试变化。

### 2.1 Canonical request digest

所有摘要都使用同一算法：先按 RFC 8785 JSON Canonicalization Scheme（JCS）对“摘要字段集”生成 UTF-8 字节，再计算 SHA-256，输出 `sha256:<64 lowercase hex>`。对象键按 JCS 排序，数组顺序保留，数字按 JCS 表达；时间和 ID 必须先通过本契约格式校验。摘要不得包含自身、`request_id`、`trace_id`、HTTP Header 或接收时间，因为这些传输字段允许重试时变化。

| 类型 | 幂等标识 | 摘要字段集 | 重放与冲突 |
|---|---|---|---|
| `RunCommand` | `dispatch_id` | `schema_version/run_id/dispatch_id/attempt/deadline_at/message/authorized_context/runtime_options` | 同 ID 同 hash 返回原接受结果；不同 hash 为 `RUNTIME_DISPATCH_IDEMPOTENCY_CONFLICT` |
| `RunEvent` | `(run_id,event_id)` | `schema_version/run_id/dispatch_id/attempt/event_id/event_seq/occurred_at/event_type/payload` | 同键同 hash 是稳定重放；不同 hash 为 `RUNTIME_EVENT_IDEMPOTENCY_CONFLICT` |
| `CancelCommand` | `cancel_id` | `schema_version/run_id/dispatch_id/attempt/cancel_id/deadline_at/reason/requested_at` | 同 ID 同 hash 返回原确认；不同 hash 为 `RUNTIME_CANCEL_IDEMPOTENCY_CONFLICT` |
| `ToolProposal` | `invocation_id` 与 `(run_id,idempotency_key)` | `schema_version/run_id/dispatch_id/attempt/invocation_id/idempotency_key/deadline_at/tool_name/tool_version/input/context/policy` | 任一键命中且 hash 不同为 `TOOL_IDEMPOTENCY_CONFLICT` |
| `ToolResult` | `invocation_id` | `schema_version/run_id/dispatch_id/attempt/invocation_id/status/data/error/meta`，其中 `meta.replayed` 排除 | 同 ID 同 hash 返回原结果；不同 hash 为 `TOOL_RESULT_CONFLICT` |
| `SqlProposal` | `invocation_id` 与 `(run_id,idempotency_key)` | `schema_version/run_id/dispatch_id/attempt/invocation_id/idempotency_key/deadline_at/datasource_id/catalog_version/question/sql/parameters/expected_shape/policy` | 任一键命中且 hash 不同为 `SQL_IDEMPOTENCY_CONFLICT` |
| `SqlResult` | `invocation_id` | `schema_version/run_id/dispatch_id/attempt/invocation_id/status/rows/error/meta`，其中 `meta.replayed` 排除 | 同 ID 同 hash 返回原结果；不同 hash 为 `SQL_RESULT_CONFLICT` |
| `RuntimeError` | `(run_id,error_id)` | `schema_version/run_id/error_id/related_event_id/related_event_seq/error` | 同键同 hash 为稳定错误重放；不同 hash 为 `RUNTIME_ERROR_IDEMPOTENCY_CONFLICT` |

表中 `message`、`input`、`context`、`policy`、`parameters`、`payload`、`data`、`rows`、`error` 和纳入摘要的 `meta` 均按完整对象计算，不能只挑部分子字段。生产者生成摘要，消费者必须独立重算后比较；摘要不匹配按 `RUNTIME_CONTRACT_INVALID` 拒绝。

## 4. RunEvent

方向：Python -> Java。用途：回传单次 dispatch 的有序运行事实。

```json
{
  "schema_version": "v1",
  "run_id": "1912345678901234569",
  "dispatch_id": "dsp_01J2Y8Q9V7J8D2M4K5N6P7R8S9",
  "attempt": 1,
  "event_id": "evt_01J2Y8QF0M1N2P3R4S5T6V7W8X",
  "event_seq": 4,
  "request_id": "req_evt_01J2Y8QG",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "request_hash": "sha256:ec3161b11a867c503aab031cb781ae9d7b826f38d905a20c34edff2ca55170c3",
  "occurred_at": "2026-07-11T11:59:10Z",
  "event_type": "run.planned",
  "payload": {
    "status": "planning",
    "plan_version": "plan_01",
    "step_count": 3,
    "prompt_versions": {
      "router": "router-3",
      "planner": "planner-5"
    }
  }
}
```

必填字段：`schema_version`、`run_id`、`dispatch_id`、`attempt`、`event_id`、`event_seq`、`request_id`、`trace_id`、`request_hash`、`occurred_at`、`event_type`、`payload`。`payload` 可为空对象，但不能省略。RunEvent 不携带 `deadline_at`。

幂等键：数据库/消费端使用 `(run_id,event_id)`；`event_id` 全局唯一更佳，但 V1 保证至少在 `run_id` 内唯一。事件重放必须保持原摘要字段和 `request_hash`；同键不同 hash 返回 `RUNTIME_EVENT_IDEMPOTENCY_CONFLICT`，与序号缺口/乱序错误区分。

### 4.1 event_seq

- 作用域是 `dispatch_id`，从 1 开始严格递增 1。
- 一个 dispatch 只能有一个序号分配者；并发节点先汇入有序事件出口。
- Java 先按 `(run_id,event_id)` 去重，再比较 `event_seq`。
- `event_seq = last_event_seq + 1` 时接受、落 inbox、推进状态并更新 `last_event_seq`。
- `event_seq <= last_event_seq` 且不是已知重复 `event_id` 时返回 `RUNTIME_EVENT_OUT_OF_ORDER`，不得回退状态。
- `event_seq > last_event_seq + 1` 时返回 `RUNTIME_EVENT_GAP`，暂停该 dispatch 的后续状态推进并请求从缺口重放。
- 终态事件一旦被 Java 接受，后续事件只可用于审计拒绝，不得改变终态。

### 4.2 event_type 与状态

| `event_type` | 建议 payload `status` | 说明 |
|---|---|---|
| `run.accepted` | `queued` | Python 接受 dispatch |
| `run.routed` | `routed` | 路由完成 |
| `run.clarification_requested` | `waiting_user` | 等待用户补充 |
| `run.planned` | `planning` | 计划已形成 |
| `run.retrieval_started` | `retrieving` | 开始检索 |
| `run.retrieval_finished` | `retrieving` | 检索完成，状态可暂不迁移 |
| `run.tool_started` | `executing` | Java 已接受 Tool invocation 后报告 |
| `run.tool_finished` | `executing` | Tool 结果已观察 |
| `run.model_usage` | 省略 | 模型调用用量事件；不直接迁移 AgentRun 状态 |
| `run.answer_stream` | `validating` 或省略 | 增量回答；不得作为业务终态 |
| `run.cancel_acknowledged` | 省略 | Python 已停止继续调度；Java 决定最终取消 |
| `run.completed` | `completed` | 终态 |
| `run.failed` | `failed` | 终态 |
| `run.cancelled` | `cancelled` | 可由 Python建议；Java 首次合法终态裁决 |

数据库 V1 状态仅允许 `queued/routed/waiting_user/planning/retrieving/executing/validating/completed/failed/cancelled`，没有 `cancelling`。内部事件不得要求 Java 写入 V1 不存在的状态。

### 4.3 model_usage RunEvent

`run.model_usage` 使用标准 RunEvent envelope；其 `payload` 完整结构如下：

```json
{
  "model_call_id": "mdl_01J2Y8MU1M2N3P4R5S6T7V8W9X",
  "provider_request_id": "siliconflow_req_8f31",
  "provider_attempt_id": "mat_01J2Y8MV1M2N3P4R5S6T7V8W9X",
  "scene": "planner",
  "provider_code": "siliconflow",
  "model_name": "Qwen/Qwen3-32B",
  "status": "success",
  "usage": {
    "input_tokens": 1240,
    "output_tokens": 318,
    "total_tokens": 1558
  },
  "latency_ms": 842,
  "cost": {
    "amount": "0.003116",
    "currency": "CNY"
  },
  "started_at": "2026-07-11T11:59:08Z",
  "finished_at": "2026-07-11T11:59:09Z"
}
```

payload 全部字段必填；供应商未返回 request ID 时 `provider_request_id` 使用 null，但字段不能省略。ID 关系固定为：

- 契约 `request_id` 是单次 HTTP 传输追踪 ID，可在事件重放时变化，不进入模型用量唯一语义，也不复制供应商 ID。
- `model_call_id` 标识一次逻辑模型调用，在该调用的供应商重试间稳定，并作为 V2 `model_usage_logs` 的逻辑关联 ID。
- `provider_request_id` 是供应商对一次真实请求返回的 ID；每次供应商重试可不同，无法获得时为 null。
- `provider_attempt_id` 是 Python 为每次本地供应商 attempt 生成的全局唯一 ID；一次逻辑调用可对应多个 attempt。

Java 仍先按 RunEvent `(run_id,event_id)` 和 event hash 幂等处理，再以 `model_call_id` 查找/创建唯一逻辑调用父记录，以 `provider_attempt_id` 幂等写 attempt 子记录。相同 `provider_attempt_id` 且 payload hash 一致是重放，不新增记录；不同 hash 返回 `MODEL_USAGE_IDEMPOTENCY_CONFLICT`。Java 汇总 attempt usage、latency、cost 和最终状态到父记录；`model_call_id` 唯一，多个 provider attempt 只存在于 V2 子表。

```json
{
  "schema_version": "v1",
  "run_id": "1912345678901234569",
  "dispatch_id": "dsp_01J2Y8Q9V7J8D2M4K5N6P7R8S9",
  "attempt": 1,
  "cancel_id": "can_01J2Y8QK1M2N3P4R5S6T7V8W9X",
  "request_id": "req_cancel_01J2Y8QM",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "request_hash": "sha256:07eac15e07e9b99da9e964763bd50a9b7ffde581fc01d40732e91d4dff1c2660",

  "reason": "user_requested",
```

必填字段：全部字段。`reason` 允许 `user_requested`、`deadline_exceeded`、`operator_requested`、`superseded`、`system_shutdown`。

## 7. ToolResult

方向：Java -> Python。用途：返回策略裁决和确定性执行结果。

```json
{
  "schema_version": "v1",
  "run_id": "1912345678901234569",
  "dispatch_id": "dsp_01J2Y8Q9V7J8D2M4K5N6P7R8S9",
  "attempt": 1,
  "invocation_id": "tool_inv_01J2Y8QR1M2N3P4R5S6T7V8W9X",
  "request_id": "req_tool_result_01J2Y8QV",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "result_hash": "sha256:03aa0993949e841a9fb0a44de23cb451f6797286803e312ae9171abc2c87e884",
  "status": "success",
  "data": {
    "food_log_id": "1912345678901234573",
    "status": "saved"
  },
  "error": null,
  "meta": {
    "tool_call_id": "1912345678901234564",
    "latency_ms": 182,
    "replayed": false
  }
}
```

必填字段：除 `data` 与 `error` 按状态互斥外，其余字段全部必填。`status` 允许 `success`、`failed`、`denied`、`confirmation_required`、`timeout`、`cancelled`。`success` 时 `data` 必填且 `error=null`；其他状态 `error` 必填，`data` 可为空对象或 null。

结果关联键：`invocation_id`。Java 持久化 `result_hash` 和用于重放的完整 Result JSON；重复请求返回同一稳定结果，仅允许把响应中的 `meta.replayed` 改为 `true`，该字段不参与摘要。同一 invocation 产生不同 `result_hash` 时返回 `TOOL_RESULT_CONFLICT`，不得覆盖原结果。

ToolResult 到 V1 `tool_calls.status` 的映射固定为：`success -> success`、`failed -> failed`、`timeout -> timeout`、`cancelled -> cancelled`；`denied` 和 `confirmation_required` 在 V1 没有对应状态，V2 新增 `result_status` 保存原契约状态，同时 V1 `status` 分别写 `failed` 和 `pending`，`error_code` 写 `TOOL_POLICY_DENIED` 或 `TOOL_CONFIRMATION_REQUIRED`。`output_json` 保存完整 `ToolResult`，因此 Java 可按原契约状态重放而不是从 V1 状态反推。

## 8. SqlProposal

方向：Python -> Java。用途：提交只读 SQL 候选；Java 必须重新授权和校验。

```json
{
  "schema_version": "v1",
  "run_id": "1912345678901234569",
  "dispatch_id": "dsp_01J2Y8Q9V7J8D2M4K5N6P7R8S9",
  "attempt": 1,
  "invocation_id": "sql_inv_01J2Y8QX1M2N3P4R5S6T7V8W9X",
  "request_id": "req_sql_01J2Y8QY",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "request_hash": "sha256:fe5dd1a718db035e5c769830c8511eac9e7c15b53f1f7ec4a360b6708e7f9782",
  "idempotency_key": "sql:1912345678901234569:protein-7d:01",
  "deadline_at": "2026-07-11T12:00:00Z",
  "datasource_id": "1912345678901234601",
  "catalog_version": "catalog_v3",
  "question": "最近 7 天蛋白质摄入总量",
  "sql": "SELECT COALESCE(SUM((nutrition_json ->> 'protein')::numeric), 0) AS total_protein FROM food_logs WHERE meal_time >= ? AND is_deleted = FALSE LIMIT 100",
  "parameters": ["2026-07-04T00:00:00+08:00"],
  "expected_shape": {
    "columns": ["total_protein"],
    "max_rows": 1
  },
  "policy": {
    "readonly": true,

}
```

必填字段：全部字段。`parameters` 可为空数组但不能省略。示例只使用 V1 真实列 `food_logs.meal_time/nutrition_json/is_deleted`，Result 的 `total_protein` 与 `expected_shape` 闭合；`policy.required_filters` 声明 `user_id`，Java 根据 `run_id` 注入可信用户过滤，不能采用 Python 提供的用户值。SQL 只允许单条 `SELECT` 或 `WITH ... SELECT` 候选；Java 仍必须执行 Schema 授权、AST Guard、敏感字段、租户/用户过滤、参数绑定、LIMIT、超时、执行和审计。

幂等键：`invocation_id` + `idempotency_key`，规则与 ToolProposal 相同。相同 invocation 的 datasource、catalog、question、SQL、参数、expected shape、policy 或其他摘要字段变化时返回 `SQL_IDEMPOTENCY_CONFLICT`。

## 9. SqlResult

方向：Java -> Python。用途：返回 SQL 校验/执行结果或拒绝原因。

```json
{
  "schema_version": "v1",
  "run_id": "1912345678901234569",
  "dispatch_id": "dsp_01J2Y8Q9V7J8D2M4K5N6P7R8S9",
  "attempt": 1,
  "invocation_id": "sql_inv_01J2Y8QX1M2N3P4R5S6T7V8W9X",
  "request_id": "req_sql_result_01J2Y8R0",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "result_hash": "sha256:66a867818d9d02c54ffedca89db81b1188debee7671ab01c8aedf91f8129b3d9",
  "status": "success",
  "rows": [
    { "total_protein": 612.5 }
  ],
  "error": null,
  "meta": {
    "sql_audit_id": "1912345678901234603",
    "row_count": 1,
    "latency_ms": 280,
    "catalog_version": "catalog_v3",
    "replayed": false,
    "truncated": false
  }
}
```

必填字段：除 `rows` 与 `error` 按状态互斥外，其余字段全部必填。`status` 允许 `success`、`rejected`、`failed`、`timeout`、`cancelled`。`success` 时 `rows` 必填且 `error=null`；其他状态 `error` 必填，`rows` 必须为空数组。

结果关联键：`invocation_id`。Java 持久化 `result_hash` 和完整脱敏 Result JSON；重放只允许把 `meta.replayed` 改为 `true`。同一 invocation 产生不同 `result_hash` 时返回 `SQL_RESULT_CONFLICT`，不得覆盖或再次执行 SQL。

SqlResult 到 V1 `sql_query_audits.status` 的映射固定为：`success -> executed`、`rejected -> rejected`；`failed/timeout/cancelled` 在 V1 没有精确状态，V2 新增 `result_status` 保存原契约状态，V1 `status` 写 `rejected` 并在 `reject_reason` 保存对应错误码。V2 `result_json` 保存完整脱敏 `SqlResult`，重放不得从 `rejected` 猜测原始失败类型。

## 10. RuntimeError

方向：双向。用途：表示 HTTP 请求级或协议级失败；运行中的业务失败优先使用有序 `run.failed` 事件。

```json
{
  "schema_version": "v1",
  "run_id": "1912345678901234569",
  "error_id": "err_01J2Y8R1M2N3P4R5S6T7V8W9X",
  "related_event_id": "evt_01J2Y8QF0M1N2P3R4S5T6V7W8X",
  "related_event_seq": 7,
  "request_id": "req_01J2Y8R2",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "request_hash": "sha256:9465644ba055f48e32d14f04a2f027f2254a8d192f1000fbc3b35ea25a5488b6",
  "error": {
    "code": "RUNTIME_EVENT_GAP",
    "message": "event_seq gap detected",
    "retryable": true,
    "category": "contract",
    "details": {
      "dispatch_id": "dsp_01J2Y8Q9V7J8D2M4K5N6P7R8S9",
      "expected_seq": 5,
      "actual_seq": 7
    }
  }
}
```

必填字段：`schema_version`、`run_id`、`error_id`、`request_id`、`trace_id`、`request_hash`、`error.code`、`error.message`、`error.retryable`、`error.category`、`error.details`。错误由 RunEvent 处理触发时，`related_event_id` 与 `related_event_seq` 必须成对出现并保持稳定；请求级错误使用 null。无法解析 `run_id` 的认证或 JSON 级错误可使用 `run_id="unknown"`。

幂等键：`(run_id,error_id)`。同键重放必须保持 `error_id`、关联事件字段、完整 `error` 和 `request_hash` 不变；消费者重算摘要，一致时返回/记录原错误且不重复推进状态，不一致时返回 `RUNTIME_ERROR_IDEMPOTENCY_CONFLICT`。因此 RuntimeError 与其他七类消息一样具有明确重放规则。

## 12. Service JWT

### 12.1 Java 调 Python

```json
{
  "iss": "foodmate-control-plane",
  "sub": "foodmate-control-plane",
  "aud": "foodmate-agent-runtime",
  "scope": "agent:run agent:cancel",
  "iat": 1783756500,
  "nbf": 1783756500,
  "exp": 1783756800,
  "jti": "jwt_01J2Y8R4"
}
```

### 12.2 Python 调 Java

```json
{
  "iss": "foodmate-agent-runtime",
  "sub": "foodmate-agent-runtime",
  "aud": "foodmate-control-plane",
  "scope": "agent:event tool:propose sql:propose",
  "iat": 1783756500,
  "nbf": 1783756500,
  "exp": 1783756800,
  "jti": "jwt_01J2Y8R5"
}
```

固定规则：

- token 有效期不超过 5 分钟；校验签名、issuer、audience、expiry、not-before、scope 和允许的算法。
- 使用 `kid` 支持密钥轮换；兼容窗口允许新旧公钥短时并存，但禁止无 `kid` 的长期共享密钥模式。
- 生产环境可叠加 mTLS，但 mTLS 不能替代 JWT scope、消息契约或 Java 业务授权。
- Service JWT 只证明服务身份，不携带或授权最终用户权限。
- `POST /foodmate/internal/v1/agent-events`（包括 `run.model_usage`）要求 `agent:event`；ToolProposal 入口要求 `tool:propose`；SqlProposal 入口要求 `sql:propose`。V1 没有独立 model usage HTTP 入口，因此不声明未使用的 `model:usage` scope。

## 13. 版本兼容

- `schema_version` 采用主版本字符串。V1 消费者只接受 `v1`；不支持的主版本返回 `RUNTIME_VERSION_UNSUPPORTED`。
- V1 内允许新增可选字段、可选 payload 字段和新非终态 `event_type`；不得删除必填字段、改变字段类型或复用既有枚举语义。
- 由于 Pydantic/DTO 边界默认拒绝未知顶层字段，新增字段必须先发布可识别但可忽略的消费者，再发布生产者。
- 发布顺序固定为：Java/Python 消费者兼容 -> 契约测试通过 -> 生产者发送新字段 -> 观察兼容窗口 -> 才允许清理旧兼容代码。
- 兼容窗口至少覆盖一个完整发布周期和最大 Run/checkpoint 保留期；窗口内旧生产者与新消费者、新生产者与已升级消费者都必须通过契约测试。
- 破坏性变更使用 `v2` 新模型和新路由，不在 `v1` 上静默改义。数据库 Flyway `V2` 与消息契约 `v1` 是不同版本域，不得混淆。

## 14. 契约验收

1. 八类 JSON 示例必须能被 Java DTO/Jackson 与 Python Pydantic v2 双向解析、再序列化并保持语义一致。
2. 必填、未知字段、枚举、UTC 时间、ID 字符串、deadline 和 Service JWT 均有正反例。
3. 重复 Run、Cancel、Tool、SQL 和 Event 均验证不重复产生副作用。
4. 事件覆盖正常顺序、重复、旧序号、缺口、终态竞争和断线重放。
5. Java/Python 版本矩阵覆盖兼容窗口；不支持版本必须明确失败，不能降级为未校验 JSON。
