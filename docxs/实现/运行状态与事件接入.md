## 2. 入站边界与身份验证

`POST /foodmate/internal/v1/agent-events` 接收包括 `run.model_usage` 在内的 `RunEvent`，要求 Python Service JWT：`iss=sub=foodmate-agent-runtime`、`aud=foodmate-control-plane`、`scope=agent:event`，并校验签名、允许算法、`kid`、`iat/nbf/exp` 和最长 5 分钟有效期。Header 必须含 `X-Contract-Version: v1`、`X-Request-Id` 和 `traceparent`。

身份、版本和原始 JSON/schema 校验在业务事务之前完成。此时若尚未得到可信 `run_id`，返回 `PreRunProtocolError`，只写 `protocol_error_audits`，不得写 `runtime_error_inbox`、不得创建或更新 AgentRun。得到并验证真实 `run_id` 后的运行内协议错误才使用 RuntimeError。

Service JWT 只证明 Python 服务身份。事件中的 `run_id/dispatch_id/attempt` 必须反查 Java dispatch 和 AgentRun；不得从 payload 接受用户、租户或权限事实。

## 3. 固定事务顺序

事务顺序必须保持：身份验证 -> inbox 幂等插入/摘要冲突 -> seq/状态校验 -> AgentRun/审计/模型用量更新 -> commit -> SSE。身份验证在事务外，其余持久化步骤处于一个数据库事务。

```text
authenticate service JWT, version, headers and schema
verify request_hash with V1 JCS + SHA-256 field set

begin transaction
  lock dispatch by dispatch_id
  assert dispatch.agent_run_id == run_id and dispatch.attempt == event.attempt
  load/lock AgentRun

  existing = find inbox by (agent_run_id, event_id)
  if existing exists:
    if existing.request_hash == event.request_hash:
      return duplicate success without state/audit/usage/SSE repetition
    else:
      return RUNTIME_EVENT_IDEMPOTENCY_CONFLICT

  insert inbox with full validated payload and request_hash
  compare event_seq with dispatch.last_event_seq
  validate event_type, payload status and AgentRun transition

  apply AgentRun, cancellation, audit and/or model usage projection
  insert one transactional SSE outbox row for every newly applied frontend-visible event
  mark inbox applied and advance last_event_seq only for contiguous application
commit

return HTTP acknowledgement after commit
publish from durable outbox after commit
```

生产者摘要字段集固定为 `schema_version/run_id/dispatch_id/attempt/event_id/event_seq/occurred_at/event_type/payload`。Java 必须独立重算；摘要不得包含 `request_id/trace_id/request_hash`。摘要格式必须是 `sha256:<64 lowercase hex>`。

## 4. Dispatch 仲裁与 fencing

V2 当前设计定义了 dispatch 技术状态和 `(agent_run_id, attempt)` 唯一性，但尚未有 active epoch/fencing 列。实现必须通过后续 V2 增量迁移或等价的已评审持久化目标补齐仲裁字段，不能把它们当成已存在：

| 目标字段/语义 | 目标约束 |
|---|---|
| `dispatch_arbitration_state` | 只允许 `active/superseded/expired`；与现有技术 `status` 分开，避免改写 V2 已定义的技术状态集合 |
| `active_epoch` | 每个 `run_id` 同时只能有一个 active epoch；创建新 active 时原 active 原子地变为 `superseded` |
| `fencing_token` | Java 根据 active epoch、`dispatch_id` 和 `attempt` 派生并持久化/校验的内部 fence；不是 V1 RunEvent 新增 wire 字段 |

事件入口在 inbox 幂等判断前后都必须确认 `dispatch_id/attempt` 是该 Run 当前 active dispatch，且派生的 fencing token 仍匹配。旧 dispatch 的事件即使 `event_seq` 连续，也只写拒绝/审计并返回稳定 ACK；不得写入可应用 inbox，不得推进 `last_event_seq`、AgentRun 状态、Tool/SQL 结果或终态。active dispatch 被 superseded/expired 的转换与新 dispatch 创建必须在同一锁定的 AgentRun 事务内完成。

## 5. 序号、重复、乱序与缺口

`event_seq` 作用域是 `dispatch_id`，从 1 开始严格递增 1。游标只能存放在 V2 `agent_run_dispatches.last_event_seq`，不能放在 AgentRun 上替代历史 dispatch。

| 输入 | inbox/游标处理 | 返回与后续动作 |
|---|---|---|
| 已知 `(run_id,event_id)`、同 hash | 不新增、不重做投影、不推进游标 | 200 `RUNTIME_EVENT_DUPLICATE` 语义；不重复 SSE |
| 已知 key、不同 hash | 原记录不覆盖 | `RUNTIME_EVENT_IDEMPOTENCY_CONFLICT` |
| `event_seq = last + 1` | 校验状态后应用，更新为 `applied`，游标加 1 | commit 后发布 SSE |
| `event_seq <= last` 且非已知重复 event | 记录为 `rejected`，游标不变 | `RUNTIME_EVENT_OUT_OF_ORDER`，不回退状态 |
| `event_seq > last + 1` | 记录为 `gap`，游标不变 | `RUNTIME_EVENT_GAP`，请求从缺口重放 |
| 同 `(dispatch_id,event_seq)` 不同 event | 不覆盖已存在事件 | 按非法乱序拒绝，保留审计证据 |

缺口事件可以先留在 inbox。`EventGapReconciler` 只在缺失序号已经补齐后，按 `last_event_seq + 1` 逐条锁定并应用连续 `gap` 记录；每条仍执行同一状态校验和投影，不能跳号提高游标。对账重启从持久化游标继续，不清空 inbox，不重复副作用。

## 6. 状态机与终态竞争

V1 AgentRun 状态只允许 `queued/routed/waiting_user/planning/retrieving/executing/validating/completed/failed/cancelled`，没有 `cancelling`。事件映射固定为：

| event_type | 状态处理 |
|---|---|
| `run.accepted` | `queued` |
| `run.routed` | `routed` |
| `run.clarification_requested` | `waiting_user` |
| `run.planned` | `planning` |
| `run.retrieval_started/finished` | `retrieving`，finished 可保持当前状态 |
| `run.tool_started/finished` | `executing` |
| `run.answer_stream` | payload 合法时可进入/保持 `validating`，绝不成为终态 |
| `run.model_usage` | 不迁移 AgentRun 状态 |
| `run.cancel_acknowledged` | 更新 cancellation，不直接要求不存在的 `cancelling` |
| `run.completed` | `completed` |
| `run.failed` | `failed` |
| `run.cancelled` | Java 首次合法终态裁决为 `cancelled` |

状态机必须显式允许自保持事件，拒绝非法回退。处理事件时锁定 AgentRun；`completed/failed/cancelled` 中第一个通过校验并提交的事件成为权威终态。后到终态或非终态只记录拒绝审计并返回 `RUNTIME_STATE_CONFLICT`，不得覆盖 `result_json/error_code` 或终态时间。

## 7. 取消与运行超时

Java 接受取消时先按 `cancel_id` 和 CancelCommand `request_hash` 幂等写 `agent_run_cancellations`，状态从 `requested` 到 `dispatched`。相同 ID 同 hash 返回原记录；不同 hash 返回 `RUNTIME_CANCEL_IDEMPOTENCY_CONFLICT`。

取消接受必须锁定与 Tool/SQL invocation 创建相同的 `agent_runs` 行，或在同一行上 CAS 递增后续 V2 增量目标 `cancellation_epoch`。取消记录与 epoch 递增同事务提交；规则固定为取消请求提交后禁止新 invocation。已创建 invocation 保存创建时 epoch，并在执行前、提交业务副作用前重新校验，防止“proposal 已通过但取消先提交”的竞态。`cancellation_epoch/invocation_epoch` 当前不在 V2 设计中，必须由后续 V2 增量迁移补齐，不得假装已存在。

收到 `run.cancel_acknowledged` 时，在事件事务内写 `ack_event_id` 并更新为 `acknowledged`。若 AgentRun 仍处于可取消非终态，Java 可裁决为 `cancelled` 并把 cancellation 置为 `resolved`；若 `completed` 或 `failed` 已先提交，只将 cancellation 置为 `resolved` 并保留该终态。收到 `run.cancelled` 同样遵循首次合法终态规则。取消后拒绝创建新的 Tool/SQL invocation，但既有 invocation 继续对账。

总运行 deadline 到期由 Java 创建/复用 `reason=deadline_exceeded` 的 CancelCommand；不能批量把超时 Run 伪造成 `completed`。HTTP 取消接受与 AgentRun 取消终态是两个阶段。

## 8. AgentRun、审计与结果投影

每个已连续接受事件在同一事务内完成适用投影：

- 更新 `agent_runs.status/plan_json/result_json/error_code/trace_id/updated_at`，但只写事件明确提供且契约允许的事实。
- 更新 `agent_run_event_inbox.processing_status/reject_code/applied_at`。
- 更新 dispatch 的 `accepted_at/status/finished_at/last_event_seq`，不把技术 dispatch 状态当成 AgentRun 状态。
- 更新 cancellation、ToolCall、SQLAudit 或模型用量引用；不得从前端 SSE 反向重建业务状态。
- 写操作审计时记录 `request_id/trace_id/session_id/operator/source/status/error_code`，敏感 payload 先脱敏。

事务任一步失败时，inbox 插入、AgentRun/审计/usage 更新和游标推进一起回滚；不得发送成功 SSE。

## 9. `run.model_usage` attempt 聚合

`run.model_usage` 先经过 RunEvent `(run_id,event_id)` 幂等和顺序校验，再处理 payload。顶层必填 `model_call_id/provider_request_id/provider_attempt_id/scene/provider_code/model_name/status/usage/latency_ms/cost/started_at/finished_at`；状态只允许 `success/failed/timeout/cancelled`。

处理顺序固定：

1. 以 `model_call_id` 查找或创建唯一父 `model_usage_logs`；兼容期父 `request_id=model_call_id`，不得写 envelope `request_id`。
2. 以 `provider_attempt_id` 全局去重；`provider_request_id` 非空时再以 `(provider_code,provider_request_id)` 去重。
3. 任一键命中且 usage payload hash 相同是重放，不新增、不聚合；不同 hash 返回 `MODEL_USAGE_IDEMPOTENCY_CONFLICT`。
4. 插入 `source_kind=native/contributes_to_parent=true` 的 attempt 后，只基于已去重且参与聚合的 native attempts 重算父记录。
5. 币种冲突返回 `MODEL_USAGE_CURRENCY_CONFLICT`，父聚合保持不变。

聚合规则不得简化：token 字段分别对非空值求和；完全未知为 null，部分未知保存已知和并标记 `complete=false`。成本只对同币种非空 amount 求和，部分未知标记 `cost_complete=false`；未知不按 0。父 `latency_ms` 是 attempts 的 latency 之和。任一 attempt 成功则父状态保持 success；否则取 `finished_at` 最新 attempt，时间相同时取 `provider_attempt_id` 字典序最大者。

`legacy_unexpanded` 父行没有 attempt 也合法；`legacy_expanded` attempt 固定 `contributes_to_parent=false`。不得从历史父行合成 RunEvent，也不得让历史子记录改写父聚合。

## 10. RuntimeError 与 PreRunProtocolError 分流

| 类型 | 识别条件 | 持久化与状态影响 |
|---|---|---|
| PreRunProtocolError | JWT、版本、JSON 解析或可信 `run_id` 前 schema 失败 | 按 `request_id/fingerprint` 写无 AgentRun FK 的 `protocol_error_audits`；不更新 Run |
| RuntimeError | 已解析并验证真实 `run_id` | 按 `(agent_run_id,error_id)` 与 `request_hash` 写 `runtime_error_inbox`；同 key 同 hash 重放，不重复推进 |
| `run.failed` | 运行中的有序业务失败 | 走 RunEvent inbox、序号和状态机，可能成为首次合法 `failed` 终态 |

RuntimeError 同键不同 hash 返回 `RUNTIME_ERROR_IDEMPOTENCY_CONFLICT`。关联事件时 `related_event_id/related_event_seq` 必须成对出现。RuntimeError 不分配 `event_seq`；PreRunProtocolError 不允许伪造 `run_id=unknown`。

## 11. Reliable SSE：transactional outbox

当前 V2 设计没有 SSE outbox 表。实现必须在后续 V2 增量迁移中新增目标表，或依赖另一个明确评审的 V2 增量目标；本文下面的表和字段是目标设计，不是现有数据库事实。推荐目标表 `agent_run_sse_outbox`：

| 字段 | 目标语义 |
|---|---|
| `sse_outbox_id` | Snowflake 主键 |
| `agent_run_id` | 对应 `run_id -> agent_runs.agent_run_id` |
| `sse_event_id` | 稳定 SSE 事件 ID，重试和恢复不变 |
| `stream_seq` | 每个 Run 严格递增的恢复游标 |
| `event_type`/`payload_json` | 已脱敏、已授权的前端事件 |
| `status` | `pending/leased/sent` 等内部投递状态，不能当成 V1 AgentRun 状态 |
| `lease_owner`/`lease_until` | publisher lease 和崩溃接管 |
| `attempt_count`/`next_attempt_at` | 有界退避和重试 |
| `created_at`/`sent_at` | 审计时间 |

每个新应用的、前端可见的事件在同一事务中写入 AgentRun/inbox/审计和一条 outbox；同一 inbox event 重放不得产生第二个 `sse_event_id`。`stream_seq` 必须在锁定 AgentRun 时分配，提交前固化。事务提交后，publisher 用 `lease_owner/lease_until` CAS 领取，发送成功再标记 `sent`；进程在提交后、发送前崩溃时，lease 到期后由另一 publisher 接管并以同一 ID 重发。

SSE 端点必须接受浏览器 `Last-Event-ID` 和持久化 resume cursor，按 `(agent_run_id,stream_seq)` 查询未确认/之后的 outbox 行。投递语义是至少一次；客户端必须按 `sse_event_id` 去重，再更新本地 cursor。publisher 发送失败只延迟重试，不回滚已提交业务事务；永久失败进入可观测告警和人工/运维处理，不能重新执行业务副作用。

- duplicate、gap、out-of-order、摘要冲突和终态冲突不重复发布业务 SSE。
- `run.answer_stream` 可发布增量，但断线恢复必须从已持久化 outbox/结果重建，不能依赖未提交内存帧。
- publisher 失败不回滚已提交业务事务；lease 重试和客户端 resume 共同处理提交后崩溃，不重新应用 inbox。
- SSE envelope 中 Snowflake ID 继续以字符串输出。

## 12. 测试矩阵

| 场景 | 关键断言 |
|---|---|
| 身份/版本失败 | PreRunProtocolError，无 AgentRun/inbox 写入 |
| 首个连续事件 | inbox、AgentRun、审计、游标同事务提交；随后才有 SSE |
| 同 event 同 hash | 只一条事实、无重复投影和 SSE |
| 同 event 不同 hash | `RUNTIME_EVENT_IDEMPOTENCY_CONFLICT`，原记录不变 |
| 旧序号 | `RUNTIME_EVENT_OUT_OF_ORDER`，状态/游标不回退 |
| 缺口与补齐 | gap 先留存；补齐后仅按连续序列应用 |
| 迟到旧 dispatch 连续事件 | 即使 `event_seq` 连续，active dispatch/fencing 不匹配也只审计/ACK，不推进 seq/state/终态 |
| 并发相同 seq | 唯一约束生效，只一条可应用 |
| 终态竞争 | completed/failed/cancelled 只接受首个合法提交 |
| 取消确认 | 无 `cancelling`；已完成 Run 不被取消覆盖 |
| 取消与 invocation 竞态 | 两者锁同一 AgentRun 或 CAS epoch；取消先提交时无新 invocation，invocation 先创建但取消先于副作用提交时被 fence 拒绝 |
| 事务回滚 | inbox、状态、审计、usage、游标均回滚且无 SSE |
| pre-run/runtime error | 两类表和幂等键完全分离 |
| model attempt 重放 | 不新增 attempt、不重复聚合 |
| 部分未知 usage | null 不按 0，completeness 标记正确 |
| success 后失败 | 父状态保持 success |
| 币种冲突 | `MODEL_USAGE_CURRENCY_CONFLICT`，父聚合不变 |
| SSE 发布失败 | 数据已提交且可恢复，不重做业务投影 |
| transactional outbox | AgentRun/inbox/outbox 同事务；提交后崩溃可由 lease 接管，`sse_event_id/stream_seq` 不变 |
| SSE 至少一次 | publisher 重试可能重复发送，但前端按 `sse_event_id` 去重；Last-Event-ID/resume cursor 补齐断线事件 |

数据库并发、约束和回滚必须使用真实 PostgreSQL Testcontainers；仅 mock Repository 不能证明事务顺序成立。
