## 1. 文档定位与审计口径

本文基于当前工作区的前端、Java、Flyway、测试和配置文件，以及本次重新执行的构建/测试命令。它是后续代码工作的当前现状与优先级依据；目标架构仅在“完善计划”中出现，不代表已经实现。

本任务只写文档和同步索引，不修复前端、Java、Flyway 或 Python 代码。事实分为两类：

- **已证实**：可由当前文件、全仓搜索或命令输出直接确认。
- **待验证风险**：由代码路径推导，但尚未通过真实运行环境或行为测试复现；每项均给出验证方法。

### 5.3 待验证风险

| 风险 | 推导依据 | 验证方法 |
|---|---|---|
| 软删除恢复可能更新 0 行 | [SoftDeleteRepositorySupport.java](../../foodmate-infra/src/main/java/com/foodmate/infrastructure/persistence/SoftDeleteRepositorySupport.java) 对已删除实体先清除标记再调用 `BaseMapper.updateById`；[SoftDeleteRepositorySupportTest.java](../../foodmate-infra/src/test/java/com/foodmate/infrastructure/persistence/SoftDeleteRepositorySupportTest.java) 的代理只记录方法调用，不执行 MyBatis-Plus 真实 SQL。逻辑删除插件可能在 UPDATE 条件中排除数据库里 `is_deleted=true` 的行 | 在 Testcontainers PostgreSQL 中插入 active 行、软删、用正常查询确认不可见、执行 restore、断言 update count=1 且三项删除字段清空；同时捕获 SQL，确认恢复条件只命中指定主键/租户/已删除行 |

## 8. 依赖顺序与完成定义

1. 先关闭 P0-1 至 P0-3，使现有单运行时骨架具有可信质量和数据库基线。
2. P0-4 先完成契约/V2 评审，再写 Runtime Client 或 Python Runtime，避免双端并行猜字段。
3. P1-1 提供身份、会话和消息权威数据；P1-2 再打通 AgentRun/SSE；P1-3 最后替换前端 mock 主链路。
4. P1-4 的工具/SQL 实际执行始终由 Java 授权和审计；Python 只能提交 proposal。
5. P2 在真实主链路稳定后推进。每项只有在表中验收命令/测试实际通过后才能标记完成。

本审计应在生产代码、迁移、测试结果或优先级发生变化时同步更新；历史命令结果不得替代新的验证。
