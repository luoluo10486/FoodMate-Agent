# FoodMate 当前实现审阅稿

更新时间：2026-07-18

## 1. 结论

当前项目已经完成 MVP 的“认证 + 会话消息 + AgentRun + Java/Python Runtime + 事件回调 + SSE + 前端真实接入”主链路。

当前可描述为：

```text
前端登录
  -> Java 认证与用户归属
  -> 创建 Session、Message、AgentRun
  -> Java dispatch Python Runtime
  -> Python 产生 RunEvent
  -> Java EventInbox 去重并更新 AgentRun
  -> Java 写入助手消息
  -> SSE 推送到前端
```

这不是完整产品闭环。Tool/SQL、RAG、模型治理、食物日志、分析、餐食计划和知识库仍属于后续阶段；PostgreSQL 事务恢复链路也尚未实库验证。

## 2. 已实现内容

### 2.1 Java 认证与账户

代码位置：`foodmate-application/src/main/java/com/foodmate/application/account/UserAccountService.java`

- 用户注册和登录。
- PBKDF2 密码哈希和密码校验。
- 服务端认证会话签发、校验和撤销。
- 会话 ID 与 CSRF Token 均只保存哈希；浏览器不保存登录 Token。
- 用户状态校验：`active`、`disabled`、`locked`。
- 用户资料查询和更新。
- local-stub 内存实现，以及存在数据源时的 JDBC 实现。

认证接口：

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/auth/register` | 注册 |
| POST | `/api/auth/login` | 登录，并设置认证 Cookie |
| POST | `/api/auth/refresh` | 刷新令牌 |
| POST | `/api/auth/logout` | 撤销当前认证会话并清理 Cookie |
| GET | `/api/users/me` | 当前用户 |
| GET | `/api/users/me/profile` | 当前资料 |
| PUT | `/api/users/me/profile` | 更新资料 |

### 2.2 会话和消息

代码位置：`foodmate-api/src/main/java/com/foodmate/api/controller/SessionController.java`

- 创建会话。
- 查询当前用户会话列表。
- 归档会话。
- 查询会话消息。
- 添加用户、助手、系统或工具消息。
- 消息具有 `sequence_no`，并可以关联 `agent_run_id`。
- 服务端根据 access token 确定用户，不信任前端传入的用户身份。

接口：

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/sessions` | 查询当前用户会话 |
| POST | `/api/sessions` | 创建会话 |
| DELETE | `/api/sessions/{sessionId}` | 归档会话 |
| GET | `/api/sessions/{sessionId}/messages` | 查询消息 |
| POST | `/api/sessions/{sessionId}/messages` | 添加消息 |

### 2.3 AgentRun 和 Chat API

代码位置：

- `foodmate-api/src/main/java/com/foodmate/api/controller/ChatController.java`
- `foodmate-application/src/main/java/com/foodmate/application/runtime/RuntimeGatewayService.java`

- 创建 Chat Run。
- 认证请求使用数字字符串作为 `run_id`，并与 `agent_runs.agent_run_id` 对应。
- 用户消息与助手消息关联同一个 `agent_run_id`。
- 查询 Run 状态和事件。
- 取消 Run。
- 校验当前用户是否为 Run 所有者，防止跨用户读取、SSE 订阅和取消。

接口：

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/chat/runs` | 创建 Run 并派发 |
| GET | `/api/chat/runs/{runId}` | 查询状态 |
| GET | `/api/chat/runs/{runId}/events` | 查询事件历史 |
| POST | `/api/chat/runs/{runId}/cancel` | 取消 Run |
| GET | `/api/chat/runs/{runId}/stream` | SSE 流 |

状态流转：

```text
DISPATCHED -> RUNNING -> SUCCEEDED
                     -> FAILED
                     -> CANCELED
```

对应业务状态：

```text
DISPATCHED -> queued
RUNNING    -> executing
SUCCEEDED  -> completed
FAILED     -> failed
CANCELED   -> cancelled
```

### 2.4 Runtime 派发和事件处理

代码位置：`foodmate-application/src/main/java/com/foodmate/application/runtime/RuntimeGatewayService.java`

已经实现：

- dispatch 幂等。
- cancel 幂等。
- 相同幂等 ID 但请求内容不同的冲突检测。
- deadline 超时后自动触发取消。
- 事件重复检测。
- 事件序号缺口检测。
- 乱序事件检测。
- 非法状态迁移拒绝。
- 事件历史保存和 SSE 重放。
- Runtime 状态更新 AgentRun。
- 成功事件中的 `answer` 写入助手消息。

Java Runtime 内部接口：

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/internal/runtime/runs:dispatch` | Java 接收或转发派发 |
| POST | `/internal/runtime/runs:cancel` | Java 接收取消 |
| POST | `/internal/runtime/runs:events` | Python 回调事件 |

内部接口使用双向 Ed25519 Service JWT 和 `X-Contract-Version`；Java/Python 使用各自私钥签发、对方公钥验签，当前仅覆盖 dispatch、cancel 和 event scope。

### 2.5 Python Runtime

代码位置：`agent-runtime/`

- 已从临时 stub 独立为 Python 工程。
- 包含 `pyproject.toml`、运行服务、README 和单元测试。
- 仅使用标准库实现最小 Runtime。
- 提供 `/health/live` 和 `/health/ready`。
- 接收 dispatch 和 cancel。
- 对相同 `dispatch_id` 做基本幂等处理。
- 按 `DISPATCHED -> RUNNING -> SUCCEEDED` 产生事件。
- 观察取消状态并产生 `CANCELED` 事件。
- 通过 HTTP 回调 Java，不直接访问业务数据库。

### 2.6 前端真实接入

代码位置：`frontend/src/services/`

当 `VITE_AGENT_MODE=real` 时：

- 登录调用 `/api/auth/login`。
- access token 写入前端会话存储。
- API 与 SSE 自动携带 HttpOnly 会话 Cookie。
- 写操作从非敏感 CSRF Cookie 读取值并发送 `X-CSRF-Token`；SSE 支持 `lastEventId`。
- Chat Run 使用真实 Java API。
- 接收事件后更新运行状态和助手答案。
- 支持取消当前 Run。

默认模式仍为 mock，便于单独开发前端原型。

## 5. 审阅重点

请重点审阅以下决策：

1. Java 是否继续作为用户、业务数据、授权和工具执行的唯一权威。
2. Python Runtime 是否只负责 Agent 编排、模型调用和 proposal 生成。
3. 是否接受双向 Ed25519 Service JWT 作为 Java/Python 服务认证基线，并将 HTTPS/mTLS 和密钥轮换列为部署阶段门禁。
4. 是否接受当前 local-stub 内存实现，下一步再用 PostgreSQL/Testcontainers 验证事务和重启恢复。
5. 是否先建设 Java Tool Gateway/Policy，再实现 Python Router/Planner，而不是让 Python 直接访问业务数据库。

## 6. 当前建议的下一步

优先补 PostgreSQL 持久化可靠性：

```text
AgentRun + dispatch outbox
  -> publisher lease/CAS
  -> Python Runtime
  -> Java EventInbox
  -> AgentRun / assistant message / SSE outbox 同一事务投影
  -> 服务重启恢复
```

完成这一步并通过 Testcontainers 验证后，再进入 Tool Gateway、SQL Guard 和 Python Planner，避免在尚未具备可靠运行基础的主链路上继续堆叠业务能力。
