# FoodMate 前端架构文档

版本：v1.0  
状态：Phase 1-2 前端实现入口文档  
详细来源：[UI 设计规范](../设计/UI设计规范.md)、[接口与数据规范](../契约/接口与数据规范.md)、[实现附录](../参考/实现附录.md)

## 1. 定位

本文是 FoodMate 前端实现入口，覆盖 Phase 1 静态 UI 原型和 Phase 2 Mock 交互。第一轮不创建真实后端依赖，不调用真实接口，只通过 mock 数据和 mock Agent 事件回放验证页面结构、组件边界、状态展示和交互体验。

前端第一轮目标：

- 让首页、会话页、分析页、规划页、个人资料页和管理后台入口具备可运行的静态或 mock 原型。
- 让会话页能在无后端情况下播放一轮 Agent mock 任务。
- 让追问卡、确认卡、工具轨迹、引用、失败和取消状态先在前端闭环。
- 为后续接入 `/foodmate/agent-runs/{agent_run_id}/events` SSE 事件保留数据结构和状态映射。

## 2. 技术栈决策

锁定技术栈：

- React
- Vite
- TypeScript
- Arco Design
- CSS Modules

选择 React 而不是 Vue 的原因：

- FoodMate 更像 Agent 工作台，不是普通表单后台；核心复杂度在消息流、工具轨迹、Agent 状态和事件回放。
- React 的组件组合和 TypeScript 契约适合承载 `AgentRun`、`ToolCall`、`SSEEvent` 这类状态驱动 UI。
- 后续接入 SSE、mock event replay、局部状态机和复杂业务卡片时，React 生态和 AI Coding Agent 生成稳定性更好。
- Vue 3 也可实现本项目，但当前文档、系统设计和执行路线已偏向 React，切换会增加文档和实现同步成本。

选择 Arco Design 的原因：

- Arco 提供稳定的 Button、Input、Modal、Drawer、Tooltip、Tabs、Table、Card、Tag、Progress、Skeleton、Message 等基础控件。
- FoodMate 需要比纯手写控件更快地搭出可用界面，同时保持比通用后台更强的产品气质。
- Arco 参与整体视觉优化，但不直接套 Arco Pro 后台模板；FoodMate 的 Agent 工作台布局、色彩、卡片层级和动效仍由项目自定义。

## 3. Phase 边界

Phase 1：静态 UI 原型

- 创建前端应用骨架。
- 实现首页、会话页、分析页、规划页的静态页面。
- 抽取共享组件，组件通过 props 接收 mock 数据。
- 覆盖空态、加载中、错误态、完成态等静态状态。

Phase 2：Mock 交互

- 使用前端本地 mock event sequence 模拟 AgentRun 生命周期。
- 支持消息发送后的状态推进、工具轨迹出现、答案逐步输出。
- 支持 `waiting_user` 追问流程和确认流程。
- 支持 `failed`、`cancelled`、停止按钮和重试入口。

不属于 Phase 1-2：

- 不接真实后端。
- 不创建真实数据库、SSE 服务或认证系统。
- 不实现真实 RAG、Tool、SQL Agent、ModelService。
- 不引入大型全局状态方案，除非 Phase 2 事件回放复杂度证明必要。

## 4. 路由与页面

推荐路由：

| 路由 | 页面 | Phase |
|---|---|---|
| `/` | 首页 | Phase 1 |
| `/login` | Mock 登录入口、用户资料和鉴权状态占位 | Phase 1 |
| `/profile` | 个人资料页：昵称、头像、邮箱、营养目标、忌口、单位偏好、修改密码入口；Phase 2 先做 mock，真实接口接入属于后端认证/个人资料阶段 | Phase 2 mock / 后续真实接入 |
| `/admin/*` | 独立管理后台，仅 `admin/operator` 可见，包含概览与治理子页面；Phase 2 先做 mock，真实接口接入属于管理接口阶段 | Phase 2 mock / 后续真实接入 |
| `/chat/:session_id?` | 会话页 | Phase 1-2 |
| `/analysis` | 分析页 | Phase 1 |
| `/planning` | 规划页 | Phase 1 |
| `/knowledge` | 知识库检索占位 | Phase 1 |

路由行为：

- Phase 1 可以先使用前端路由，不依赖后端 session。
- 首页快捷任务可跳转到会话页并带入 mock prompt。
- 会话页如果没有 `session_id`，使用 mock 当前会话。
- 分析页和规划页第一版使用静态 mock 数据。
- `/login` 第一版提供 mock 登录、注册入口、忘记密码入口、用户资料、营养目标、权限范围和鉴权状态说明，不写入真实 token。
- `/profile` 第一版展示 mock 个人资料，后续接 `GET/PATCH /foodmate/users/me`、头像上传和修改密码接口。
- `/admin` 第一版展示 mock 管理概览，普通用户不可见；真实接入后根据 `/foodmate/auth/me` 的 `role` 控制入口和路由访问。
- `/admin/users`、`/admin/runs`、`/admin/tools`、`/admin/usage`、`/admin/knowledge`、`/admin/deleted` 为管理后台独立子路由，左侧后台导航必须真实跳转。
- `/knowledge` 第一版只展示知识库检索与 ACL 过滤占位，不接真实 RAG 接口。

真实认证接入后的路由守卫：

- 未登录访问受保护页面跳转 `/login?redirect=当前路径`。
- 登录过期时先调用 `/foodmate/auth/refresh`，刷新失败再跳转登录页。
- 普通用户访问 `/admin` 展示 403 权限不足状态。
- `operator` 可进入管理后台只读视图，涉及用户状态、工具启停、恢复删除资源等操作必须隐藏或禁用。
- `admin` 可看到完整管理入口和高风险操作确认。

分层规则：

- `pages/` 只负责页面组合，不直接写复杂业务状态。
- `components/` 放通用 UI 组件，不绑定具体业务数据来源。
- `features/agent/` 放 AgentRun、ToolCall、mock event replay、状态映射。
- `features/auth/` 放登录态、token 刷新、路由守卫、角色权限映射。
- `features/profile/` 放个人资料、头像上传、修改密码相关状态。
- `features/admin/` 放管理后台概览、用户管理、运行审计、知识库和工具管理状态。
- `features/session/` 放会话列表、消息、Composer 相关逻辑。
- `mock/` 放静态数据和事件序列，Phase 2 不写死在组件里。
- `styles/` 放 FoodMate design tokens、Arco 主题覆盖和轻量动效。

## 6. Arco Design 使用策略

Arco 参与整体视觉风格，但必须服务 FoodMate 的餐饮 Agent 工作台定位。

使用 Arco 的场景：

| Arco 组件 | FoodMate 用法 |
|---|---|
| `Button` | 发送、停止、确认、重试、快捷任务操作 |
| `Input` / `Input.TextArea` | Composer 输入、搜索会话、追问回复 |
| `Modal` | 高风险确认、保存计划、删除或恢复确认 |
| `Drawer` | 工具轨迹详情、引用详情、移动端侧栏 |
| `Tooltip` | 工具按钮、状态图标、不熟悉的操作 |
| `Tabs` | Tools / Agents 展开面板或管理视图占位 |
| `Table` | 餐食计划、分析明细、后续管理列表 |
| `Upload` | 头像上传、知识库文档导入 |
| `Card` | 指标卡、结果卡、推荐任务卡的基础容器 |
| `Tag` | Agent 状态、Tool 状态、风险等级 |
| `Progress` | Agent 阶段推进、计划校验进度 |
| `Skeleton` | 页面加载和局部加载 |
| `Message` | 操作反馈、mock 保存成功、错误提示 |

必须自定义的场景：

- 工作台整体布局。
- 左侧会话栏。
- 底部 Composer。
- Tools / Agents 状态区。
- 消息流和 Agent 运行轨迹。
- ResultCard、CitationBlock、ClarificationCard、ConfirmationCard。
- FoodMate 的背景、卡片层级、餐饮相关色彩和轻量动效。

禁止事项：

- 不直接套 Arco Pro 后台模板。
- 不把所有内容都做成默认 Arco Card。
- 不使用默认蓝色作为唯一主视觉。
- 不让组件库默认间距破坏 `1440 x 760` 单屏工作台结构。

## 7. 视觉与动效策略

视觉方向：

- Arco 的控件清晰度 + FoodMate 的温暖餐饮任务感。
- 浅色中性背景，搭配绿色、陶土橙、信息蓝和少量危险红。
- 页面应像一个每天可用的 Agent 工作台，而不是普通后台管理系统。

动效允许范围：

- 页面进入时的轻量 fade/slide。
- 首页任务卡 hover/focus 的轻微抬升和阴影变化。
- Agent 状态推进时的进度条和状态 Tag 过渡。
- 工具轨迹展开/收起时的高度或透明度过渡。
- 追问卡、确认卡出现时的轻量弹入。
- answer stream mock 输出时的文本逐步出现。

性能边界：

- Phase 1-2 优先使用 CSS transition / animation。
- 不引入 Framer Motion 等额外动画库，除非后续明确需要。
- 动效不阻塞首屏渲染。
- 底部输入区高度必须稳定。
- 消息流、工具轨迹和表格只允许局部滚动。
- 必须支持 `prefers-reduced-motion`，用户减少动态时关闭非必要动效。

## 8. Mock 数据与事件

Phase 1 mock 数据：

- sessions：会话列表、置顶、最近会话、当前会话。
- tasks：快捷任务、推荐问题。
- messages：用户消息、助手消息、结果卡。
- analysis：指标、趋势、目标、异常提示。
- mealPlans：约束、多日菜单、购物清单、预算、校验结果。

Phase 2 mock event sequence 必须模拟：

- `run.created`
- `run.routed`
- `run.clarification_requested`
- `run.planned`
- `run.retrieval_started`
- `run.retrieval_finished`
- `run.tool_started`
- `run.tool_finished`
- `run.answer_stream`
- `run.completed`
- `run.failed`
- `run.cancelled`

状态规则：

- AgentRun 状态和 UI 状态必须遵循[UI 设计规范](../设计/UI设计规范.md)的映射表。
- ToolCall 状态至少覆盖 `pending`、`running`、`success`、`failed`、`timeout`、`cancelled`。
- 停止按钮触发后进入 `cancelled`，停止追加 mock 工具和答案事件。
- 追问流程进入 `waiting_user`，用户补充后回到 `planning`。
- 确认流程未确认前不能展示“已保存”。

## 9. 状态管理

Phase 1：

- 页面使用静态 mock 数据和组件 props。
- 不引入全局状态库。

Phase 2：

- 优先使用 React state / reducer 管理 mock event replay。
- 建议把事件回放封装为 `useMockAgentRun` 或等价 hook。
- 如果后续 mock 交互跨页面共享明显变复杂，再评估 Zustand 等轻量状态库。
- 不提前引入 Redux、MobX 或复杂状态框架。

真实认证接入阶段：

- Access Token 优先保存在内存状态，不写入 localStorage。
- Refresh Token 由 HttpOnly Cookie 承载，前端不读取明文 refresh token。
- `auth/me` 结果作为用户菜单、路由守卫、管理入口和权限展示的数据源。
- 头像上传、资料保存、修改密码等操作都使用真实接口反馈，不伪造已保存状态。

个人资料页规划：

- 展示并编辑昵称、头像、邮箱、身高、体重、活动水平、营养目标、忌口、过敏原和单位偏好。
- 头像上传使用图片预览、替换确认和删除入口。
- 修改密码需要输入旧密码和新密码。
- 用户不能修改自己的角色和账号状态。

管理后台页面规划：

- 概览：AgentRun 数量、失败率、工具调用量、模型用量、知识库索引状态。
- 用户管理 `/admin/users`：用户列表、详情、启用、禁用、锁定、重置登录会话；该页按后端契约仅 `admin` 可访问。
- Agent 运行 `/admin/runs`：AgentRun、ToolCall、SQLAudit、Trace 查询，ModelUsage 单独放在 `/admin/usage`。
- 工具调用 `/admin/tools`：工具注册表、版本、入参 schema、启停、风险等级和权限范围。
- 模型用量 `/admin/usage`：供应商、模型、场景、token、成本和耗时。
- 知识库 `/admin/knowledge`：文档列表、详情、上传、下线、恢复、索引状态。
- 软删除资源 `/admin/deleted`：查看软删除资源并恢复，仅 `admin` 可操作。

管理后台接口与页面对照：

| 后端能力 | 前端体现 |
|---|---|
| `/foodmate/admin/overview` | `/admin` 概览指标、运行审计摘要、治理资源卡片 |
| `/foodmate/admin/users`、`/foodmate/admin/users/{user_id}` | `/admin/users` 用户列表和用户详情侧栏，仅 `admin` |
| `/foodmate/admin/users/{user_id}/status` | `/admin/users` 行内锁定、禁用、启用类按钮，二次确认和审计提示 |
| `/foodmate/admin/users/{user_id}/sessions/reset` | `/admin/users` 重置会话按钮和登录会话表 |
| `/foodmate/admin/agent-runs` | `/admin/runs` 的 AgentRun 标签页 |
| `/foodmate/admin/tool-calls` | `/admin/runs` 的 ToolCall 标签页 |
| `/foodmate/admin/sql-audits` | `/admin/runs` 的 SQLAudit 标签页 |
| `/foodmate/admin/model-usage` | `/admin/usage` 模型调用明细表 |
| `/foodmate/admin/knowledge-documents` | `/admin/knowledge` 文档列表、详情和索引状态 |
| `/foodmate/admin/knowledge-documents` 写入接口 | `/admin/knowledge` 上传弹窗、下线/恢复二次确认 |
| `/foodmate/admin/tools`、`/foodmate/admin/tools/{name}/status` | `/admin/tools` 工具列表、详情、启停二次确认 |
| `/foodmate/admin/deleted-resources`、restore | `/admin/deleted` 软删除列表和恢复二次确认，仅 `admin` |
| 管理写操作审计 | `/admin`、`/admin/users`、`/admin/tools`、`/admin/knowledge`、`/admin/deleted` 展示 mock 审计表或审计提示 |

## 10. 验收标准

Phase 1 完成时：

- 首页、会话页、分析页、规划页可本地访问。
- 页面结构符合 SVG 原型的信息层级。
- Arco 组件和 FoodMate 自定义样式融合，不呈现通用后台模板感。
- 桌面端首屏能看到侧栏、主内容、Tools/Agents、Composer。
- 移动端不横向溢出。

Phase 2 完成时：

- 无后端时可演示一轮完整 Agent mock 任务。
- Agent 状态、工具轨迹、引用、答案流同步变化。
- 追问、确认、失败、取消都有明确 UI。
- mock 事件名与[接口与数据规范](../契约/接口与数据规范.md)中的 `run.*` SSE 事件一致。
- 后续接真实 SSE 时，不需要重写核心 UI 组件。

认证与管理页面接入完成时：

- `/profile` 能展示和编辑当前用户资料，头像上传路径与后端接口一致。
- `/admin` 只对 `admin/operator` 可见，普通用户访问时展示 403。
- `/admin` 使用独立后台布局，不复用 Agent 工作台会话侧栏。
- `/admin` 左侧管理导航点击后能切换到用户管理、Agent 运行、工具调用、模型用量、知识库和软删除资源子路由。
- `operator` 访问用户管理和软删除资源时展示 admin-only 权限状态。
- 管理写操作按钮必须打开二次确认，提交后展示审计写入反馈。
- 顶部用户菜单包含个人资料、管理后台和退出登录；普通用户不显示管理后台入口。
- 401 触发 refresh，refresh 失败回到 `/login?redirect=...`。
- 403 展示权限不足，不自动重试。
