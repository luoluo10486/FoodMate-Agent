# FoodMate 测试策略

## 1. 原则与当前基线

测试结论必须来自实际命令、容器或浏览器结果。当前代码和目标设计分开验收：现有骨架通过不等于双运行时、业务 API、Tool/SQL 或部署链路已完成。

| 范围 | 当前命令 | 2026-07-11 基线 | 结论 |
|---|---|---|---|
| 前端构建 | `cd frontend; npm run build` | 通过 | TypeScript 检查和 Vite 生产构建通过 |
| 前端 lint | `cd frontend; npm run lint` | 失败，35 个 warning，`--max-warnings 0` | 当前质量门禁失败 |
| 前端测试 | `cd frontend; npm run test` | 失败，未发现匹配测试文件 | 当前没有前端回归测试 |
| Java | `.\mvnw.cmd clean verify` | 通过，29 tests，0 failure/error/skipped | 仅覆盖当前骨架与既有测试 |
| PostgreSQL/Flyway | 无当前容器测试命令 | 未执行 | 现有测试只读取 SQL 文本，不证明真实 PostgreSQL 可迁移 |
| Python Runtime | 无命令 | 工程不存在 | 不可安装、启动或测试 |

## 2. 前端最小测试矩阵

前端测试文件补齐后，最小矩阵应使用 Vitest、React Testing Library 和 MemoryRouter，覆盖以下行为：

| 场景 | 最小断言 | 当前状态 |
|---|---|---|
| 路由与 mock 登录 | mock 状态下的预期页面与受限入口行为明确 | 未实现真实认证或路由守卫 |
| 会话切换 | 从 `/chat/:session_id` 的会话 A 切到 B 时，A 的消息、Run、卡片和 seed 不泄漏到 B | 待补测试；审计已识别风险 |
| Agent replay | seed 只执行一次，完成、失败和取消的 mock 显示一致 | 待补测试 |
| 无行为控件 | 未实现的 Tools、Agents、附件、搜索和菜单项应禁用、移除或有明确可见状态 | 待实现/测试 |
| 管理 mock | 禁用用户、工具切换、删除恢复会刷新 UI，且每个用例重置 fixture | 待补测试；当前共享可变 mock 有污染风险 |
| 构建与风格 | `npm run build`、`npm run lint`、`npm run test` 均退出 0 | 当前 lint/test 未达到 |

真实 HTTP、refresh、RBAC、SSE 重连和浏览器 E2E 属于后续真实主链路测试，不得用当前 mock 结果代替。

## 3. Java 单元、集成与 Testcontainers

保持当前 `.\mvnw.cmd clean verify` 作为骨架基线。后续数据库和 V2 工作必须新增真实 PostgreSQL Testcontainers，而不是只断言迁移 SQL 文本。Testcontainers 验收至少包含：

- 空库从 V1 升级到 V2、带代表性 V1 数据升级到 V2，以及 Flyway `migrate`/`validate`。
- V2 阶段 1-4 的可空新增列与上一兼容 Java 版本的读写 smoke test。
- dispatch 的 partial unique 单 active、active/admission epoch、fencing token、旧 dispatch 拒绝和事务回滚。
- `runtime_event_inbox` 只接受 canonical event：`(agent_run_id,event_id)` 与 active `(dispatch_id,event_seq)` 唯一；同 seq 不同 event、旧/superseded dispatch、乱序和 gap 只写无冲突唯一键的 `runtime_event_rejections`，并返回稳定 RuntimeError。
- 取消/admission epoch 与完成/失败终态竞争，首个 Java 合法终态不回退；旧 epoch/fence invocation 在副作用提交前被拒绝。
- Tool/SQL 的 `invocation_id` 与 `(agent_run_id,idempotency_key)` 幂等，`pending/leased/succeeded/failed/unknown` CHECK、owner token CAS、lease 过期安全接管，以及恶意 SQL、越权 schema、限行、超时和审计断言。
- SSE transactional outbox 的 `source_event_key/sse_event_id/(agent_run_id,stream_seq)` 唯一、业务事务原子性、publisher lease/retry/dead-letter、崩溃接管和 `Last-Event-ID` 恢复。
- `model_call_id`、provider attempt 去重和 V1 `request_id` 兼容。

当前 POM 没有 Testcontainers 依赖；在依赖、容器可用性和测试 profile 落地前，不能声称上述集成测试已运行。V2 阶段 4 启用 Runtime/Tool/SQL/SSE 主链路前，上述表、字段、索引、约束和 fixtures 必须全部通过，不能留作后续增量。

## 4. 双运行时契约与故障注入

V1 内部契约落地后，以 JSON Schema 或等价严格 DTO 校验覆盖 RunCommand、RunEvent、CancelCommand、ToolProposal/Result、SqlProposal/Result、RuntimeError 与 PreRunProtocolError。每个消息验证 `v1`、snake_case、字符串 ID、UTC 时间、未知顶层字段拒绝、RFC 8785 JCS + SHA-256 摘要、幂等键和错误码映射。

Java Runtime Client 的 MockWebServer 或统一 WireMock 测试应注入并验证：connect/read/run timeout、503、连接重置、重复 dispatch、取消重试、401/403/400/409 不重试、426 版本不兼容、PreRun 协议错误与流内 `runtime.error`。每次网络重试必须复用业务幂等标识、重新生成 request ID 和短期 Service JWT，且不越过 `deadline_at`。

## 5. 目标 Python 测试，当前不可执行

Python `agent-runtime/` 创建后，在该项目虚拟环境执行 pytest。其测试应覆盖编排、模型/RAG adapter、checkpoint 可重建性、Prompt/评测的离线 golden、V1 Schema fixture、Service JWT 验证、事件序列和 Tool/SQL proposal。Python 测试还必须证明它不持有业务数据库凭据，也不能直接执行 Tool 或 SQL。

目标 Runtime 的路由契约测试还必须覆盖 `GET /foodmate/internal/health/live` 和 `GET /foodmate/internal/health/ready`，并拒绝任何缺少 `/foodmate/internal` 前缀的 Python health 路由。readiness 只断言实例可安全认证、接收和持久化请求；单个 run gap、duplicate conflict、lease backlog 与 outbox lag 另作业务告警和发布/灰度门禁测试，不能通过摘除恢复事件入口来“恢复” readiness。

当前没有 Runtime 目录、`pyproject.toml`、虚拟环境或 pytest 配置，因此不存在可以诚实列出的 Python 安装、启动或 pytest 命令。

## 6. E2E 与质量门禁

真实主链路可用后，浏览器 E2E 应覆盖登录、会话、发送消息、AgentRun 流式事件、Tool/SQL 确认、取消、断线恢复和终态展示。E2E 前置条件是 Java 业务 API、认证、V2 迁移、Runtime Client 和 Python Runtime 都已实际实现；当前不得创建“通过”的替代结论。

建议门禁分层如下：

1. 每次前端变更：typecheck/build、lint、单元测试均为 0 退出码。
2. 每次 Java 变更：`.\mvnw.cmd clean verify`，并对迁移/持久化改动运行 PostgreSQL Testcontainers。
3. 每次契约改动：Java/Python 两端 Schema fixture、canonical digest、JWT 和错误映射互测；所有文档 JSON 示例程序化解析并按字段集重算摘要。
4. 每次发布候选：兼容矩阵、PostgreSQL V2 reliability fixtures、故障注入、双运行时 smoke、迁移回滚演练和浏览器 E2E。

在现有 lint/test 失败、没有容器验证或 Python Runtime 不存在时，相应门禁状态必须是失败或不可执行，而非豁免通过。
