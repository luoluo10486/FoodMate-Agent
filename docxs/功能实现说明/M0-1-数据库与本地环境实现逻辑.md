## 2. 功能概述

通过 Docker Compose 提供本地 PostgreSQL、Redis 和 MinIO，并通过人工 SQL 建立 FoodMate 核心表及 M1-1 所需账户隐私表。Flyway 默认关闭，数据库变更由人工脚本控制和校验。

## 3. 接口清单

本阶段没有新增业务 HTTP 接口。基础设施通过数据库连接、Redis 连接和 MinIO S3 兼容接口提供能力。

## 4. 实现逻辑

### 4.1 本地基础设施启动

- `docker/compose.yml` 定义 PostgreSQL、Redis 和 MinIO 三个容器。
- Compose 文件定义三个容器的端口、数据卷和健康检查。
- PostgreSQL 负责业务表持久化。
- Redis 提供缓存基础设施。
- MinIO 提供 S3 兼容对象存储。
- 数据卷保证容器重建后数据仍然保留。

### 4.2 数据库脚本执行

- 先将 PostgreSQL 导出到 `script/sql/FoodMate/backups/`。
- 再执行 `V1__init_core_schema.sql`，建立核心表。
- 然后执行 `V2__m1_account_and_privacy.sql`。
- V2 脚本增加账户隐私、导出和注销相关表。
- 先备份、再建表，保证基础表先于业务扩展表创建。

### 4.3 数据库结构校验

- `V2__m1_account_and_privacy_validation.sql` 查询数据库表数量。
- 校验脚本检查关键索引、外键和唯一约束。
- 校验脚本检查角色数据。
- 脚本确认 `users`、`user_auth_sessions` 等表可用。
- 脚本确认 `superadmin` 角色存在。
- 校验失败时不把数据库标记为可用。

### 4.4 应用连接与迁移策略

- 应用启动配置从 `.env` 或环境变量读取 PostgreSQL 地址和凭据。
- 应用读取 Redis 配置并建立 Redis 连接。
- 应用读取 MinIO endpoint 和凭据并建立 MinIO 客户端。
- 当前项目关闭 Flyway 自动迁移。
- 数据库变更必须先执行人工 SQL，再执行校验脚本。

## 5. 涉及的数据库表

V1 建立核心用户、资料、头像、会话、消息、运行时、审计等表；V2 增加或调整：

- `users`
- `user_profiles`
- `user_avatar_assets`
- `user_auth_sessions`
- `password_reset_tokens`
- `data_export_jobs`
- `account_deletion_jobs`

## 6. 外部依赖与配置

| 依赖 | 用途 | 不可用时行为 |
|---|---|---|
| PostgreSQL | 业务持久化 | local 真实模式无法启动或提供持久化能力 |
| Redis | 缓存/基础设施预留 | 由运行配置决定是否阻断启动 |
| MinIO | 私有头像和导出文件对象存储 | 头像、导出任务不可执行 |

敏感配置只从 `.env` 读取，`.env` 不提交 Git；`.env.example` 仅作为变量模板。
