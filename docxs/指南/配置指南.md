# FoodMate 配置指南

## 1. 配置分层与现状

配置事实以 `foodmate-bootstrap/src/main/resources/application*.yml` 为准。当前只有 Java 配置；Python Runtime、内部 Runtime URL、Service JWT 和目标态模型/RAG 配置尚未在仓库中实现。配置名出现在设计文档中不等于已有可用配置项。

Secret 只能从环境变量或受控 Secret 挂载注入。不要在 Markdown、YAML 默认值、提交记录、终端输出或日志中写入真实密钥、JWT、私钥、数据库密码或模型请求体。

## 2. 当前 Java YAML

| 文件 | 用途 | 当前事实 |
|---|---|---|
| `application.yml` | 公共默认配置 | 服务名 `foodmate`、端口 `8080`、Actuator 暴露 `health,info`、UTC Jackson、MyBatis-Plus、旧 AI 相关键和 `local-stub` 默认 profile |
| `application-local-stub.yml` | 无数据库本地骨架 | 排除 DataSource/Flyway 自动配置，`foodmate.runtime.stub-enabled=true` |
| `application-local.yml` | 本机 PostgreSQL | `DB_URL`、`DB_USERNAME`、`DB_PASSWORD` 可覆盖的 datasource，Flyway 启用，stub 关闭 |
| `application-dev.yml` | 开发环境 PostgreSQL | 必须提供 `DB_URL`、`DB_USERNAME`、`DB_PASSWORD`，Flyway 启用，stub 关闭 |
| `application-prod.yml` | 生产 PostgreSQL | 必须提供 `DB_URL`、`DB_USERNAME`、`DB_PASSWORD`，Flyway 启用，stub 关闭 |

`local` 中现有默认 URL 是 `jdbc:postgresql://localhost:5432/foodmate`，默认用户名和密码是 `foodmate`。这些仅是当前本地便利默认值，不应复制为共享环境或生产凭据。`dev` 与 `prod` 没有默认值，缺失环境变量时应让启动失败而不是用猜测的连接信息继续运行。

## 3. 当前旧 AI 配置，待迁移

`application.yml` 仍包含下列旧职责配置：

- `foodmate.model`：默认聊天和嵌入模型、调用超时。
- `foodmate.gateway`：`base-url` 和 `MODEL_GATEWAY_KEY`。
- `foodmate.rag`：`top-k`、`rerank-top-n`、`collection`。
- `foodmate.milvus`：host、port 和 `MILVUS_TOKEN`。
- `foodmate.mcp`、`foodmate.sql-agent` 与 `foodmate.tool`：旧 Agent、SQL 和工具相关开关。

这些键仍是现有 Java YAML 的静态事实，但当前没有 Runtime Client、模型网关调用、RAG、Tool 执行或 SQL Guard 实现。迁移前不得把它们解释为已运行能力；迁移时应由 Java 控制面与 Python Runtime 分别接管职责，避免同一 secret 被两个运行时无边界复用。

## 4. 目标 Java Runtime 配置，当前不可执行

已批准的 Java Runtime Client 蓝图使用 `foodmate.agent-runtime` 前缀，替代旧 model gateway 语义。目标配置至少包括：

| 配置类别 | 目标项 | 约束 |
|---|---|---|
| Runtime 连接 | `base-url`、`contract-version`、connect/read/run timeout、重试上限与退避 | 生产地址必须为内部 HTTPS；版本固定 `v1`；任一 timeout 不得超过绝对 `deadline_at` |
| Java 到 Python 身份 | JWT issuer、audience、key id、TTL、私钥引用 | audience 固定为 `foodmate-agent-runtime`；短期 JWT；TTL 不得超过 300 秒；私钥只从 Secret 挂载或环境引用读取 |
| Python 到 Java 身份 | 反向 Service JWT 的 issuer、audience、scope、key id 与公钥引用 | Python 事件、ToolProposal 和 SqlProposal 必须经过 Java 验证；JWT 不替代 Java 的业务授权 |
| 契约与可观测性 | `X-Contract-Version`、request/trace 标识与脱敏日志 | 所有 ID 是字符串，时间是 UTC RFC 3339；禁止记录 JWT、私钥、完整 Prompt 或原始请求体 |

当前 YAML 不含这些最终键，当前 Java 也不签发或校验上述 Service JWT。实现前不得复制示例私钥或将静态 Bearer token 写入配置。

## 5. 目标 Python 模型与 RAG 配置，当前不可执行

Python Runtime 落地后，应由它拥有模型供应商选择、模型名、推理超时、Prompt 版本、RAG 检索参数、embedding/rerank 配置和技术 checkpoint。模型 API key、向量检索 token、供应商端点认证与任何 RAG 存储凭据必须通过 Runtime 的 Secret 引用传入。

Python 不应获得 PostgreSQL 业务库 URL、用户名或密码，也不应直接复用 Java 的 Tool/SQL 执行凭据。Java 只向 Python 传递经授权的 `authorized_context`、Runtime 选项和 V1 契约消息；Python 的 Tool/SQL 请求必须以 proposal 回到 Java。

## 6. 配置变更检查

修改当前 Java YAML 后，使用相应 profile 启动 Java，并检查 `/actuator/health`。对需要数据库的 profile，先确认环境变量和可用 PostgreSQL；`local-stub` 不验证数据库或未来 Runtime。目标双运行时配置落地后，再增加启动时的 HTTPS、契约版本、JWT TTL、`kid`、私钥引用和 timeout 校验，并在测试中验证不泄漏 secret。
