# FoodMate V2 双运行时数据迁移设计

版本：v1.0 目标设计

维护基线：2026-07-11

对应契约：[双运行时内部契约 V1](../契约/双运行时内部契约V1.md)

实现基线：[V1 数据库设计](./数据库设计.md)、[当前实现审计与完善计划](../实现/当前实现审计与完善计划.md)

文档定位：本文只设计后续 Flyway V2 的数据结构、迁移顺序和验收方式，不创建 SQL、不修改迁移文件，也不表示 V2 已实现。已发布的 `V1__init_core_schema.sql` 不得回改；实际 V2 文件必须在实现任务中根据本文评审后另行创建。

## 1. V1 事实与目标边界

当前 V1 的已证实事实：

- `agent_runs` 主键列是 `agent_run_id BIGINT`，状态只允许 `queued/routed/waiting_user/planning/retrieving/executing/validating/completed/failed/cancelled`，没有 `cancelling`。
- `agent_runs` 没有 `dispatch_id`、`attempt`、`deadline_at`、`last_event_seq` 或取消请求字段。
- V1 没有事件 inbox；无法持久化 `(run_id,event_id)` 去重和事件序号缺口。
- V1 没有 dispatch active epoch/fencing、Run admission/cancellation epoch、事件拒绝审计或可靠 SSE outbox。
- `tool_calls` 和 `sql_query_audits` 没有 `invocation_id`、`idempotency_key`、请求摘要、执行租约或 owner token。
- `model_usage_logs.request_id` 非空并具有全局唯一索引；V1 没有 `model_call_id`。
- Java 代码尚无 AgentRun、Runtime Client、事件消费、Tool/SQL 幂等或 Python Runtime 实现。

目标所有权不变：Java 是业务数据、AgentRun、事件接受结果、取消裁决、工具/SQL 调用、模型用量和审计的唯一权威；Python 只维护可丢弃或可重建的技术 checkpoint，不获得业务库凭据。

命名规则：跨运行时契约统一使用 `run_id`；PostgreSQL 为兼容 V1 继续使用物理列 `agent_run_id`。本文表格中的 `agent_run_id` 均表示 `run_id -> agent_runs.agent_run_id` 的数据库映射，不是第二个运行标识。

## 2. 设计原则

1. 只追加：新建表、新增可兼容列和索引，不重写、重命名或删除 V1 列、约束和索引。
2. 先兼容后收紧：先允许旧 Java 版本继续写 V1，再双写、回填、验证，最后启用新读路径和非空约束。
3. 副作用可去重：dispatch、event、cancel、Tool、SQL 和 model call 都有稳定逻辑 ID。
4. canonical event 才能入 inbox：Java 先查 event ID/hash，再校验 active dispatch 与序号；接受后才在同一事务写 inbox、推进权威状态和 `last_event_seq`，拒绝只写独立 rejection audit。
5. 终态不回退：完成、失败和取消并发时，以 Java 首次接受的合法终态为准。
6. deadline 是绝对时间：使用 `TIMESTAMPTZ`，不以本地 duration 替代跨进程截止时间。

## 3. Dispatch 与运行游标

### 3.1 新表 `agent_run_dispatches`

一条记录表示一次逻辑 dispatch。网络重试复用同一 `dispatch_id`；重新派发使用新 `dispatch_id` 并递增 `attempt`。

| 字段 | 类型目标 | 约束目标 | 说明 |
|---|---|---|---|
| `agent_run_dispatch_id` | `BIGINT` | PK | 应用层 Snowflake 主键 |
| `agent_run_id` | `BIGINT` | FK, NOT NULL | 契约 `run_id` 的物理映射 |
| `dispatch_id` | `VARCHAR(64)` | UNIQUE, NOT NULL | RunCommand 幂等键 |
| `attempt` | `INT` | NOT NULL | 同一 Run 从 1 递增 |
| `deadline_at` | `TIMESTAMPTZ` | NOT NULL | 本次 dispatch 截止时间 |
| `status` | `VARCHAR(32)` | NOT NULL | 技术执行状态 `pending/accepted/running/completed/failed/cancelled` |
| `request_hash` | `VARCHAR(71)` | NOT NULL | 按契约 JCS + SHA-256 计算的 RunCommand 摘要 |
| `last_event_seq` | `BIGINT` | NOT NULL, default 0 | 当前 dispatch 已连续接受的最大序号 |
| `dispatch_arbitration_state` | `VARCHAR(32)` | NOT NULL | `active/superseded/expired`；与技术执行状态分离 |
| `active_epoch` | `BIGINT` | NOT NULL | Run 内单调递增的 dispatch 仲裁 epoch |
| `fencing_token` | `VARCHAR(128)` | UNIQUE, NOT NULL | Java 签发并持久化的内部 fence；不进入 V1 wire |
| `admission_epoch` | `BIGINT` | NOT NULL | 创建 dispatch 时捕获的 Run admission epoch |
| `accepted_at` | `TIMESTAMPTZ` | NULL | Python 接受时间 |
| `finished_at` | `TIMESTAMPTZ` | NULL | dispatch 技术终止时间 |
| `created_at/updated_at` | `TIMESTAMPTZ` | NOT NULL | 审计时间 |

约束目标：

- `(agent_run_id,attempt)` 唯一，防止同一 Run 出现两个相同 attempt。
- `(agent_run_id,active_epoch)` 唯一，且 `active_epoch >= 1`。
- PostgreSQL partial unique index `UNIQUE (agent_run_id) WHERE dispatch_arbitration_state='active'` 强制每个 Run 至多一个 active dispatch；应用锁不能替代该约束。
- `dispatch_id` 全局唯一，便于回调和排障。
- `fencing_token` 全局唯一；`attempt >= 1`，`last_event_seq >= 0`，`admission_epoch >= 0`。
- `deadline_at` 不要求晚于数据库当前时间才能插入，因为过期命令仍可能作为拒绝审计落库；是否执行由应用层决定。

`last_event_seq` 放在 dispatch 记录而不是 `agent_runs`，因为契约规定序号作用域为 `dispatch_id`。`agent_runs.active_dispatch_id` 可作为可空 FK 指针，但必须与 partial unique index、active epoch 在同一锁定 Run 事务内更新，不能代替 dispatch 历史。创建新 active dispatch 时，原 active 原子转为 `superseded`；deadline 仲裁转为 `expired`；旧 token 永久失效，不能重新激活。

### 3.2 Run admission 与 cancellation epoch

V2 在 `agent_runs` 追加下列兼容列；它们是阶段 4 主链路的前置条件，不是后续增量：

| 字段 | 类型目标 | 约束目标 | 说明 |
|---|---|---|---|
| `active_dispatch_id` | `BIGINT` | NULL, FK | 当前 active dispatch 指针；终态或关闭 admission 后可为空 |
| `admission_state` | `VARCHAR(16)` | NOT NULL, default `open` | `open/closed`；关闭后禁止新 dispatch/invocation |
| `admission_epoch` | `BIGINT` | NOT NULL, default 0 | admission 关闭、重开或运维 fence 时单调递增 |
| `cancellation_epoch` | `BIGINT` | NOT NULL, default 0 | 每次首次接受新的取消意图时单调递增 |
| `sse_last_stream_seq` | `BIGINT` | NOT NULL, default 0 | 在 Run 行锁内分配 SSE 恢复序号 |

所有 epoch 均有非负 CHECK。dispatch 与 invocation 创建时复制当前 `admission_epoch`；invocation 还复制 `cancellation_epoch` 和 active dispatch `fencing_token`。进入执行器前、提交业务副作用前和持久化结果前必须重新比较 Run 当前 epoch、active dispatch 与 token，任一不匹配即 fenced，禁止新副作用。

### 3.3 Dispatch 查询索引

- `UNIQUE (agent_run_id) WHERE dispatch_arbitration_state='active'`：单 active dispatch 硬约束。
- `UNIQUE (agent_run_id,active_epoch)`、`UNIQUE (agent_run_id,attempt)`、`UNIQUE (dispatch_id)`、`UNIQUE (fencing_token)`：仲裁与幂等。
- `(dispatch_arbitration_state,deadline_at)`：过期扫描；`(agent_run_id,created_at DESC)`：运行历史；`(status,updated_at)`：技术状态对账。

## 4. Event inbox

### 4.1 新表 `runtime_event_inbox`

| 字段 | 类型目标 | 约束目标 | 说明 |
|---|---|---|---|
| `runtime_event_inbox_id` | `BIGINT` | PK | 应用层 Snowflake 主键 |
| `agent_run_id` | `BIGINT` | FK, NOT NULL | 契约 `run_id` 的物理映射 |
| `dispatch_id` | `VARCHAR(64)` | FK, NOT NULL | 对应 dispatch |
| `event_id` | `VARCHAR(64)` | NOT NULL | RunEvent 稳定 ID |
| `event_seq` | `BIGINT` | NOT NULL | dispatch 内从 1 递增 |
| `event_type` | `VARCHAR(64)` | NOT NULL | 内部事件类型 |
| `occurred_at` | `TIMESTAMPTZ` | NOT NULL | Python 发生时间 |
| `received_at` | `TIMESTAMPTZ` | NOT NULL | Java 接收时间 |
| `payload_json` | `JSONB` | NOT NULL | 原始已校验 payload |
| `request_hash` | `VARCHAR(71)` | NOT NULL | 覆盖完整 RunEvent 稳定字段，不只是 payload |
| `processing_status` | `VARCHAR(32)` | NOT NULL | `accepted/applied`；拒绝与 duplicate 不插入本表 |
| `applied_at` | `TIMESTAMPTZ` | NULL | 已推进状态/审计的时间 |
| `created_at/updated_at` | `TIMESTAMPTZ` | NOT NULL | 审计时间 |

逻辑幂等键必须是 `(run_id,event_id)`；物理唯一约束对应 `(agent_run_id,event_id)`。另设 `(dispatch_id,event_seq)` 唯一约束，防止 active dispatch 的同一序号对应不同 canonical event。`dispatch_id` 必须通过复合 FK 或事务校验映射到同一 `agent_run_id`。本表只保存已通过 schema/hash、active dispatch、attempt、fencing、严格连续 seq 和状态迁移校验的 canonical event。

### 4.2 新表 `runtime_event_rejections`

拒绝审计与 canonical inbox 分离，避免 gap、旧 dispatch 或同序号冲突预占 inbox 唯一键：

| 字段 | 类型目标 | 约束目标 | 说明 |
|---|---|---|---|
| `rejection_id` | `VARCHAR(64)` | PK | 每次拒绝审计的唯一 ID；本表唯一的幂等/冲突无关唯一键 |
| `agent_run_id` | `BIGINT` | FK, NOT NULL | 已验证真实 Run；pre-run 错误仍走 `protocol_error_audits` |
| `dispatch_id` | `VARCHAR(64)` | NOT NULL | 原 envelope dispatch |
| `attempt` | `INT` | NOT NULL | 原 envelope attempt |
| `event_id` | `VARCHAR(64)` | NOT NULL | 原 envelope event ID |
| `event_seq` | `BIGINT` | NOT NULL | 原 envelope seq |
| `request_hash` | `VARCHAR(71)` | NOT NULL | 收到并已通过格式校验的 canonical digest |
| `reason` | `VARCHAR(128)` | NOT NULL | 结构化拒绝原因，如 `same_seq_different_event/old_dispatch/gap/out_of_order/state_conflict` |
| `error_code` | `VARCHAR(64)` | NOT NULL | 返回的既有 V1 RuntimeError code |
| `raw_envelope_json` | `JSONB` | NOT NULL | 脱敏后的原始 envelope；不得包含 JWT 或 Secret |
| `created_at` | `TIMESTAMPTZ` | NOT NULL | 拒绝时间 |

除 `rejection_id` 主键外，不得对 run、dispatch、event、seq 或 hash 建唯一约束；同一拒绝重放可以留下多条审计尝试。索引只用于查询：`(agent_run_id,created_at DESC)`、`(dispatch_id,event_seq,created_at DESC)`、`(error_code,created_at DESC)`。

### 4.3 接受与拒绝事务

schema、Service JWT 和 `request_hash` 重算通过后，Java 在一个数据库事务中按固定顺序处理：

1. 先按 `(agent_run_id,event_id)` 查询 `runtime_event_inbox`。已知 event 且 hash 相同返回 duplicate success，不新增 inbox/rejection/SSE；hash 不同写 rejection audit，并返回稳定 `RUNTIME_EVENT_IDEMPOTENCY_CONFLICT`。
2. 锁定 AgentRun 与 `dispatch_id`，确认 run/attempt 映射、`dispatch_arbitration_state=active`、`agent_runs.active_dispatch_id`、active/admission epoch 和 fencing token 一致。旧、superseded 或 expired dispatch 写 rejection audit，返回稳定 `RUNTIME_STATE_CONFLICT`。
3. 查询 `(dispatch_id,event_seq)`；若已被不同 event 占用，写 rejection audit，返回稳定 `RUNTIME_EVENT_OUT_OF_ORDER`。再比较游标：旧序号写 `RUNTIME_EVENT_OUT_OF_ORDER` rejection，gap 写 `RUNTIME_EVENT_GAP` rejection。两者都不得插入 inbox。
4. 校验 event type、终态和 AgentRun 状态迁移；失败写 rejection audit 与稳定 `RUNTIME_STATE_CONFLICT`，不得插入 inbox。
5. 只有全部校验通过且 `event_seq=last_event_seq+1` 时，插入 `runtime_event_inbox`，应用 AgentRun/cancellation/audit/model usage，写前端可见事件的 SSE outbox，标记 inbox `applied` 并推进 `last_event_seq`。
6. 接受路径提交后返回成功；拒绝路径提交 rejection 与稳定 RuntimeError 后返回对应 409。任何回滚都不得提前发送 SSE。

稳定 RuntimeError 的 `error_id` 由 `run_id/dispatch_id/attempt/event_id/event_seq/error_code/reason` 的规范化组合确定性派生，同一拒绝重放返回相同 envelope 和 `request_hash`；本地生成的错误仍按 `(agent_run_id,error_id)` 写/重放 `runtime_error_inbox`。gap envelope 未进入 inbox，因此缺失事件补齐后，生产者可原样重试该 event 并在届时被接受。

### 4.4 RuntimeError inbox

新表 `runtime_error_inbox` 持久化 RuntimeError 的幂等语义：

| 字段 | 类型目标 | 约束目标 | 说明 |
|---|---|---|---|
| `runtime_error_inbox_id` | `BIGINT` | PK | 应用层 Snowflake 主键 |
| `agent_run_id` | `BIGINT` | FK, NOT NULL | 契约 `run_id` 的物理映射 |
| `error_id` | `VARCHAR(64)` | NOT NULL | 稳定错误 ID |
| `related_event_id` | `VARCHAR(64)` | NULL | 关联 RunEvent |
| `related_event_seq` | `BIGINT` | NULL | 关联事件序号；与 ID 成对为空或非空 |
| `request_hash` | `VARCHAR(71)` | NOT NULL | 完整 RuntimeError canonical digest |
| `error_json` | `JSONB` | NOT NULL | 完整脱敏 RuntimeError |
| `received_at` | `TIMESTAMPTZ` | NOT NULL | 首次接收时间 |

物理唯一约束为 `(agent_run_id,error_id)`。同键同 hash 返回原错误记录，不重复推进状态；不同 hash 返回 `RUNTIME_ERROR_IDEMPOTENCY_CONFLICT`。`related_event_id/related_event_seq` 只用于关联，不替代错误自己的幂等键。

运行前认证、版本协商和 JSON 解析错误不写本表，也不伪造 `agent_run_id`。新增 `protocol_error_audits`：

| 字段 | 类型目标 | 约束目标 | 说明 |
|---|---|---|---|
| `protocol_error_audit_id` | `BIGINT` | PK | 应用层 Snowflake 主键 |
| `request_id` | `VARCHAR(64)` | UNIQUE, NOT NULL | pre-run 重试关联键 |
| `error_id` | `VARCHAR(64)` | UNIQUE, NOT NULL | PreRunProtocolError ID |
| `trace_id` | `VARCHAR(64)` | NOT NULL | 传输 Trace |
| `request_fingerprint` | `VARCHAR(71)` | NOT NULL | method/path/version/body hash/error code 摘要 |
| `error_hash` | `VARCHAR(71)` | NOT NULL | PreRunProtocolError canonical digest |
| `error_json` | `JSONB` | NOT NULL | 脱敏错误 envelope |
| `expires_at` | `TIMESTAMPTZ` | NOT NULL | 默认首次接收后 7 天 |
| `created_at` | `TIMESTAMPTZ` | NOT NULL | 首次接收时间 |

该表没有 `agent_run_id` 外键。相同 `request_id/fingerprint` 返回原 envelope；fingerprint 不同返回 `PRE_RUN_IDEMPOTENCY_CONFLICT`。在线查询默认只命中 `expires_at` 未过期记录；到期记录由审计归档/清理任务按安全策略处理。

## 5. 取消请求

### 5.1 新表 `agent_run_cancellations`

| 字段 | 类型目标 | 约束目标 | 说明 |
|---|---|---|---|
| `agent_run_cancellation_id` | `BIGINT` | PK | 应用层 Snowflake 主键 |
| `agent_run_id` | `BIGINT` | FK, NOT NULL | 契约 `run_id` 的物理映射 |
| `dispatch_id` | `VARCHAR(64)` | FK, NOT NULL | 被取消的 dispatch |
| `cancel_id` | `VARCHAR(64)` | UNIQUE, NOT NULL | CancelCommand 幂等键 |
| `request_hash` | `VARCHAR(71)` | NOT NULL | 完整 CancelCommand canonical digest |
| `cancellation_epoch` | `BIGINT` | NOT NULL | 本次新取消意图在 Run 上递增后的 epoch |
| `reason` | `VARCHAR(32)` | NOT NULL | 契约取消原因 |
| `requested_by` | `VARCHAR(32)` | NOT NULL | `user/operator/system/deadline` |
| `requested_at` | `TIMESTAMPTZ` | NOT NULL | Java 接受取消请求时间 |
| `deadline_at` | `TIMESTAMPTZ` | NOT NULL | CancelCommand 截止时间 |
| `status` | `VARCHAR(32)` | NOT NULL | `requested/dispatched/acknowledged/rejected/resolved` |
| `ack_event_id` | `VARCHAR(64)` | NULL | 对应 `run.cancel_acknowledged` |
| `resolved_at` | `TIMESTAMPTZ` | NULL | Java 完成取消裁决时间 |
| `error_code` | `VARCHAR(64)` | NULL | 取消失败原因 |
| `created_at/updated_at` | `TIMESTAMPTZ` | NOT NULL | 审计时间 |

V2 不给 `agent_runs.status` 增加 `cancelling`，因此无需改动 V1 CHECK。取消中状态由 cancellation 记录派生展示；收到确认后，Java 仅在当前状态允许时写 `cancelled`。若 `completed` 或 `failed` 已先成为合法终态，取消记录仍保留为 `resolved`，但不得覆盖终态。

相同 `cancel_id` 且 `request_hash` 一致时返回原取消记录且不得再次递增 epoch；不同 hash 返回 `RUNTIME_CANCEL_IDEMPOTENCY_CONFLICT`。首次接受新取消意图时，锁定 AgentRun、递增 `agent_runs.cancellation_epoch`、写 cancellation 记录并关闭后续 invocation admission，全部在同一事务完成。

## 6. Tool invocation 幂等

在 V1 `tool_calls` 上只追加以下目标列：

| 字段 | 类型目标 | 约束目标 | 说明 |
|---|---|---|---|
| `dispatch_id` | `VARCHAR(64)` | 后期 NOT NULL, FK | 产生 proposal 的 dispatch |
| `attempt` | `INT` | 后期 NOT NULL | dispatch attempt 快照 |
| `invocation_id` | `VARCHAR(64)` | UNIQUE | ToolProposal 逻辑调用 ID |
| `idempotency_key` | `VARCHAR(128)` | 后期 NOT NULL | Java 副作用幂等键 |
| `request_hash` | `VARCHAR(71)` | 后期 NOT NULL | 覆盖契约定义的完整 ToolProposal 稳定字段 |
| `deadline_at` | `TIMESTAMPTZ` | 后期 NOT NULL | 工具截止时间 |
| `admission_epoch` | `BIGINT` | 后期 NOT NULL | invocation 创建时捕获的 Run admission epoch |
| `cancellation_epoch` | `BIGINT` | 后期 NOT NULL | invocation 创建时捕获的 Run cancellation epoch |
| `dispatch_fencing_token` | `VARCHAR(128)` | 后期 NOT NULL | 创建时捕获的 active dispatch fence |
| `execution_state` | `VARCHAR(32)` | 后期 NOT NULL | `pending/leased/succeeded/failed/unknown` |
| `owner_token` | `VARCHAR(128)` | NULL | 当前 executor 的随机不可猜 owner token |
| `lease_until` | `TIMESTAMPTZ` | NULL | `leased` 状态的租约截止时间 |
| `lease_attempt_count` | `INT` | 后期 NOT NULL, default 0 | 首次领取与接管次数 |
| `policy_decision` | `VARCHAR(32)` | NULL | `allow/deny/require_approval` |
| `result_status` | `VARCHAR(32)` | NULL | 原始 ToolResult 状态，补足 V1 状态枚举 |
| `result_hash` | `VARCHAR(71)` | NULL | ToolResult canonical digest |
| `replayed_from_tool_call_id` | `BIGINT` | NULL, FK | 重放结果来源；正常设计应复用原行 |

物理唯一约束目标：`invocation_id` 全局唯一，以及 `(agent_run_id,idempotency_key)` 唯一。重复请求摘要一致时从 `output_json/result_status/result_hash` 返回原 ToolResult；摘要不一致时返回 `TOOL_IDEMPOTENCY_CONFLICT`。`result_hash` 冲突返回 `TOOL_RESULT_CONFLICT`。V1 `status` 按契约映射；V1 无法表达的 `denied/confirmation_required` 由 `result_status` 和完整 `output_json` 保存。任何 Tool adapter 执行必须发生在 `pending` 幂等占位成功建立、CAS 转为 `leased` 并取得 owner token 之后。

租约约束：`pending` 不得有 owner/lease；`leased` 必须同时有 owner/lease；`succeeded/failed/unknown` 必须清空 active lease。领取使用 `UPDATE ... WHERE execution_state='pending'` CAS；接管使用 `UPDATE ... WHERE execution_state='leased' AND lease_until < now()` CAS，并更换 owner token。只有 registry 明确 `idempotent=true` 且 `retryable=true`、deadline 未过期、Run epochs 与 dispatch fence 仍匹配、且外部副作用可由 provider idempotency key 证明安全时才允许接管。结果不确定时转 `unknown`，禁止自动执行，由人工对账收敛。索引至少包含 `(execution_state,lease_until)`、`(agent_run_id,execution_state,updated_at)`。

## 7. SQL invocation 幂等

在 V1 `sql_query_audits` 上只追加：

| 字段 | 类型目标 | 约束目标 | 说明 |
|---|---|---|---|

物理唯一约束目标：`invocation_id` 全局唯一，以及 `(agent_run_id,idempotency_key)` 唯一。重放不得再次执行 SQL；Java 从 `result_status/result_hash/result_json` 返回原 SqlResult。V1 `status` 无法表达的 `failed/timeout/cancelled` 仍写兼容状态 `rejected`，原状态只从 V2 列/JSON 读取。请求摘要冲突返回 `SQL_IDEMPOTENCY_CONFLICT`，结果摘要冲突返回 `SQL_RESULT_CONFLICT`。

SQL invocation 使用与 Tool 相同的 owner/lease CHECK、CAS 领取、epoch/fence 重检和 `(execution_state,lease_until)` 索引。只读查询只有在数据库/驱动能证明失败前未提交结果或同一 execution 可安全重放时才允许过期 lease 接管；无法证明时写 `unknown`，不得再次执行 SQL，按契约映射为稳定失败或不可重试 RuntimeError，并进入人工对账。

## 8. SSE transactional outbox

阶段 1 新建 `agent_run_sse_outbox`；它与 AgentRun、canonical inbox 和业务投影同事务写入，是阶段 4 前端 SSE 主链路的硬依赖：

| 字段 | 类型目标 | 约束目标 | 说明 |
|---|---|---|---|
| `agent_run_sse_outbox_id` | `BIGINT` | PK | 应用层 Snowflake 主键 |
| `agent_run_id` | `BIGINT` | FK, NOT NULL | 对应契约 `run_id` |
| `source_event_key` | `VARCHAR(160)` | UNIQUE, NOT NULL | 稳定投影键；同一 canonical event/投影不得重复建行 |
| `sse_event_id` | `VARCHAR(64)` | UNIQUE, NOT NULL | SSE `id:` 与浏览器 `Last-Event-ID` 的稳定值 |
| `stream_seq` | `BIGINT` | NOT NULL | 每个 Run 从 1 严格递增的持久化恢复游标 |
| `event_type` | `VARCHAR(64)` | NOT NULL | 面向前端的 `run.*` 事件类型 |
| `payload_json` | `JSONB` | NOT NULL | 已授权、已脱敏的完整 SSE payload |
| `status` | `VARCHAR(32)` | NOT NULL | `pending/leased/sent/dead_letter` |
| `owner_token` | `VARCHAR(128)` | NULL | 当前 publisher owner token |
| `lease_until` | `TIMESTAMPTZ` | NULL | `leased` 状态租约截止时间 |
| `retry_count` | `INT` | NOT NULL, default 0 | 已完成的失败重试次数 |
| `next_retry_at` | `TIMESTAMPTZ` | NOT NULL | 下次可领取时间 |
| `last_error_code` | `VARCHAR(64)` | NULL | 脱敏后的最近投递错误 |
| `created_at/updated_at` | `TIMESTAMPTZ` | NOT NULL | 审计时间 |
| `sent_at` | `TIMESTAMPTZ` | NULL | 首次确认发送成功时间 |

约束与索引固定为：`UNIQUE (agent_run_id,stream_seq)`、`UNIQUE (sse_event_id)`、`UNIQUE (source_event_key)`；`stream_seq >= 1`、`retry_count >= 0`；`pending/sent/dead_letter` 不得持有 active lease，`leased` 必须同时有 owner/lease。领取索引为 `(status,next_retry_at,lease_until)`，断线恢复索引为 `(agent_run_id,stream_seq)`，运维索引为 `(status,updated_at)`。

Java 在锁定 AgentRun 时递增 `sse_last_stream_seq`，固化 `stream_seq/sse_event_id/source_event_key` 并与业务事务一起提交。publisher 以 CAS 从 `pending` 或已过期 `leased` 领取，接管时更换 owner token；发送失败递增 `retry_count` 并按有界退避设置 `next_retry_at`，超过策略进入 `dead_letter` 告警，不重做业务副作用。

SSE 端点接收 `Last-Event-ID` 时，先按唯一 `sse_event_id` 解析其 `(agent_run_id,stream_seq)`，再读取该 Run 中更大的 `stream_seq`；未知、跨 Run 或越权 ID 明确拒绝。投递语义为至少一次，客户端按 `sse_event_id` 去重后再推进 cursor。重放 canonical inbox event、publisher lease 接管或断线恢复都必须保持同一 `sse_event_id/stream_seq/payload_json`。

## 9. model_call_id

在 V1 `model_usage_logs` 上追加：

| `agent_run_id` | `BIGINT` | NULL, FK | 契约 `run_id` 的物理映射；兼容非 Run 模型任务 |

| `migration_state` | `VARCHAR(32)` | NOT NULL | `native/legacy_unexpanded/legacy_expanded` |
| `migration_audit_json` | `JSONB` | NOT NULL | 历史展开判定、原因和源摘要；不改写原值 |

新增 `model_call_attempts` 子表，一行对应一次 provider attempt：

| 字段 | 类型目标 | 约束目标 | 说明 |
|---|---|---|---|
| `model_call_attempt_id` | `BIGINT` | PK | 应用层 Snowflake 主键 |
| `model_usage_log_id` | `BIGINT` | FK, NOT NULL | 关联逻辑调用父记录 |
| `event_id` | `VARCHAR(64)` | NULL | native attempt 必填；legacy expanded 没有事件时为 null |
| `provider_attempt_id` | `VARCHAR(64)` | UNIQUE, NOT NULL | 本地单次 attempt ID |
| `provider_request_id` | `VARCHAR(128)` | NULL | 供应商单次请求 ID |
| `provider_code/model_name` | `VARCHAR` | NOT NULL | 实际供应商与模型 |
| `status` | `VARCHAR(32)` | NOT NULL | `success/failed/timeout/cancelled` |
| `usage_json` | `JSONB` | NOT NULL | token 键始终存在；失败/超时/取消时未知值为 null |
| `latency_ms` | `INT` | NOT NULL | 本次 attempt 耗时 |
| `cost_amount/cost_currency` | `NUMERIC/VARCHAR` | NULL | 未知成本成对为 null；确认零成本才写 0 |
| `usage_hash` | `VARCHAR(71)` | NOT NULL | model usage payload canonical digest |

| `contributes_to_parent` | `BOOLEAN` | NOT NULL | native 为 true；legacy expanded 固定 false |

ID 语义固定：契约 envelope 的 `request_id` 只追踪一次 HTTP 传输；`model_call_id` 标识逻辑模型调用；`provider_request_id` 标识供应商单次请求；`provider_attempt_id` 标识本地一次 attempt。`model_call_id` 在父表唯一，多个 attempt 通过子表关联同一父记录。

V1 `request_id` 在兼容期仍必须全局唯一。新 Java 创建父记录时使用 `request_id = model_call_id`，绝不写 RunEvent envelope `request_id`；这样 V1 唯一键与 V2 `model_call_id` 唯一键表达同一个逻辑调用。每个 native `run.model_usage` 事件按 `provider_attempt_id` 幂等写子表，并汇总/更新父表 usage、latency、cost 和最终 status。V2 不删除 `uk_model_usage_logs_request_id`。

### 9.1 V1 历史父行确定性迁移

所有 V1 历史 `model_usage_logs` 父行原样保留。迁移只追加稳定 `model_call_id = legacy-call:<model_usage_log_id>`、`migration_state` 和审计 JSON；不得改写原 `provider_code/model_name/status/usage_json/latency_ms/cost_amount/created_at/updated_at`。

默认路径是 `migration_state=legacy_unexpanded`，不创建 attempt 子记录。`migration_audit_json` 至少记录 `evaluated_at/source_row_hash/reasons[]`；原因码包括 `missing_provider`、`missing_model`、`non_standard_status:<value>`、`missing_latency`、`invalid_usage`、`missing_cost_currency`、`missing_started_at`、`missing_finished_at`。这不是迁移失败，且不阻止 Flyway V2 或阶段 5。

单行展开判定前必须执行全局唯一性预检：从全部 V1 历史父行读取非空 `provider_code` 和 `usage_json.provider_request_id`，按 `(provider_code,provider_request_id)` 分组。任何计数大于 1 的组，其整组父行全部标记 `legacy_unexpanded`，每行 `migration_audit_json.reasons` 增加 `duplicate_provider_request_id`，且整组不创建任何 attempt。不得按最小主键、最早时间或其他规则任意保留一条。该组完成审计后不再进入后续单行完整性判定。

只有通过上述全局唯一性预检，且同时满足以下条件，历史父行才可标记 `legacy_expanded` 并创建一个子记录：

1. V1 `provider_code/model_name` 非空。
2. V1 `status` 精确属于 `success/failed/timeout/cancelled`；`fallback`、空字符串和其他值均不映射，保持 `legacy_unexpanded`。
3. V1 `latency_ms` 非空且大于等于 0。
4. V1 `usage_json` 明确包含 `input_tokens/output_tokens/total_tokens`；success 时三者均为非负整数，其他标准状态允许 null，但未知值不转成 0。
5. V1 `cost_amount` 与 `usage_json.cost_currency` 成对有效；未知 currency 不猜测默认币种。
6. V1 `usage_json.started_at/finished_at` 均为有效 RFC 3339 时间且顺序合法；不得用 `created_at/updated_at` 冒充 attempt 起止时间。

满足条件时，`provider_attempt_id` 可确定性生成 `legacy-attempt:<model_usage_log_id>`，它只表示迁移记录 ID，不声称是历史运行时产生的 ID；`provider_request_id` 仅在 `usage_json.provider_request_id` 明确存在时复制，否则为 null；`event_id=null`、`source_kind=legacy_expanded`、`contributes_to_parent=false`。迁移创建该子记录只用于保留可证明的历史 attempt 细节，不触发父聚合，父行所有既有 usage/cost/latency/status 值保持不变。

attempt 约束与父聚合固定如下：

- `status` 只允许 `success/failed/timeout/cancelled`；`latency_ms >= 0`。
- `provider_attempt_id` 全局唯一；`provider_request_id` 非空时 `(provider_code,provider_request_id)` 唯一。两层去重成功后才参与父聚合。
- Native attempt 继续受非空 `(provider_code,provider_request_id)` 唯一约束；历史重复组被整体排除不会削弱、绕过或替代该约束。
- 父 token/cost 只对 `contributes_to_parent=true` 的已去重 native attempts 的非空值分别求和；没有已知值则为 null，存在部分未知值则保存已知和并在父 `usage_json` 标记 `complete=false/cost_complete=false`。未知值不按 0 计算。
- 父 `latency_ms` 是所有 `contributes_to_parent=true` 的已去重 native attempts 的 `latency_ms` 之和。
- 任一 attempt 成功则父 `status=success` 且不降级；否则取 `finished_at` 最新 attempt 的状态，同时间按 `provider_attempt_id` 字典序最大者裁决。
- cost currency 必须一致；不同币种返回 `MODEL_USAGE_CURRENCY_CONFLICT`，不更新父聚合。

## 10. 迁移与发布顺序

### 阶段 0：前置确认

- 固定并校验 V1 checksum，确认 24 张表和现有约束与生产一致。
- 用 PostgreSQL Testcontainers 建立“空库 V1”和“含代表数据 V1”两类升级 fixture。
- 冻结 V1 文件，任何差异都作为独立修复迁移处理。

### 阶段 1：纯追加结构

- 新建 `agent_run_dispatches`、`runtime_event_inbox`、`runtime_event_rejections`、`agent_run_cancellations`、`agent_run_sse_outbox`、RuntimeError/pre-run audit 和 model attempt 表。
- 给 `agent_runs` 追加 active dispatch、admission/cancellation epoch 与 SSE cursor；给 Tool/SQL 追加 invocation epoch/fence、`execution_state/owner_token/lease_until/lease_attempt_count`；给 model usage 追加兼容列。
- 建立单 active dispatch partial unique、dispatch epoch/token、canonical event、rejection 查询、invocation lease pickup、SSE stable ID/stream/outbox pickup 等索引和 CHECK。大表在线索引方式在实现评审时结合 Flyway 事务和锁预算决定，但不能把这些约束推迟到阶段 4 之后。

旧 Java 版本在此阶段仍可只写 V1 字段；新表为空不影响旧读取。

### 阶段 2：部署兼容 Java

- Java 先支持读取新旧两种记录，并为新 Run 写 active dispatch/epochs/cancellation；只把已接受 canonical event 写 `runtime_event_inbox`，拒绝写 `runtime_event_rejections`。
- Tool/SQL 新调用写 invocation epoch/fence/lease 字段；历史/旧路径仍允许空值。
- 前端可见事件同事务写 SSE outbox，publisher 与 Last-Event-ID 路径先在 feature flag 下运行。
- 模型逻辑调用写唯一 `model_call_id`，兼容期令 V1 `request_id = model_call_id`；provider attempt 写子表。
- Python 尚未上线时用 Runtime stub 验证最小 dispatch/event/cancel。

### 阶段 3：回填与对账

- 为迁移前仍可能运行的 AgentRun 建立明确的 legacy dispatch，attempt/active epoch 从 1 开始并签发 fence；每个 Run 最多一个 active。已终态 Run 不伪造事件历史，admission 置为 closed。
- Tool/SQL 历史行只生成可识别的 legacy invocation/idempotency 值，不把它们伪装成真实跨运行时调用。
- 为每个历史模型父行回填稳定 `model_call_id`；先执行全局 `(provider_code,provider_request_id)` 重复组预检并整组标记，再对剩余行执行 9.1 单行展开判定。不满足条件时标记 `legacy_unexpanded` 且不创建子记录，满足时创建 `contributes_to_parent=false` 的 legacy 子记录。所有路径都不得改变原 V1 父值或 `request_id`。
- 对账空值、重复值、孤儿外键、attempt/active epoch 连续性、单 active 约束、epoch/fence、canonical inbox/rejection 分流、lease 状态和 outbox cursor；失败即停止主链路启用。

### 阶段 4：启用新主链路

- 硬门禁：本 V2 定义的 dispatch active/fencing、admission/cancellation epoch、Tool/SQL owner lease、`runtime_event_rejections` 和 SSE outbox 表/列/索引/CHECK 已全部存在，阶段 0-3 的 PostgreSQL Testcontainers fixtures 与兼容矩阵全部通过；任一缺失都不得启用主链路。
- 开启 Java Runtime Client 与 canonical event inbox/rejection audit 事务消费。
- 开启 Tool/SQL invocation 幂等、epoch/fence/lease 门禁和取消对账任务。
- 开启 transactional SSE outbox publisher 与 Last-Event-ID 恢复；禁止回退为仅内存 after-commit 发送。
- Python 生产者只发送契约 V1 字段；Java 校验 Service JWT、版本、deadline 和幂等。
- 观察至少一个完整兼容窗口，覆盖最大 Run/checkpoint 保留期。

### 阶段 5：收紧约束

- 收紧门槛：所有生产/回滚候选 Java 版本都已写新列，旧版本已退出可回滚集合，回填与 Testcontainers 兼容矩阵全部通过。门槛未满足时新增列必须保持可空，不得进入本阶段。
- 门槛满足后验证唯一约束并按新写路径收紧 epoch/fence/lease/outbox 非空与 CHECK；从此不能直接回滚到不写新列的旧 Java。
- 非空与 CHECK 只约束新写父行、native attempt，以及实际创建的 legacy expanded attempt。历史父行可以永久保持 `legacy_unexpanded` 且没有任何子 attempt；阶段 5 不增加“每个父行至少一个子记录”的约束。
- 切换读取逻辑以新表/新列为权威，保留 V1 字段和兼容读取至少一个发布周期。
- 不在本阶段删除 V1 `request_id` 唯一索引或增加 `cancelling` 状态。

## 11. 兼容窗口

| 组合 | 必须支持的行为 |
|---|---|
| 旧 Java + V2 schema（阶段 1-4） | 既有 V1 行允许新增列为空，旧 Java 可继续写 V1；V2 新表/索引存在但 feature flag 关闭，这是直接应用回滚窗口 |
| 新 Java + 无 Python | local-stub/Runtime stub 可启动；Run 明确失败或排队，不伪造成功 |
| 新 Java + 旧契约生产者 | 在 V1 契约范围内兼容；缺少目标必填字段时明确拒绝，不猜默认业务语义 |

| 回滚 Java + V2 schema（阶段 5 后） | 禁止直接回滚到不写新列的旧 Java；只能回滚到已兼容新列的版本，或先执行前向兼容迁移恢复可空/兼容默认 |

消息契约版本 `v1` 与 Flyway 数据库版本 `V2` 属于不同版本域。兼容窗口结束前不得把 V2 数据列的存在误当成 Python Runtime 已上线。

## 12. 回滚策略

1. 阶段 1-4 首选应用回滚、数据结构前向保留：先关闭 admission 并递增 admission epoch，停止创建新 dispatch/invocation，再关闭 Runtime 主链路；不删除 V2 表列、索引或已接收审计，成立前提是兼容列仍可空。
2. 有界 drain 后将旧 active dispatch 原子标记 `superseded/expired` 并失效 fence；进行中的 Run 只按现有状态机收敛，不能批量改成 `completed` 或发明暂停状态。
3. canonical inbox、rejection、cancellation、invocation lease/unknown、SSE outbox 和 model call 数据都是审计证据，回滚期间只读保留；未发送 outbox 可由兼容 publisher drain 或明确转 dead-letter，不得删除后假装已投递。
4. 阶段 5 收紧后，不允许直接回滚到不写新列的旧 Java。必须先发布新的前向兼容 Flyway 迁移，把目标列恢复为可空或提供不会伪造业务语义的兼容默认，并通过旧 Java smoke test；之后才能回滚应用。不改写 V2，不手工篡改 Flyway history。
5. 只有在确认 V2 从未承载生产数据且经过单独审批时，才考虑结构删除；本文不提供破坏性 down SQL。
6. 重新前滚时从保留的 active epoch/fence、`last_event_seq`、canonical inbox/rejection、invocation state 和 SSE `stream_seq` 对账恢复，不能清空后从 1 重放副作用。

### 13.1 Flyway 基线

- 阶段 1-4（新增列仍可空）的旧 Java schema 访问 smoke test 在 V2 数据库上通过。

### 13.2 Dispatch 与 event

- 同一 `dispatch_id` + 同摘要重复写只得到一条 dispatch；摘要变化冲突。
- 同一 Run 的 `(agent_run_id,attempt)` 冲突被数据库约束阻止。
- 两个并发 active dispatch 只能有一个提交成功；`active_epoch/fencing_token` 唯一，旧 dispatch 原子进入 `superseded/expired`。
- 相同逻辑 `(run_id,event_id)`，即物理 `(agent_run_id,event_id)`，只能进入 canonical inbox 一次。
- 相同 event key、不同 `request_hash` 返回 `RUNTIME_EVENT_IDEMPOTENCY_CONFLICT` 并写 rejection；event hash 覆盖 event envelope 稳定字段和完整 payload。
- 同一 active `(dispatch_id,event_seq)` 不允许两个不同 canonical event；冲突 event 只写 rejection，不预占/覆盖 inbox。
- 旧/superseded/expired dispatch、旧序号和 gap 都只写 `runtime_event_rejections`，不写 `runtime_event_inbox`；重放返回稳定 RuntimeError。
- gap 补齐后原 envelope 可重试并进入 inbox；并发连续事件只推进严格连续 `last_event_seq`，AgentRun 不回退。
- 事务回滚时 inbox、AgentRun 状态和 `last_event_seq` 一起回滚。

### 13.3 取消与终态竞争

- 相同 `cancel_id` 重试只保留一个取消请求。
- 相同 `cancel_id`、不同 `request_hash` 返回 `RUNTIME_CANCEL_IDEMPOTENCY_CONFLICT`。
- 首次取消在同一 Run 锁内只递增一次 `cancellation_epoch`；相同 cancel 重放不重复递增。
- admission 关闭/epoch 提升后拒绝新 dispatch 和 invocation；已捕获旧 epoch/fence 的 executor 在副作用提交前被拒绝。
- `run.cancel_acknowledged` 前不写不存在的 `cancelling`。
- completed/failed/cancelled 并发下只接受首个合法终态，取消审计仍完整。
- 取消后拒绝创建新的 Tool/SQL invocation，但已存在 invocation 可完成对账。

### 13.3.1 错误分流

- 运行内 RuntimeError 必须用真实 `run_id` 命中 AgentRun，并按 `(agent_run_id,error_id)` 去重写 `runtime_error_inbox`。
- 同 RuntimeError key 同 hash 只处理一次；不同 hash 返回 `RUNTIME_ERROR_IDEMPOTENCY_CONFLICT`。Python checkpoint 恢复后也不得重复应用同一错误。
- 认证、版本和无效 JSON 构造 PreRunProtocolError，不含 `run_id`，只写无 AgentRun FK 的 `protocol_error_audits`。
- 相同 pre-run `request_id/fingerprint` 返回原 envelope；不同 fingerprint 冲突。将时间推进超过 7 天后，在线幂等记录按留存策略归档/清理。

### 13.4 Tool 与 SQL 幂等

- 相同 `invocation_id` 或 `(agent_run_id,idempotency_key)` 的重复 proposal 返回原结果且执行计数为 1。
- 相同键、不同 request hash 返回对应冲突错误。
- Tool/SQL 相同 invocation、不同 result hash 返回 `TOOL_RESULT_CONFLICT/SQL_RESULT_CONFLICT`，原 Result JSON 不被覆盖。
- 逐项断言 ToolResult/SqlResult 到 V1 status、V2 `result_status` 和 JSON 的映射及重放内容。
- deadline 过期不进入 adapter/SQL executor，但审计记录可查询。
- 并发领取只允许一个 `owner_token` 从 pending 进入 leased；未过期 lease 不可被抢占。
- 过期 lease 仅在 idempotent/retryable、deadline、epoch/fence 和 provider idempotency 全部通过时接管；每次接管更换 owner token 并递增 attempt count。
- 无法证明副作用结果时进入 `unknown`，不会自动再次执行；人工对账可查询完整审计。
- SQL Guard 拒绝、多语句、敏感字段、超时和成功路径都保留唯一审计。

### 13.5 SSE outbox

- 同一 canonical event/投影的 `source_event_key` 只能生成一条 outbox，`sse_event_id` 与 `(agent_run_id,stream_seq)` 均唯一稳定。
- AgentRun/inbox/业务投影/outbox 同事务提交或回滚；提交后、发送前崩溃由过期 lease 接管，不产生新 SSE ID。
- publisher 失败递增 retry、遵守 `next_retry_at`，超过策略进入 `dead_letter`；不重新应用业务事件。
- `Last-Event-ID` 能解析稳定 `sse_event_id` 并只返回同 Run 更大 `stream_seq`；未知、跨 Run、越权 ID 拒绝。
- 至少一次重发下客户端按 `sse_event_id` 去重，payload 和 stream cursor 不变化。

### 13.6 Model usage

- 构造 V1 fixture A：`provider_code/model_name/latency_ms/cost_amount` 全空、`usage_json={}`；迁移成功，父行获得唯一 `model_call_id` 和 `legacy_unexpanded` 审计，不生成子记录。
- 构造 V1 fixture B：`status=fallback` 及另一个任意非标准 status；迁移成功，原因记录 `non_standard_status`，不映射为 success/failed，不生成子记录。
- 构造 V1 fixture C：provider/model、标准 status、latency、完整 token、cost currency、started/finished 全部明确；只为该行生成 `source_kind=legacy_expanded/contributes_to_parent=false` 的确定性子记录。
- 构造 V1 fixture D：两个分别满足单行完整条件的父行使用相同非空 `(provider_code,usage_json.provider_request_id)`；迁移成功，二者都标记 `legacy_unexpanded`、都记录 `duplicate_provider_request_id`，无任何子 attempt，不允许只展开其中一行。
- 对 A/B/C/D 逐列断言 V1 父行 `provider_code/model_name/status/usage_json/latency_ms/cost_amount/created_at/updated_at/request_id` 迁移前后完全一致；fixture D 恰有两条 migration audit。
- 契约 envelope `request_id` 不写入模型唯一列；兼容期新父记录满足 `request_id = model_call_id`。
- 重复 `model_call_id` 被父表唯一键拒绝；同一逻辑调用的多个 `provider_attempt_id` 可关联同一父记录。
- 相同 `provider_attempt_id` 同 hash 只写一次，不同 hash 返回 `MODEL_USAGE_IDEMPOTENCY_CONFLICT`；父表 usage/cost/status 聚合结果正确。
- 覆盖 `success/failed/timeout/cancelled`：未知 token/cost 为 null，确认零值才为 0，所有状态 latency 非空且非负。
- 两次已去重 attempts 的 token/cost 与 latency 分别求和；部分未知值设置 completeness 标记；任一 success 时父状态保持 success，否则按最新 finished_at、再按 attempt ID 裁决。
- 非空 `(provider_code,provider_request_id)` 重复不得二次聚合；不同 currency 返回 `MODEL_USAGE_CURRENCY_CONFLICT` 且父聚合不变。
- 新建 native attempt 时继续验证非空 `(provider_code,provider_request_id)` 唯一约束；fixture D 的历史排除策略不改变 native 冲突行为。
- 阶段 5 约束启用后，A/B 历史父行没有子 attempt 仍可保留并正常查询；新增 native 写入和实际存在的 attempt 必须满足全部新约束。

### 13.7 回滚兼容

- 阶段 1-4 的 V2 schema 上启动上一兼容 Java 版本，验证可空新增列不破坏 V1 读写。
- 阶段 5 NOT NULL 后验证旧 Java 直接写入明确失败；执行前向兼容迁移恢复可空/兼容默认后，旧 Java smoke test 才允许通过。
- 关闭 feature flag 与 admission 后不再创建新 dispatch/invocation，但 canonical inbox、rejection、lease/unknown 和 outbox 审计可读。
- 应用回滚再前滚后，事件与 SSE 分别从持久化 `last_event_seq/stream_seq` 继续，副作用执行计数不增加。

验收报告必须记录 PostgreSQL 镜像版本、Testcontainers 版本、Flyway 版本、测试数量、失败数和实际命令输出；未真实启动 PostgreSQL 容器不得声明 V2 迁移通过。
