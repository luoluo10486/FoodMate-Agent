# ADR-0002：数据所有权

- Status: Accepted
- Date: 2026-07-11

## 背景

AgentRun、消息、取消、Tool/SQL 调用和模型用量需要可审计、可幂等、可回滚的业务记录。V1 表结构缺少跨运行时 dispatch、event inbox 和 invocation 证据；V2 设计以追加迁移补齐这些能力。当前 Java 尚无业务持久化实现，Python Runtime 也尚未创建。

## 决策

Java 和其 PostgreSQL 业务库拥有所有业务事实：用户、会话、消息、AgentRun、dispatch、event/error inbox、取消裁决、Tool/SQL invocation、模型用量和审计。Python 只拥有可重建的推理技术状态，例如临时编排上下文、checkpoint、模型/RAG 过程状态；它通过 V1 事件和 proposal 报告事实，不直接写业务库。

## 原因

单一业务权威使状态机、幂等、审计和回滚有明确裁决点。Java 在一个事务内先写 inbox、再校验 event_seq 和状态迁移，防止重复、乱序或终态回退。V2 使用先追加、再双写/回填/对账、最后收紧约束的方式，保留兼容窗口。

## 替代方案

1. 双写 Java/Python 两个业务库：需要分布式一致性和复杂对账，冲突裁决不清晰。
2. Python 直接读写 PostgreSQL：会绕过 Java 授权、审计、Flyway 和状态机边界。
3. 只存内存事件或日志：无法提供可恢复的幂等、取消和审计证据。

## 后果

需要实现 V2 dispatch、inbox、cancellation、invocation 和模型调用结构，并使用 PostgreSQL Testcontainers 验证 Flyway、幂等、终态竞争和回滚兼容。Python 重启后需由 checkpoint 和 Java 权威事件恢复。现有 V1 SQL、文本测试和 marker 类不构成这一决策已落地的证明。

## 约束

所有跨运行时业务消息使用 `run_id`，事件以 `(run_id,event_id)` 去重，Tool/SQL 使用 `invocation_id` 与 `(run_id,idempotency_key)`。Java 首次接受的合法终态不可回退。V2 迁移不得重写 V1 或篡改 Flyway history；阶段 5 收紧后不得直接回滚到不写新列的 Java 版本。

## 重新评估条件

只有在数据库技术、法定数据驻留、审计模型或业务一致性需求显著变化时重评。变更必须包含数据血缘、权限、幂等、迁移、兼容窗口、备份恢复和 Testcontainers 验证，且不得为了性能优化取消 Java 的权威裁决。
