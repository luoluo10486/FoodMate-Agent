# FoodMate 部署与回滚

## 1. 当前可启动范围

当前只能诚实支持 Java `local-stub` 启动：

```powershell
.\mvnw.cmd -pl foodmate-bootstrap -am package
& java -jar '.\foodmate-bootstrap\target\foodmate-bootstrap-0.1.0-SNAPSHOT.jar' '--spring.profiles.active=local-stub'
```

在另一个 PowerShell 中：

```powershell
Invoke-WebRequest http://localhost:8080/actuator/health
```

上述 JAR 命令已于 2026-07-11 实际验证：package 成功、health 返回 HTTP 200 和 `UP`，进程随后已终止。前台运行时按 Ctrl+C 正常停止。该 profile 排除数据库和 Flyway，当前通用 `/actuator/health` 只算 Java 骨架进程检查，不能作为数据库、Runtime 或业务就绪证明。前端目前是静态 mock 原型；Python Runtime 尚不存在。仓库没有 Docker/Compose、CI 工作流或部署清单，因而没有现成的容器、流水线或生产发布命令可执行。

## 2. 目标双运行时发布前置条件

以下是目标态运行手册，只有相应实现、环境和验证实际存在后才可执行：

- Java 具备业务 API、AgentRun 状态机、Runtime Client、事件 inbox、Service JWT 验证和 Tool/SQL 控制面。
- Python Runtime 提供内部 HTTPS 端点、V1 Schema 校验、短期 Service JWT 验证、推理/编排/RAG 与可重建 checkpoint。
- PostgreSQL 可执行 Flyway，且 V2 追加迁移、兼容 Java 和 Testcontainers 验收已通过。
- 前端已从 mock 切换到实际 API/SSE，并完成 E2E。
- 部署平台已有受控 Secret、日志、指标、追踪、健康检查和回滚权限；本仓库目前不提供这些清单。

## 3. 目标健康检查、依赖矩阵与 Secret

目标发布门禁固定使用下列路径：Java liveness 为 `/actuator/health/liveness`，Java readiness 为 `/actuator/health/readiness`；Python liveness 为 `/health/live`，Python readiness 为 `/health/ready`。Java 的两个端点只有在 Actuator probes 已配置且数据库、Flyway、Runtime Client 等实际健康指标已实现后才能作为目标门禁；当前 YAML 虽启用了 probes，`local-stub` 的 200 响应不证明这些依赖。Python Runtime 和其健康端点当前都不存在。健康端点不得把“仅进程存活”误报为“可处理 Run”。

| 依赖或证据 | Java stub | 目标双运行时 | 阻断条件 |
|---|---|---|---|
| Java 进程 | `/actuator/health` 仅检查进程响应 | liveness 返回成功 | 无法启动或 liveness 非成功 |
| PostgreSQL 与 Flyway | 不适用；profile 已排除二者 | readiness 必须验证连接、迁移版本和 Flyway 状态 | 未连接、迁移失败或 schema 不兼容 |
| Runtime V1 契约 | 不适用 | Java/Python Schema、版本、canonical digest 和错误映射互测 | 版本、Schema 或摘要不匹配 |
| 双向 Service JWT | 不适用 | issuer、audience、scope、TTL、`kid` 和密钥轮换通过 | 认证/授权失败、TTL 或密钥配置错误 |
| dispatch outbox 与事件/error inbox | 未实现 | readiness 监测可接受的发布/消费 lag、无卡死 active lease 和无未裁决序列缺口 | lag 超阈值、lease 未释放、inbox 缺口或重复冲突 |
| Tool/SQL registry 与执行面 | 未实现 | registry 版本、policy、审计、只读 SQL Guard 和执行依赖可用 | registry 缺失、policy/审计失败或 SQL Guard 不可用 |
| Python 模型/RAG | Runtime 不存在 | `/health/ready` 验证已配置的模型/RAG 依赖与回调身份，不泄漏 Secret | 未配置、依赖不可达或回调身份失败 |

stub 只允许通过第一行的进程检查；其余任一目标依赖未满足时，双运行时 readiness 必须阻断流量或发布推进。

部署前准备以下 Secret 引用，而非明文值：PostgreSQL URL/用户名/密码、Java 到 Python 的 JWT 私钥与 `kid`、Python 验证 Java 的公钥、Python 到 Java 的 JWT 签发材料与 Java 公钥、模型供应商 key、RAG/向量存储 token。生产 Runtime URL 必须是内部 HTTPS；JWT audience、scope、TTL 和契约版本须与 V1 一致。Python 不得获得业务数据库 Secret，Tool/SQL 凭据只留在 Java 控制面。

## 4. 目标迁移与发布顺序

V2 与 V1 契约版本不同域：V2 是 Flyway 数据版本，V1 是跨运行时消息版本。采用仅追加、先兼容后收紧的顺序：

1. 备份并验证可恢复性；运行真实 PostgreSQL 的 Flyway/Testcontainers 验收。先发布仅追加且可空的 V2 schema，建立 dispatch、event inbox、cancellation、Tool/SQL 幂等和模型调用结构。
2. 发布兼容 Java。Java 同时理解 V1 与 V2 数据、持久化 dispatch 和 inbox，并在启用 Python 前通过 Runtime stub 验证 dispatch/event/cancel；此时仍保留旧 Java 回滚窗口。
3. 发布 Python Runtime。先完成内部健康、JWT、V1 Schema、事件回调和故障注入 smoke，再让 Java 对小范围流量启用 Runtime Client。
4. 发布前端静态资源。前端必须在真实 API/SSE、认证与错误行为验证后才切换；当前 mock 前端不能作为目标部署完成证据。
5. 兼容矩阵、回填和对账完成，所有回滚候选 Java 都能写新列后，才收紧 V2 非空/唯一约束。

Java 先于 Python 的原因是 Java 保存业务权威状态、派发和事件接入；前端最后发布可避免用户进入尚未存在的业务链路。

## 5. Outbox、inbox 与降级

目标态中，Java 必须先在同一事务持久化 AgentRun 与 V2 dispatch，再由可靠派发 outbox 或等价的持久化派发机制调用 Python；不能先调用再猜测数据库状态。Python 的 RunEvent、RuntimeError、ToolProposal 和 SqlProposal 回到 Java 后，Java 以 inbox 幂等、序号和状态机事务接受。当前仓库尚未实现 outbox、inbox 或实际 Runtime 调用。

Python 不可用、超时、契约不兼容或认证失败时，Java 不得伪造完成、重试副作用或绕过 Tool/SQL 策略。按已批准的错误映射和绝对 `deadline_at` 处理：可重试的传输/503 在边界内重试；401/403/400/409/426、幂等冲突和终态不自动重试。用户侧降级应明确显示排队、失败或需要稍后重试；只有明确设计并实现的本地 stub 才能用于受控开发，不得在生产伪装为推理结果。

## 6. 回滚与恢复

目标双运行时回滚按以下固定顺序执行，步骤在当前仓库尚未实现，不能当作现有操作命令：

1. 关闭新的 Run 和 dispatch 入口，拒绝新的业务副作用请求。
2. 递增 run/dispatch fencing epoch，停止创建新的 Tool/SQL invocation；旧 epoch 的 proposal、结果或事件只可按审计规则拒绝或去重，不能重新激活。
3. 向每个 active dispatch 的 Python Runtime 发送复用既有取消意图的 `CancelCommand`，并等待有界 drain。期间 outbox 继续发布已持久化的取消/事件，inbox 继续接收、去重和应用 RunEvent、RuntimeError、Tool/SQL 结果；不能先停事件通道。
4. drain 到期仍 active 的 Run，按 Java 状态机使用现有合法 `failed` 或 `cancelled` 终态收敛，并写入审计 reason；不引入或写入“暂停”状态。unknown 外部副作用必须标记为待人工对账，不能根据超时推定成功或失败。
5. 确认没有 active lease、未完成 outbox/inbox 记录、序列缺口或 unknown side effect 后，才停止 Python Runtime 并回滚 Java 到兼容版本。随后由人工对账 unknown 外部副作用和相关审计证据。
6. 阶段 1-4 的 V2 新列保持可空时，可回滚到仍兼容 V2 schema 的 Java。阶段 5 收紧约束后，必须先发布前向兼容 Flyway 迁移，使目标列恢复可空或提供不伪造业务语义的兼容默认值，并通过旧 Java smoke test；之后才能回滚旧应用。不得修改既有迁移或手工篡改 Flyway history。

重新前滚时，从保留的 `last_event_seq`、dispatch、outbox 和 inbox 对账恢复；不得清空后从 event_seq 1 重放副作用。每次目标部署与回滚演练应记录版本、迁移版本、health/readiness 结果、契约版本、fencing epoch、drain 结果、unknown side effect 对账和恢复结果。
