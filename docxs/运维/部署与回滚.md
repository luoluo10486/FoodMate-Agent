# FoodMate 部署与回滚

## 1. 当前可启动范围

当前只能诚实支持 Java `local-stub` 启动：

```powershell
.\mvnw.cmd -pl foodmate-bootstrap -am package
& java -jar '.\foodmate-bootstrap\target\foodmate-bootstrap-0.1.0-SNAPSHOT.jar' '--spring.profiles.active=local-stub'
```

在另一个 PowerShell 中：

```powershell
Invoke-WebRequest http://localhost:8080/actuator/health
```

上述 JAR 命令已于 2026-07-11 实际验证：package 成功、health 返回 HTTP 200 和 `UP`，进程随后已终止。前台运行时按 Ctrl+C 正常停止。该 profile 排除数据库和 Flyway，当前通用 `/actuator/health` 只算 Java 骨架进程检查，不能作为数据库、Runtime 或业务就绪证明。前端目前是静态 mock 原型；Python Runtime 尚不存在。仓库没有 Docker/Compose、CI 工作流或部署清单，因而没有现成的容器、流水线或生产发布命令可执行。

## 2. 目标双运行时发布前置条件

以下是目标态运行手册，只有相应实现、环境和验证实际存在后才可执行：

- Java 具备业务 API、AgentRun 状态机、Runtime Client、事件 inbox、Service JWT 验证和 Tool/SQL 控制面。
- Python Runtime 提供内部 HTTPS 端点、V1 Schema 校验、短期 Service JWT 验证、推理/编排/RAG 与可重建 checkpoint。
- PostgreSQL 可执行 Flyway，且 V2 的 active dispatch/fencing、immutable RunCommand dispatch outbox、Run epochs、canonical inbox/rejection、Tool/SQL lease、SSE outbox、兼容 Java 和 Testcontainers 验收已通过。
- 前端已从 mock 切换到实际 API/SSE，并完成 E2E。
- 部署平台已有受控 Secret、日志、指标、追踪、健康检查和回滚权限；本仓库目前不提供这些清单。

## 3. 目标健康检查、依赖矩阵与 Secret

目标发布门禁固定使用下列路径：Java liveness 为 `/actuator/health/liveness`，Java readiness 为 `/actuator/health/readiness`；Python liveness 为 `/foodmate/internal/health/live`，Python readiness 为 `/foodmate/internal/health/ready`。所有 Python health 调用必须保留完整的 `/foodmate/internal` 前缀。Java 的两个端点只有在 Actuator probes 已配置且实际健康指标已实现后才能作为目标门禁；当前 YAML 虽启用了 probes，`local-stub` 的 200 响应不证明这些依赖。Python Runtime 和其健康端点当前都不存在。健康端点不得把“仅进程存活”误报为“可处理 Run”。

实例 readiness 只回答本实例是否能安全认证、接收并持久化请求：Java 至少验证数据库连接、Flyway 兼容、契约解析、JWT key 和 outbox 写入；Python 至少验证服务身份材料、契约解析、可持久化技术状态以及 Java 回调客户端的必要配置。单个 run gap、duplicate conflict、lease backlog 和 outbox lag 是业务告警与发布/灰度门禁，不是把负责恢复的事件入口摘除的理由。只有系统性持久化、认证或必需依赖不可用时，实例 readiness 才为 DOWN。

| 依赖或证据 | Java stub | 目标双运行时 | 阻断条件 |
|---|---|---|---|
| Java 进程 | `/actuator/health` 仅检查进程响应 | liveness 返回成功 | 无法启动或 liveness 非成功 |
| PostgreSQL 与 Flyway | 不适用；profile 已排除二者 | 实例 readiness 验证连接、迁移兼容和必要 schema 可写 | 系统性连接失败、迁移不兼容或无法持久化 |
| Runtime V1 契约 | 不适用 | 实例可解析/校验已批准的 Schema、版本、canonical digest 和错误映射 | 契约解析或校验器不可用；单条消息冲突改为业务告警 |
| 双向 Service JWT | 不适用 | 本实例可加载并验证 issuer、audience、scope、TTL、`kid` 和轮换材料 | key 缺失、认证材料无效或验证器不可用 |
| active dispatch、dispatch/SSE outbox、canonical event/error inbox 与 rejection | 未实现 | 实例 readiness 验证 dispatch、immutable RunCommand outbox、SSE outbox、inbox/rejection 可持久化；publisher 与事件入口持续运行以恢复状态 | 无法写入/读取持久化结构才 DOWN；lag、gap、duplicate conflict、lease backlog 为告警和发布/灰度阻断 |
| Tool/SQL registry 与执行面 | 未实现 | 必需 registry/policy 可加载，执行面故障以受控拒绝和审计处理 | registry/policy 系统性不可加载才 DOWN；单次 invocation 失败为业务告警 |
| Python 模型/RAG | Runtime 不存在 | `/foodmate/internal/health/ready` 验证必要配置、身份和技术状态持久化能力，不泄漏 Secret | 系统性必需依赖不可用才 DOWN；单个模型调用失败为业务告警 |

stub 只允许通过第一行的进程检查。双运行时中，实例安全接收/持久化能力失败才使 readiness DOWN；lag、gap、duplicate conflict 和 lease backlog 保持事件入口运行，同时阻断发布或灰度推进并触发告警。

部署前准备以下 Secret 引用，而非明文值：PostgreSQL URL/用户名/密码、Java 到 Python 的 JWT 私钥与 `kid`、Python 验证 Java 的公钥、Python 到 Java 的 JWT 签发材料与 Java 公钥、模型供应商 key、RAG/向量存储 token。生产 Runtime URL 必须是内部 HTTPS；JWT audience、scope、TTL 和契约版本须与 V1 一致。Python 不得获得业务数据库 Secret，Tool/SQL 凭据只留在 Java 控制面。

## 4. 目标迁移与发布顺序

V2 与 V1 契约版本不同域：V2 是 Flyway 数据版本，V1 是跨运行时消息版本。采用仅追加、先兼容后收紧的顺序：

1. 备份并验证可恢复性；运行真实 PostgreSQL 的 Flyway/Testcontainers 验收。先发布仅追加且兼容的 V2 schema，完整建立 single-active dispatch/fencing、immutable RunCommand dispatch outbox、admission/cancellation epoch、canonical event inbox、event rejection、Tool/SQL owner lease、SSE transactional outbox 和模型调用结构及索引。
2. 发布兼容 Java。Java 同时理解 V1 与 V2 数据，在 feature flag 下同事务持久化 AgentRun/active dispatch/dispatch outbox，并持久化 canonical inbox/rejection、invocation lease 和 SSE outbox；启用 Python 前用 Runtime stub 验证 dispatch publisher 启动恢复、多实例单领取、幂等接受、event/cancel/Last-Event-ID，此时仍保留旧 Java 回滚窗口。
3. 阶段 4 门禁通过后发布 Python Runtime。门禁必须证明上述 V2 表/列/索引/CHECK 存在，PostgreSQL Testcontainers 覆盖 dispatch outbox 原子写、提交后崩溃、ACK 丢失、单 active、旧 fence、rejection 分流、lease 接管、多实例竞争、unknown 和 SSE 崩溃恢复；先完成内部健康、JWT、V1 Schema、事件回调和故障注入 smoke，再让 Java 对小范围流量启用 Runtime Client。
4. 发布前端静态资源。前端必须在真实 API/SSE、认证与错误行为验证后才切换；当前 mock 前端不能作为目标部署完成证据。
5. 兼容矩阵、回填和对账完成，所有回滚候选 Java 都能写新列后，才收紧 V2 非空/唯一约束。

Java 先于 Python 的原因是 Java 保存业务权威状态、派发和事件接入；前端最后发布可避免用户进入尚未存在的业务链路。

## 5. Outbox、inbox 与降级

目标态中，Java 必须在同一事务持久化 AgentRun、V2 active dispatch/epoch/fence 和唯一 `runtime_dispatch_outbox`。该 outbox 固化完整 immutable RunCommand payload、hash、schema、run/dispatch/attempt、deadline 和 fencing epoch；publisher 在提交后以 lease/CAS 领取并发送完全相同 payload，不能先调用、提交后临时重建或以“等价内存机制”替代。启动恢复扫描 pending/过期 lease，多实例同一时刻只允许一个 owner。RunEvent 回到 Java 后先查 known event ID/hash，再校验 active dispatch/fence/seq；只有接受的 canonical event 才写 `runtime_event_inbox`，gap、乱序、同 seq 不同 event 和旧 dispatch 只写 `runtime_event_rejections` 并返回稳定 RuntimeError。ToolProposal/SqlProposal 必须通过 epoch/fence 和 owner lease。当前仓库尚未实现这些结构或实际 Runtime 调用。

Python 必须先按 `dispatch_id/request_hash` 持久化接受，再返回接受/已接受 2xx；ACK 丢失时 Java 重发原 RunCommand，Python 返回原结果且不双执行。Java 只有验证身份、版本和 runs endpoint 成功语义后才标 `delivered`。deadline 到期或旧 dispatch 已 `superseded` 时不发送并标 `expired`；hash 冲突标 `failed` 且不自动新建 dispatch。显式重新派发必须产生新 `dispatch_id/attempt/outbox`，旧 dispatch 先标 `superseded`。

面向前端的 SSE 只能由 `agent_run_sse_outbox` 发布。业务事务固定 `source_event_key/sse_event_id/stream_seq/payload_json`；publisher 以 owner token/lease CAS 领取并有界重试，浏览器 `Last-Event-ID` 解析到持久化 stream cursor。不得以仅内存 after-commit 发送替代，也不得因重试生成新的 SSE ID。

Python 不可用、超时、契约不兼容或认证失败时，Java 不得伪造完成、重试副作用或绕过 Tool/SQL 策略。按已批准的错误映射和绝对 `deadline_at` 处理：可重试的传输/503 在边界内重试；401/403/400/409/426、幂等冲突和终态不自动重试。用户侧降级应明确显示排队、失败或需要稍后重试；只有明确设计并实现的本地 stub 才能用于受控开发，不得在生产伪装为推理结果。

## 6. 回滚与恢复

目标双运行时回滚按以下固定顺序执行，步骤在当前仓库尚未实现，不能当作现有操作命令：

1. 关闭 admission gate：原子设置 `admission_state=closed` 并递增 `admission_epoch`，先禁止新的 Run、dispatch/outbox 与 Tool/SQL invocation 创建；已捕获旧 epoch/fence 的 executor 在副作用提交前必须被 fenced。保持 dispatch publisher 和当前 active dispatch 的接收通道运行。
2. dispatch publisher 继续发送关闭前已提交且仍 active、未过 deadline 的 immutable RunCommand；向已接受的 active dispatch 发送复用既有取消意图的 `CancelCommand`，并等待有界 drain。期间 dispatch/SSE outbox 持续发布，canonical inbox/rejection 持续接收、去重或拒绝 RunEvent/RuntimeError；已有 invocation 的确定结果仍可接收，不能先停 publisher 或事件通道。
3. 只有派发已关闭且 dispatch outbox 全部进入 `delivered/expired/failed` 后，才算 dispatch drain 完成。随后提升 active/admission epoch、失效 fencing token、将已 drain 的 dispatch 标记为 `superseded`，确认无 active dispatch/Tool/SQL/SSE lease，再停止 dispatch publisher、Python Runtime，并回滚 Java 到兼容版本。
4. drain 超时才提升 epoch，以 fencing 阻止剩余执行；仍 active 的 Run 按 Java 状态机使用现有合法 `failed` 或 `cancelled` 终态和审计 reason 收敛，不引入或写入“暂停”状态。旧 epoch 后续事件不再推进状态，以 state conflict 记录审计。
5. unknown 外部副作用不阻止技术回滚，但必须登记并 fenced，禁止对应业务重试或补偿，并建立人工对账阻断项。可回滚条件是无 active Tool/SQL/SSE lease，且每项 unknown/dead-letter 已登记并 fenced，不要求 unknown 为零。
6. 阶段 1-4 的 V2 新列保持可空时，可回滚到仍兼容 V2 schema 的 Java。阶段 5 收紧约束后，必须先发布前向兼容 Flyway 迁移，使目标列恢复可空或提供不伪造业务语义的兼容默认值，并通过旧 Java smoke test；之后才能回滚旧应用。不得修改既有迁移或手工篡改 Flyway history。

重新前滚时，从保留的 dispatch outbox payload/hash/status、active/admission/cancellation epoch、fencing token、`last_event_seq`、canonical inbox/rejection、invocation state 和 SSE `stream_seq` 对账恢复；不得清空后重建 RunCommand 或从 event_seq 1 重放副作用。每次目标部署与回滚演练应记录版本、迁移版本、health/readiness 结果、契约版本、fencing epoch、dispatch/SSE drain 结果、unknown/dead-letter 对账和恢复结果。
