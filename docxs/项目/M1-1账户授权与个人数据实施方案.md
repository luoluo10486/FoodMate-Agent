# FoodMate M1-1 账户、授权与个人数据实施方案

## 1. 目标与边界

本方案将 M1-1 落地为可实施、可验收的账户与个人数据能力。它以现有的服务端会话表 + Cookie/CSRF 认证为基础，不引入浏览器 Access Token、Refresh Token 或第三方登录。

范围：

- 注册、登录、登出、当前用户、改密、密码重置与设备会话管理。
- `user`、`operator`、`admin`、`superadmin` 的资源授权与审计。
- 个人资料、营养偏好、过敏原、忌口、单位偏好与头像。
- QQ 邮箱通知、MinIO 私有对象存储、个人数据导出和账号注销。

非目标：第三方登录、原生 App 登录、开放 API Token、OAuth2/OIDC、多租户运营能力，以及让 Python Runtime 直连业务数据库。

事实优先级：现有代码和测试 > ADR-0004 > 本方案 > 旧接口文档。旧文档中若仍出现 Web JWT 登录描述，以 ADR-0004 的 Cookie 会话方案为准。

## 2. 已确认决策

| 主题 | 决策 |
|---|---|
| Web 认证 | 服务端 `user_auth_sessions` + `foodmate_session` / `foodmate_csrf` Cookie；已认证写请求继续要求同源 Origin 和 `X-CSRF-Token`。 |
| 邮件 | 仅使用 QQ SMTP；重置令牌只保存哈希、单次使用、短时过期；请求重置时总返回统一成功提示。未配置 SMTP 时明确不可用，不能返回令牌或伪装发送成功。 |
| 改密/重置 | 成功后撤销该账号的全部设备会话，包括当前设备；所有设备重新登录。 |
| 角色 | `user` 仅操作自己的资源；`operator` 只读运营与审计信息；`admin` 管理 `user/operator` 的账号状态与角色；`superadmin` 可跨用户查看业务与审计数据。 |
| 角色提升 | 首个 `superadmin` 通过受控人工 SQL 创建。仅 `superadmin` 可授予/撤销 `admin` 和 `superadmin`；`admin` 仅可管理 `user/operator`；任何人不得修改同级或更高角色。 |
| 凭据边界 | 即使 `superadmin` 也不能通过 API 读取或导出密码哈希、会话/CSRF 令牌哈希、重置令牌、SMTP 密钥或模型供应商密钥。 |
| 头像存储 | 采用私有 MinIO Bucket，通过 S3 兼容接口访问；PostgreSQL 不保存图片二进制。校验 JPG/PNG/WebP、最大 2 MB、像素尺寸和解码结果。 |
| 对象访问 | Bucket 不公开。头像使用短时签名访问；导出 ZIP 使用一次性、短时有效下载链接，过期后清理。 |
| 数据导出 | 异步生成 ZIP，包含用户可见的个人资料、头像、对话、饮食记录、报告、餐计划、购物清单、个人记忆及与本人有关的运行结果；不包含认证凭据、服务密钥、其他用户数据和管理员审计记录。 |
| 账号注销 | 用户以当前密码和不可恢复确认二次确认后，立即撤销全部会话、禁用账号、创建不可撤销的异步物理清理任务；不设冷静期和撤销入口。 |

## 3. 配置与密钥

`.env` 是本地未追踪的密钥入口，`application.yml` 显式以 `optional:file:.env[.properties]` 导入。真实值不得写入 YAML、示例配置、测试快照、日志或 Git 历史。

| 配置 | 用途 | 规则 |
|---|---|---|
| `MAIL_HOST`、`MAIL_PORT`、`MAIL_PROTOCOL` | QQ SMTP 地址与协议 | V1 使用 `smtp.qq.com:465` 和 `smtps`。 |
| `MAIL_USERNAME`、`MAIL_PASSWORD` | QQ 邮箱与 SMTP 授权码 | 仅 Secret 注入；日志中只记录是否可用，不记录值。 |
| `MINIO_ENDPOINT`、`MINIO_ACCESS_KEY`、`MINIO_SECRET_KEY`、`MINIO_BUCKET` | 私有 MinIO Bucket | 新增对象存储适配时接入；prod 缺失时头像/导出能力明确不可用。 |
| `MINIO_REGION`、`MINIO_PRESIGN_TTL_SECONDS` | MinIO 区域和签名过期 | 默认短时；导出链接额外只能消费一次。 |
| `MODEL_GATEWAY_KEY`、`MILVUS_TOKEN`、Runtime 私钥/公钥 | 后续供应商与内部服务密钥 | 继续仅存在 `.env` 或部署 Secret 中，M1-1 不读取或返回其值。 |

真实邮件发送验收必须在已授权的测试收件箱完成；单元测试和本地默认使用受控 Outbox/Fake Sender，不得向真实用户邮箱发送测试邮件。

## 4. 数据库与人工 SQL

Flyway 在所有 profile 中保持关闭。每项 DDL 必须作为 `script/sql/FoodMate/migration/V2__m1_account_and_privacy.sql` 的人工执行脚本，并同时补充变更说明、执行记录、校验 SQL、回滚前置条件和对应反向 SQL。已执行脚本禁止原地修改。

### 4.1 增量结构

| 对象 | 变更 |
|---|---|
| `users` | 将 `chk_users_role` 扩展为 `user/admin/operator/superadmin`；账号注销继续使用 `status=disabled`，不新增可恢复状态。 |
| `user_auth_sessions` | 复用现有哈希会话表；补充按用户、设备、有效状态查询与批量撤销所需索引。不得保存明文 Cookie 或 CSRF 值。 |
| `password_reset_tokens` | 新表：用户 ID、令牌哈希、用途、过期时间、已使用时间、创建时间；令牌哈希唯一，不能保存明文令牌。 |
| `user_avatar_assets` | 复用 `storage_key` 作为持久对象定位；移除对持久 `url` 的依赖，使其可为空或仅保留稳定应用路径，签名 URL 不入库；增加对象摘要、原始文件名和删除时间等必要元数据。 |
| `data_export_jobs` | 新表：任务 ID、用户 ID、状态、导出对象键、过期时间、下载消费时间、失败原因类别、创建/完成时间；不得存储 ZIP 内容或明文下载令牌。 |
| `account_deletion_jobs` | 新表：用户 ID、状态、开始/完成时间、失败原因类别、已删除对象计数和重试次数；任务仅可推进，不能恢复已提交注销。 |
| `operation_audits` | 复用现有审计表记录角色变更、会话撤销、导出、注销与跨用户读取；详情中不写密码、令牌或密钥。 |

### 4.2 执行、校验和回滚

1. 在维护窗口对 `FoodMate` 执行 `pg_dump --format=custom`，记录路径、大小和 SHA-256。
2. 在备份副本先演练 V2 脚本，并执行对应 `validation/V2__m1_account_and_privacy_validation.sql`。
3. 执行人将环境、数据库版本、脚本版本、备份、输出摘要和回滚结论追加到 `EXECUTION_RECORD.md`。
4. 回滚仅适用于尚未产生依赖新结构的业务数据的场景。已生成导出、已提交注销或已分配角色变更时，先停止任务并评估数据一致性，不得直接删表或修改历史记录。

## 5. 服务设计与接口契约

所有已认证写接口遵守 Cookie/CSRF 规则。错误响应只返回稳定错误码、request ID 和 trace ID；不得返回用户是否存在、令牌、内部 SQL、对象存储地址或异常栈。

### 5.1 认证与会话

| 接口 | 行为 | 授权与安全规则 |
|---|---|---|
| `POST /api/auth/register` | 注册并创建初始会话 | 邮箱和用户名在未删除记录中唯一；密码不写日志。 |
| `POST /api/auth/login` | 校验密码、创建独立设备会话 | 返回 Session/CSRF Cookie；失败信息统一。 |
| `POST /api/auth/logout` | 撤销当前会话 | 需要当前 Cookie + CSRF。 |
| `GET /api/me` | 返回当前用户基础信息 | 绝不返回密码哈希、角色内部备注或会话哈希。 |
| `GET /api/me/sessions` | 列出当前用户设备会话元数据 | 不返回会话/CSRF 哈希；当前设备可明确标识。 |
| `DELETE /api/me/sessions/{id}` | 撤销单个其他设备会话 | 仅资源所有者；不可撤销其他用户会话。 |
| `POST /api/me/sessions/revoke-all` | 撤销全部会话 | 先执行撤销，再清除当前 Cookie。 |
| `POST /api/me/password` | 校验当前密码并更新密码 | 改密成功后撤销全部会话并清除当前 Cookie。 |

### 5.2 密码重置与邮件

| 接口 | 行为 | 安全规则 |
|---|---|---|
| `POST /api/auth/password-reset-requests` | 接收邮箱并投递重置邮件 | 无论邮箱是否存在，都返回相同的 `202`；限流、审计，不泄露令牌。 |
| `POST /api/auth/password-resets` | 提交链接令牌与新密码 | 校验哈希、过期和单次使用；成功后撤销全部会话。 |

邮件链接只携带随机一次性令牌；数据库仅保存哈希。没有可用邮件发送器时，重置请求返回稳定的“邮件服务暂不可用”错误，不能在响应或日志中返回替代令牌。

### 5.3 个人资料、头像与数据权利

| 接口 | 行为 | 授权与安全规则 |
|---|---|---|
| `GET/PATCH /api/me/profile` | 读取或更新个人资料与饮食偏好 | 仅本人；JSON 数组/单位配置做类型、长度和数值范围校验。 |
| `PUT /api/me/avatar` | 上传并替换头像 | 服务端重新校验 MIME、文件签名、2 MB、像素上限和解码；先写新对象、事务更新元数据、再异步删除旧对象。 |
| `GET /api/me/avatar-download` | 获取头像短时访问地址 | 仅本人或具备授权的跨用户读取者；不返回永久 MinIO 地址。 |
| `POST /api/me/data-exports` | 创建异步导出任务 | 仅本人；限制并发任务；完成后通知注册 QQ 邮箱。 |
| `GET /api/me/data-exports/{id}` | 查询本人导出状态 | 仅任务所有者；失败只返回稳定失败类别。 |
| `POST /api/me/data-exports/{id}/download` | 消费一次性下载链接 | 校验所有者、未过期、未消费；原子标记消费后返回短时签名地址。 |
| `POST /api/me/account-deletion` | 提交注销 | 请求包含当前密码和明确确认语；成功即禁用账号、撤销会话、创建清理任务、清 Cookie。 |

### 5.4 管理授权

| 操作 | `user` | `operator` | `admin` | `superadmin` |
|---|---:|---:|---:|---:|
| 自己的资料、会话、导出、注销 | 是 | 是 | 是 | 是 |
| 查看跨用户运营/审计数据 | 否 | 只读 | 受管理范围限制 | 是 |
| 修改 `user/operator` 账号状态或角色 | 否 | 否 | 是 | 是 |
| 授予/撤销 `admin/superadmin` | 否 | 否 | 否 | 是 |
| 修改同级或更高角色 | 否 | 否 | 否 | 否 |
| 读取认证凭据、重置令牌、服务密钥 | 否 | 否 | 否 | 否 |

`superadmin` 的跨用户读取、角色变更和账号状态变更必须写入 `operation_audits`，包括操作者、目标用户、动作、原因和 request/trace ID。无理由字段的高风险操作不执行。

## 6. 异步任务与一致性

### 6.1 数据导出

1. 创建任务时写入 `data_export_jobs=queued`，同一用户只允许有限数量未完成任务。
2. Worker 在只读事务中拉取归属用户数据，按版本化 JSON 写入 ZIP，并复制头像原文件。
3. ZIP 上传私有 MinIO 对象，任务置为 `completed`，设置过期时间，随后发送 QQ 通知。
4. 下载消费必须原子化：同一任务只能成功一次；到期、已消费或非所有者均不可下载。
5. 定时清理过期 ZIP 和失败残留对象，并记录审计结果。

### 6.2 账号注销与物理清理

1. 验证当前密码和不可恢复确认语后，在同一事务内禁用用户、撤销全部认证会话、创建 `account_deletion_jobs=queued` 和审计记录。
2. Worker 按依赖顺序删除私有对象、导出文件、个人资料、会话、对话和归属业务数据；每步可重试且可幂等。
3. 清理完成后物理删除或按数据保留策略去标识化账号主记录；不允许恢复登录或撤销清理。
4. 任务失败只能重试清理，不得自动重新启用账号；告警与审计中不包含个人内容或凭据。

## 7. 实施顺序

1. 完成 V2 人工 SQL、校验 SQL、回滚说明、变更说明和执行记录模板；先在 local PostgreSQL 验证。
2. 将当前 `UserAccountService` 的 JDBC 路径扩展为密码策略、全设备撤销、会话列表和角色守卫；local-stub 保持隔离测试用途。
3. 实现邮件端口、QQ SMTP 实现与 Fake Sender，完成密码重置全链路。
4. 实现 MinIO S3 端口、私有 Bucket 初始化检查、头像上传/替换/删除与短时签名访问。
5. 实现导出与注销 Job、Worker、对象清理和通知。
6. 实现后台角色/账号管理接口与所有跨用户资源归属校验。
7. 将 Profile 页面从 mock 接到真实资料、改密、会话、头像、导出和注销接口；高风险操作必须二次确认。
8. 通过全部验收后更新 TODO 状态和实现状态文档；没有真实证据不得标记完成。

## 8. 验收矩阵

| 场景 | 预期 |
|---|---|
| 注册、登录、登出和重启读取 | PostgreSQL 会话持久化正确；Cookie/CSRF 属性符合 M0-3。 |
| 改密或重置成功 | 当前和其他设备会话全部失效，旧 Cookie 不能访问。 |
| 重置不存在邮箱 | 返回与存在邮箱相同的 `202`，不发送令牌、不泄露用户存在性。 |
| 重置令牌重放/过期 | 稳定失败，不更新密码，不创建会话。 |
| 私有头像 | 非所有者不能获取；上传伪造 MIME、超限文件、损坏图片或超像素图片均失败；旧对象最终清理。 |
| 数据导出 | ZIP 仅包含本人可见数据；不含密码哈希、会话/CSRF 哈希、重置令牌和任何服务密钥；下载只能一次且会过期。 |
| 账号注销 | 当前密码或确认语错误时零副作用；成功后立即无法认证，清理任务可恢复重试但账号不可恢复。 |
| RBAC | `user` 无法读写他人资源；`operator` 不能写；`admin` 无法提升自己或修改 admin/superadmin；仅 superadmin 可提升高权限。 |
| superadmin 边界 | 可跨用户读业务与审计；API 仍不返回认证哈希、令牌或基础设施密钥；每次高风险操作均有审计。 |
| 配置缺失 | prod 缺 QQ SMTP 或 MinIO Secret 时，对应功能明确失败且日志不泄露 Secret。 |
| 回归 | Java 单元/集成测试、local PostgreSQL E2E、前端 typecheck/lint/Vitest 与浏览器 E2E 全部通过。 |

## 9. 完成定义

M1-1 只有在人工 SQL 已留存执行证据、真实 local PostgreSQL 验证通过、QQ 测试邮箱发送与单次链接验收通过、MinIO 私有访问验收通过、角色越权回归通过，以及前后端真实接口替代对应 mock 后，才可标记完成。仅编译通过或仅 local-stub 通过不构成完成证据。
