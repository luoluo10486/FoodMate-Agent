# M1-2 会话、消息与前端真实接入实施方案

## 1. 目标与边界

M1-2 将 FoodMate 前端的认证、会话和消息功能切换到真实 PostgreSQL API，并完成可恢复的会话管理与稳定的页面状态。

本阶段发送消息的成功语义仅为“用户消息已保存”。不创建 AgentRun，不派发 Runtime，不生成 AI 回复，不接入 SSE；这些能力全部留给 M1-3。

附件入口保留，但明确提示暂不支持发送附件，不上传文件、不创建附件记录、不展示伪造成功。

## 2. 已确认决策

| 项目 | 决策 |
|---|---|
| 认证 | 服务端 Cookie 会话校验；不引入前端 refresh token |
| 认证失效 | API `401` 清理前端身份并跳转登录页，保留原访问地址 |
| 无权限 | API `403` 留在当前页展示无权限状态 |
| 网络错误 | 不视为退出登录；保留当前页面数据并提供重试 |
| 会话删除 | 软删除，可恢复，恢复期 30 天；物理清理由后续后台任务负责 |
| 会话搜索 | 搜索本人未删除会话的标题和消息正文，返回会话摘要和命中片段 |
| 附件 | 保留入口和降级提示，不实施文件消息 |

## 3. 现状与缺口

已有后端基础接口：

| 地址 | 方法 | 当前能力 |
|---|---|---|
| `/api/sessions` | GET/POST | 读取和创建会话，无分页、搜索 |
| `/api/sessions/{id}` | DELETE | 当前实际为归档，不是软删除 |
| `/api/sessions/{id}/messages` | GET/POST | 读取和写入消息，无分页；可由客户端指定任意合法 role |
| `/api/users/me` | GET | Cookie 会话身份校验 |

主要缺口：重命名、归档/取消归档、软删除/恢复、搜索、分页、用户消息角色约束、统一 HTTP Client、登录/注册真实接入、聊天页真实消息接入、路由守卫和网络状态处理。

## 4. 后端实施

### 4.1 会话 API

新增或调整以下接口：

| 地址 | 方法 | 功能 |
|---|---|---|
| `/api/sessions` | GET | 分页读取本人未删除会话；支持 `page`、`size`、`q`、`status` |
| `/api/sessions` | POST | 创建会话；标题为空时使用“新会话” |
| `/api/sessions/{id}` | PATCH | 重命名；标题长度限制 1-255 |
| `/api/sessions/{id}/archive` | POST | 归档会话 |
| `/api/sessions/{id}/unarchive` | POST | 取消归档 |
| `/api/sessions/{id}` | DELETE | 软删除会话，写入 `is_deleted/deleted_at/deleted_by` |
| `/api/sessions/{id}/restore` | POST | 恢复未超过 30 天保留期的会话 |
| `/api/sessions/deleted` | GET | 分页读取本人可恢复的已删除会话 |

所有接口都通过当前 Cookie 用户获取 `user_id`，SQL 必须同时限制 `session_id` 与 `user_id`。资源不存在、已删除、归属错误统一返回 `NOT_FOUND`，避免枚举其他用户资源。

### 4.2 消息 API

| 地址 | 方法 | 功能 |
|---|---|---|
| `/api/sessions/{id}/messages` | GET | 按 `sequence_no` 稳定排序，支持游标或页码分页 |
| `/api/sessions/{id}/messages` | POST | 仅接受 `role=user`，保存用户消息并更新 `last_message_at` |

服务端生成消息序号，禁止前端指定 `assistant/system/tool` 角色。内容限制采用非空、最大长度和 UTF-8 文本校验；重复提交通过客户端请求 ID 或幂等键保证不产生重复用户消息。M1-2 不写 `agent_run_id`。

### 4.3 搜索与索引

先使用 PostgreSQL `ILIKE` 实现关键词搜索：匹配 `sessions.title` 或同一会话下 `messages.content`。查询只返回当前用户的未删除会话，以最近消息时间倒序，并聚合一个截断后的命中片段。

新增面向查询的索引：

- `sessions(user_id, is_deleted, status, last_message_at DESC)`
- `messages(session_id, is_deleted, sequence_no)` 保持现有索引
- 根据真实数据量评估后再增加 `messages.content` 的全文索引；M1-2 不提前引入向量检索。

### 4.4 数据与清理

涉及表：`sessions`、`messages`、`users`、`user_auth_sessions`。

会话软删除不立即删除消息；恢复时恢复会话本身，关联消息保持原状态。30 天后的物理清理任务只在后续明确实现，不由请求接口执行。

## 5. 前端实施

### 5.1 统一 API Client

新增统一客户端，负责：

- `credentials: 'include'`；
- 从 `foodmate_csrf` Cookie 读取 token，并为所有非安全方法发送 `X-CSRF-Token`；
- 统一解析 API 成功与错误包；
- `401` 清除本地认证状态并跳转 `/login?redirect=...`；
- `403` 抛出可展示的无权限错误；
- 网络异常、超时和服务器错误保留错误类型，供页面显示重试操作；
- 请求取消和页面卸载清理，避免会话切换时旧响应覆盖新页面。

### 5.2 认证与路由

登录、注册、登出、当前用户和密码重置调用真实 `/api/auth/*`、`/api/users/me` 接口。应用启动时调用当前用户接口建立认证状态；受保护路由等待校验完成后再渲染，未登录用户跳转登录页并保留原地址。

### 5.3 会话与聊天页面

`WorkspaceLayout` 和侧边栏改用真实会话列表，提供创建、搜索、重命名、归档、删除和恢复入口。`ChatPage` 根据路由会话 ID 加载真实消息，按 `sequence_no` 渲染；发送后乐观显示“发送中”，服务端成功后替换为持久化消息，失败时显示可重试状态。

附件按钮保留，点击显示“暂不支持发送附件”的明确提示。工具选择、Agent 选择保持当前降级状态，不能暗示已经生效。

### 5.4 页面状态

每个真实数据页面实现：首次加载、空会话、空消息、分页加载、搜索无结果、`401`、`403`、网络中断和可重试错误状态。切换会话时取消旧请求，防止消息串页。

## 6. 测试与验收

### 后端

- PostgreSQL HTTP E2E：注册、登录、Cookie/CSRF、创建、重命名、归档、软删除、恢复、搜索、消息写入和分页。
- 越权回归：用户 A 不可读取、修改、删除、恢复或搜索用户 B 的会话/消息。
- 消息角色回归：外部接口提交 `assistant/system/tool` 返回明确 4xx。
- 稳定排序与并发写入：分页不重不漏，序号唯一。
- 30 天恢复期边界：过期会话恢复被拒绝。

### 前端

- 浏览器 E2E：登录跳转、刷新后仍登录、登出、会话 CRUD、搜索、消息发送、失败重试与附件降级提示。
- Cookie/CSRF 矩阵：同源成功、缺失 CSRF 拒绝、会话失效 `401` 跳转、权限不足 `403` 留页。
- 网络中断：不清理登录状态，页面显示重试并避免旧请求覆盖新会话。

### 完成标准

只有在真实 PostgreSQL 下完成上述 HTTP 与浏览器 E2E，且前端涉及的认证、会话、消息页面不再回退 mock 后，M1-2 才可标记完成。Agent 回复、SSE 与运行态不纳入本阶段验收。

## 7. 实施顺序

1. 固化 API 错误码、分页响应和会话状态契约。
2. 扩展 PostgreSQL 会话/消息服务与控制器，补齐资源归属和软删除恢复。
3. 添加数据库索引与校验 SQL，并执行后端单元测试和 PostgreSQL HTTP E2E。
4. 落地统一前端 API Client、认证状态和路由守卫。
5. 替换登录、会话侧栏、聊天页和消息发送 mock。
6. 完成加载/空态/错误态、网络中断与附件降级。
7. 执行浏览器 E2E、跨域 Cookie/CSRF 验收，更新功能实现说明。

## 8. 回滚原则

数据库只新增索引或兼容字段时采用独立人工 SQL 与回滚 SQL；不删除现有会话和消息。前端通过现有 `VITE_AGENT_MODE` 保留开发期 mock 回退，但 real 模式接口失败不得伪造成功或静默展示 mock 数据。
