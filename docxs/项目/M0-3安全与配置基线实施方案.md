# FoodMate M0-3 安全与配置基线实施方案

## 1. 目的与范围

本方案将 M0-3 落为可实施、可验证的安全和配置门禁。目标是让生产环境在密钥、数据库、Cookie 或内部服务身份配置不完整时拒绝启动，而不是带着弱配置继续运行。

本文只覆盖 Java Web 服务的生产配置、Cookie/CSRF、Service JWT、日志和错误输出。Python Runtime、模型供应商、RAG、Tool/SQL Gateway 的完整生产实现仍属于后续里程碑。

事实优先级：实际代码和测试 > ADR/契约 > 本实施方案 > 其他文档。

## 2. 已确认决策

| 主题 | 决策 |
|---|---|
| Web 部署 | V1 前端静态资源与 Java `/api/**` 必须同源部署：外部访问的 scheme、host 和 port 均一致；由同一个 HTTPS 反向代理统一暴露。 |
| Cookie | 生产必须 `Secure=true`、`HttpOnly`（CSRF Cookie 除外）、`SameSite=Lax`、`Path=/`。 |
| 本地开发 | `local`/`local-stub` 可使用 HTTP 和 `cookie-secure=false`；不得使用生产密钥或生产数据。 |
| 数据库 | `prod` 禁止空配置、`localhost` 地址和已知默认凭据；连接信息只能由环境变量或受控 Secret 注入。 |
| Runtime JWT | 仅当 `RUNTIME_SERVICE_JWT_ENABLED=true` 时要求 Java 私钥、Java/Python 公钥和 Java `kid` 全部存在且格式正确；否则拒绝启动。 |
| 未部署 Runtime | 未部署 Runtime、未配置 `agent-base-url` 或关闭 Service JWT 时，所有触发 Agent 执行的接口必须在创建运行记录前失败，不得伪造“已派发/排队”成功。 |
| 日志 | 默认不记录请求体、Cookie、Authorization、JWT、密码、数据库 URL/密码、模型 Key、完整 Prompt 或原始 SQL 结果。 |
| 错误响应 | Runtime 不可用时返回 HTTP `500`、`error.code=TOOL_FAILED` 和 `error.details.runtime_code=RUNTIME_UNAVAILABLE`；其他错误同样只返回稳定错误码、request ID 和 trace ID，不得暴露异常栈、数据库或密钥细节。 |

## 3. 配置契约

### 3.1 Profile 边界

| Profile | 数据库 | Cookie Secure | 启动门禁 |
|---|---|---|---|
| `local-stub` | 不加载 DataSource | `false` | 不验证数据库或 Runtime 密钥 |
| `local` | 允许当前本机开发配置 | `false` | 允许本机 HTTP，但不得加载生产 Secret |
| `dev` | 仅环境变量注入 | 默认 `true` | 缺失 DB 配置时拒绝启动 |
| `prod` | 仅环境变量/Secret 注入 | 强制 `true` | 执行全部 M0-3 校验 |

Flyway 在所有应用 profile 中保持关闭。数据库建表、迁移和回滚只允许人工执行 `script/sql/FoodMate/` 中的版本化脚本。

### 3.2 生产启动校验

`prod` 启动时必须校验：

1. `DB_URL`、`DB_USERNAME`、`DB_PASSWORD` 非空。
2. `DB_URL` 不指向 `localhost`、`127.0.0.1` 或本机 Unix socket；用户名、密码不使用明确的开发默认值。
3. `FOODMATE_COOKIE_SECURE=true`。
4. `RUNTIME_SERVICE_JWT_ENABLED=true` 时，`RUNTIME_JAVA_PRIVATE_KEY`、`RUNTIME_JAVA_PUBLIC_KEY`、`RUNTIME_PYTHON_PUBLIC_KEY`、`RUNTIME_JAVA_KID` 均非空，且密钥可解析为 Ed25519 格式。
5. Java Runtime JWT 当前固定签发 60 秒，且不得超过 300 秒；M0-3 不新增 TTL 配置项。issuer、audience、scope 和 `kid` 继续由现有 Service JWT 契约校验。

任一条件不满足时，抛出不含 Secret 内容的启动错误并终止应用。校验错误消息只写缺失配置名或弱配置类别，不回显值。

## 4. 实施拆分

### 4.1 配置校验器

- 在 `foodmate-bootstrap` 增加集中式启动校验组件，仅对 `prod` 强制执行。
- 使用类型化配置对象承接 `foodmate.security` 与 `foodmate.runtime.service-jwt`，不在日志中输出完整配置对象。
- 保持 `local` 和 `local-stub` 可启动，避免影响已有数据库 E2E 与 stub 测试。

### 4.2 Cookie、CSRF 与会话撤销

- 保持认证 Cookie 的 `HttpOnly + Secure + SameSite=Lax + Path=/` 组合；CSRF Cookie 保持非 HttpOnly。
- 对所有认证后的 `/api/**` 写请求校验同源 Origin 和 `X-CSRF-Token`；“同源”指相同 scheme、host 和 port，与外部反向代理暴露的 Origin 一致。
- 补充登出后会话撤销、Cookie 失效和被撤销会话无法继续访问的测试。
- V1 不允许 CORS 放行任意来源；新增跨域需求必须先更新本方案和外部 API 契约。

### 4.3 Service JWT

- 保留 `EdDSA`、`kid`、签名、issuer、audience、scope、expiry 和 `jti` 校验。
- Java 侧 Service JWT 签发 TTL 固定为 60 秒；只保留 300 秒的上限校验，不在本阶段引入可配置 TTL。
- 增加启动阶段的公私钥解析校验；运行期错误统一映射为稳定认证错误，不暴露密钥解析细节。
- 将过期、缺 scope、错误 audience、错误 `kid`、损坏签名和空密钥作为独立正反例覆盖。
- 增加显式不可用 Gateway 或在派发前校验 Runtime 依赖：未配置 `agent-base-url`、Runtime 未部署或 Service JWT 已关闭时，必须先返回 `RUNTIME_UNAVAILABLE`，再停止处理；不得落库为已派发、排队或成功。

### 4.4 日志和错误脱敏

- Trace 日志只允许 request ID、trace ID、HTTP 方法、路径、状态码和耗时等白名单字段。
- 全局异常处理不得记录请求体、认证头、Cookie 或配置值；异常根因仅在受控日志中以错误类别呈现。
- 对数据库、JWT、模型和 SQL 调用添加脱敏单元测试，断言敏感样例不会出现在响应或日志输出中。

## 5. 验收测试

| 场景 | 预期 |
|---|---|
| `prod` 缺 DB 配置 | 启动失败，错误不回显值。 |
| `prod` 使用 localhost 或默认 DB 凭据 | 启动失败。 |
| `prod` Cookie Secure=false | 启动失败。 |
| JWT 已启用但缺密钥/`kid` | 启动失败。 |
| 未配置 `agent-base-url`、Runtime 未部署或 JWT 已关闭 | 应用可启动；触发 Agent 执行时返回 HTTP `500`、`TOOL_FAILED` 和 `runtime_code=RUNTIME_UNAVAILABLE`，且不产生运行记录。 |
| 正确 prod 配置 | 成功启动，Cookie 含 Secure/HttpOnly/SameSite 属性。 |
| 缺 CSRF 或跨源写请求 | 返回 403，不执行写入。 |
| 登出后继续使用旧 Cookie | 返回认证失败。 |
| JWT 正反例 | 签名、issuer、audience、scope、expiry、`kid` 逐项覆盖。 |
| 脱敏 | 响应和日志中不含密码、JWT、Cookie、DB URL/密码、模型 Key 或完整 Prompt。 |

验收命令以新增测试类和 Maven profile 为准。当前已执行并通过：

- `mvnw.cmd -pl foodmate-bootstrap,foodmate-gateway-client,foodmate-application -am test`：模块测试通过，失败数为 0。
- `mvnw.cmd -pl foodmate-bootstrap -am -Dtest=M03ProductionStartupMatrixTest -Dsurefire.failIfNoSpecifiedTests=false test`：prod 启动矩阵 6/6 通过。
- `mvnw.cmd -pl foodmate-bootstrap -am -Dfoodmate.local-e2e=true -Dtest=LocalPostgresE2ETest,LocalRuntimeRecoveryTest -Dsurefire.failIfNoSpecifiedTests=false test`：真实 PostgreSQL local E2E 通过 2/2。

上述证据覆盖配置校验、JWT/Runtime fail-closed、账号会话持久化和运行状态恢复；CSRF 跨源矩阵、Cookie 属性和完整生产启动矩阵仍需在后续验收环境补齐。没有实际通过的启动和测试证据时，不得将 M0-3 标记为完成。

## 6. 非目标与后续事项

- 不在本阶段实现 JWKS、mTLS、第三方登录、OAuth2/OIDC 或多租户。
- 不在本阶段启动 Python Runtime 或模型供应商；这些能力上线时必须复用本方案的 Secret、日志和 JWT 约束。
- 密钥轮换流程、设备会话上限、密码变更后的全量会话撤销和账号注销清理属于 M1 账户能力的后续实现，但不得削弱本阶段的登出撤销验证。
