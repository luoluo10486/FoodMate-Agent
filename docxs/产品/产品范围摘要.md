# FoodMate 需求文档 PRD

版本：v1.0  
状态：工作流入口文档  
详细来源：[产品需求文档](./产品需求文档.md)、[实现附录](../参考/实现附录.md)

## 1. 产品定位

FoodMate 是一个面向餐饮、营养、饮食记录、摄入分析和备餐规划的任务型 Agent 产品。它不是普通聊天机器人，也不是纯 RAG 知识问答系统。

用户用自然语言提出任务后，系统需要判断应该直接回答、检索知识、调用工具、追问缺失信息、写入记录、分析历史数据，还是生成可执行计划。

## 2. 目标用户

| 用户类型 | 核心任务 | 成功标准 |
|---|---|---|
| 普通用户 | 记录饮食、估算热量、问烹饪常识 | 输入少、响应快、结果容易理解 |
| 健身用户 | 跟踪蛋白质、热量、碳水、脂肪 | 输出可量化、可看趋势、可对比目标 |
| 备餐用户 | 制定周菜单和购物清单 | 计划可执行、食材复用高、浪费少 |
| 家庭用户 | 为多人制定餐食安排 | 能兼顾预算、偏好、忌口和人数 |
| 管理或运营用户 | 管理知识、提示词、工具和报表 | 行为可审计、数据稳定、任务可回放 |

## 3. 现有方案与不足

| 现有方式 | 不足 |
|---|---|
| 手工笔记或表格 | 记录成本高，连续性差，后续分析困难 |
| 普通热量 App | 结构化输入重，规划和解释能力弱 |
| 通用聊天机器人 | 数值容易不稳定，写入和审计能力弱 |
| 纯知识库搜索 | 可以回答知识，但不能完成计算、记录、分析、规划 |
| 人工备餐规划 | 耗时，很难同时平衡营养、预算和食材复用 |

FoodMate 的差异化是把 Agent 编排、确定性工具、RAG、结构化记录和可追踪执行整合到一个餐饮垂直工作流里。

## 4. MVP 范围

MVP 必须打通最小可用闭环：

- 创建和继续会话。
- 发送自然语言消息。
- 将意图路由为计算、记录、分析、规划、知识问答或混合任务。
- 当关键参数缺失时主动追问。
- 使用确定性工具完成计算、时间解析、知识检索、结构化查询、饮食记录写入和计划校验。
- 以摘要、步骤、引用、风险提示、下一步动作的结构输出结果。
- 在 UI 中展示 Agent 状态、工具状态和执行轨迹。

## 5. 暂不做

第一版不做以下内容：

- 通用开放域搜索引擎。
- 任意第三方插件市场。
- 复杂多人协作编辑。
- 未确认的敏感自动写入。
- 娱乐型自由聊天。
- 默认启用浏览器自动化、代码解释器或外部网站操作。
- 完整多 Agent 平台管理。

## 6. 核心用户流程

### 6.1 计算

用户输入“20 克鸡胸肉多少卡路里”。系统识别为计算任务，标准化食材和克重，调用营养查询或计算工具，返回数值、来源和估算说明。

### 6.2 饮食记录

用户要求记录一餐。系统抽取时间、食物、份量和单位。缺少关键字段时追问，写入前展示确认卡。

### 6.3 摄入分析

用户要求分析近期蛋白质或热量趋势。系统解析时间范围，查询饮食日志，计算总量和趋势，再输出摘要、图表数据和假设说明。

### 6.4 备餐规划

用户要求生成周计划。系统收集人数、天数、预算、目标、忌口、厨具和烹饪时间，生成计划并校验，最后输出菜单和购物清单。

### 6.5 知识问答

用户询问烹饪或营养知识。系统改写查询、检索知识库、重排结果，并带引用回答。检索为空时不能编造引用。

## 7. 成本与性能边界

| 项目 | MVP 目标 |
|---|---|
| 首屏可交互 | 页面加载后 2 秒内 |
| 普通回答首 token | 无长工具链时 3 秒内 |
| 常规任务完成 | 15 秒内 |
| 长规划任务 | 支持异步进度和状态事件 |
| 工具调用 | 每个工具必须有明确超时和重试策略 |
| 模型调用 | 统一经过 `ModelService`，记录 token、耗时、供应商、成本和状态 |
| 服务资源 | 文档解析、索引、长报告等重任务走 Worker 或异步任务 |
| 成本控制 | 模型网关或等价策略层需要支持路由、配额、降级和账单记录 |

## 8. 验收标准

- 新用户能创建会话、发送任务、看到状态并收到结构化结果。
- 关键参数缺失时系统追问，不直接猜测。
- 数值类结果由工具生成或校验。
- 知识类回答在检索成功时展示引用。
- 写操作需要确认，并产生可审计记录。
- UI 能展示当前 Agent 状态、工具进度、结果依据和失败原因。
- 每个核心场景在实现完成前至少有一条回归测试用例。
