# FoodMate 后端建设说明文档

更新时间：2026-07-03

## 1. 当前结论

目前 FoodMate 后端已经完成从 0 到 B3 工程骨架，以及 B4-1 首版数据库迁移脚本。

已经完成：

- Java 21 + Spring Boot 3.5.16 多模块 Maven 后端工程骨架。
- Maven Wrapper，保证本机没有全局 Maven 也能构建。
- 13 个后端模块骨架。
- 统一 API 响应、错误码、分页、Trace/Request 上下文、Snowflake ID。
- Spring MVC 基础 Web 层、统一异常处理、Trace 过滤器、SSE 事件 envelope。
- MyBatis-Plus 基础配置、PO 基类、软删除基础能力。
- Flyway 迁移目录和首版 PostgreSQL 核心表建表脚本。
- 24 张核心业务表的建表 SQL、索引、约束、表注释、字段注释。
- 单元测试与架构测试，用来防止基础契约被破坏。

尚未完成：

- 真实 PostgreSQL 实例尚未执行迁移。当前是 Flyway SQL 脚本已建立并通过工程测试，本机 Docker daemon 和 `psql` 不可用，所以还没有把表实际建进数据库实例。
- 用户注册/登录 API 尚未实现。
- 真实业务 PO / Mapper / Service 尚未进入 B4-2。
- Agent 主链路、RAG、模型网关、SQL Agent 仍是模块骨架。

## 2. 后端技术选择与理由

| 选择 | 当前结果 | 理由 |
| --- | --- | --- |
| Java 21 | 根 `pom.xml` 固定 Java 21 | 长期支持版本，适合后端服务；也符合项目后续可维护性。 |
| Spring Boot 3.5.16 | 后端统一使用 Spring Boot 3 | 当前文档要求 Spring Boot 3，生态稳定；没有直接跳到 4.x，避免依赖兼容风险。 |
| Maven 多模块 | 创建 13 个 Maven 子模块 | 先用模块化单体，降低早期分布式复杂度，同时保留清晰边界。 |
| Maven Wrapper | 新增 `mvnw.cmd` / wrapper 配置 | 本机无全局 Maven 时也能执行 `clean verify`。 |
| MyBatis-Plus | infra 层配置基础能力 | 更贴近 Java 后端常用 Mapper/PO 写法，便于后续控制 SQL、软删除和复杂查询。 |
| Flyway | 固定 `db/migration` 目录 | 数据库变更必须可追踪、可回滚评估、可在不同环境重复执行。 |
| PostgreSQL | SQL 使用 PostgreSQL 类型和约束 | 支持 `JSONB`、部分索引、强约束，适合 Agent/RAG 这类半结构化数据。 |
| Spring MVC | Web 层基础 | 普通 HTTP API 简洁稳定；当前没有引入 WebFlux 的必要复杂度。 |
| SseEmitter | B5 事件流方案 | SSE 足够支撑 Agent 运行状态流式返回，比 WebSocket 更简单，和 HTTP 基础设施兼容更好。 |

## 3. 模块建设情况

| 模块 | 当前作用 | 设计理由 |
| --- | --- | --- |
| `foodmate-bootstrap` | 应用启动装配，持有 `FoodMateApplication` 和 profile 配置 | 把启动入口集中在一个模块，避免业务模块相互污染。 |
| `foodmate-api` | Web 层、Controller、异常处理、SSE envelope、Trace filter | 对外 HTTP/SSE 契约统一放这里，后续 Controller 不直接碰 Mapper。 |
| `foodmate-application` | 应用服务编排骨架 | 后续承接用例级服务，例如注册、登录、创建会话。 |
| `foodmate-domain` | 领域模型骨架 | 保留核心业务规则承载位置，避免业务逻辑全部堆在 Controller。 |
| `foodmate-orchestrator` | Agent 编排模块骨架 | 后续承接 Agent 计划、执行、状态流转。 |
| `foodmate-rag` | RAG 能力模块骨架 | 后续承接知识库检索、chunk 检索、embedding 相关逻辑。 |
| `foodmate-tool` | 工具注册与调用模块骨架 | 后续承接工具定义、权限、执行适配。 |
| `foodmate-sql-agent` | SQL Agent 模块骨架 | 后续承接自然语言到 SQL、SQL 校验、查询审计。 |
| `foodmate-model` | 模型网关领域模块骨架 | 后续承接模型路由、调用、用量统计。 |
| `foodmate-gateway-client` | 外部网关客户端骨架 | 后续承接第三方模型、存储、消息等客户端封装。 |
| `foodmate-infra` | 基础设施层，MyBatis-Plus、PO 基类、Flyway 脚本、软删除 | 数据库相关能力必须集中在 infra，避免上层直接依赖持久化细节。 |
| `foodmate-worker` | 后台任务模块骨架 | 后续承接异步任务、消息消费、定时任务。 |
| `foodmate-shared` | 通用契约、错误、分页、ID、Trace | 只放无业务方向的公共能力，避免形成反向依赖。 |

## 4. 已实现的通用后端能力

### 4.1 统一 API 响应

已建立 `ApiResponse<T>`、`ErrorBody`、`ResponseMeta`。

理由：

- 所有接口成功/失败结构保持一致。
- 错误响应中统一带 `request_id` 和 `trace_id`，方便排查问题。
- 前端不用针对每个接口重新猜响应结构。

### 4.2 错误码与业务异常

已建立 `ErrorCode`、`BusinessException` 和 `GlobalExceptionHandler`。

理由：

- 业务错误和系统错误要分开表达。
- 参数校验、业务异常、未知异常都应该返回统一错误结构。
- 未知异常不能把堆栈暴露给前端。

### 4.3 分页契约

已建立 `PageRequest`、`PageResult`。

理由：

- 后续列表接口统一使用同一套分页字段。
- 避免每个业务模块自己定义分页参数。

### 4.4 Trace 与 Request 上下文

已建立 `TraceContext`、`RequestContext`、`TraceContextHolder`、`TraceContextFilter`。

理由：

- 每个请求都有可追踪 ID。
- 后续日志、审计、模型调用、工具调用都能串联同一个 trace。
- SSE 事件也可以携带同一上下文。

### 4.5 ID 生成与 Long ID 序列化

已建立 `IdGenerator`、`SnowflakeIdGenerator`、`LongIdJsonSerializer`。

理由：

- Java 内部使用 `Long` 存 Snowflake ID，数据库使用 `BIGINT`。
- 对外 JSON 把 Long ID 序列化为字符串，避免 JavaScript 数字精度丢失。

### 4.6 SSE 事件 envelope

已建立 `SseEventEnvelope`、`SseTraceContext`。

当前约定：

| 字段 | 用途 |
| --- | --- |
| `event_type` | 事件类型，例如后续 `run.started`、`run.delta`、`run.completed`。 |
| `run_id` | 本次 Agent 运行 ID，对外作为字符串。 |
| `timestamp` | 事件产生时间。 |
| `payload` | 事件业务负载。 |

理由：

- B5 以后 Agent 会流式返回状态，必须先固定事件外壳。
- 前端只需要识别 envelope，再根据 `event_type` 处理不同 payload。

### 4.7 持久化基础

已建立 `BasePo`、`SoftDeleteSupport`、`SoftDeleteRepositorySupport`、`RestoreCommand`、`MybatisPlusConfig`。

理由：

- 所有 PO 统一继承审计字段和软删除字段。
- 正常删除走逻辑删除，不直接物理删除数据。
- 恢复操作必须显式走 admin restore 语义。
- Mapper 只能放在 infra 层，避免 Controller/Application 层直接操作数据库实现细节。

## 5. 配置与启动

已建立配置文件：

- `application.yml`
- `application-local.yml`
- `application-dev.yml`
- `application-prod.yml`
- `application-local-stub.yml`

当前默认可用 `local-stub` profile 启动。

理由：

- B3 阶段需要在没有 PostgreSQL、Redis、Milvus、MinIO、RocketMQ 的情况下启动后端骨架。
- 真实数据库连接和 Flyway 执行放到后续 profile 中逐步启用。

常用命令：

```powershell
.\mvnw.cmd clean verify
.\mvnw.cmd -pl foodmate-bootstrap spring-boot:run
```

## 7. 数据库迁移建设情况

迁移脚本位置：

```text
foodmate-infra/src/main/resources/db/migration/V1__init_core_schema.sql
```

当前脚本做了这些事：

- 创建 24 张核心表。
- 使用 `BIGINT` 主键，配合 Java Snowflake ID。
- 使用 `JSONB` 存放可扩展结构。
- 使用 `VARCHAR + CHECK` 代替数据库 enum，便于后续迁移和兼容。
- 所有表都带审计字段和软删除字段。
- 建立关键唯一索引、查询索引和外键。
- 为每张表和每个字段添加 PostgreSQL `COMMENT ON TABLE` / `COMMENT ON COLUMN` 注释。

重要说明：

- 当前没有真实 PostgreSQL 数据库实例可用，所以脚本尚未 apply 到数据库。
- 等 Docker Desktop 或本地 PostgreSQL 可用后，启动真实 profile 时 Flyway 可以执行该迁移。

## 9. 数据库表与字段说明

### 9.1 `users` 用户账号表

用途：保存用户登录账号、安全状态和基础展示信息。

| 字段 | 用途 |
| --- | --- |
| `user_id` | 用户主键，Snowflake ID。 |
| `tenant_id` | 租户 ID，预留多租户。 |
| `user_no` | 对外展示或业务引用的稳定用户编号。 |
| `username` | 登录用户名。 |
| `email` | 登录邮箱。 |
| `password_hash` | 密码哈希，不存明文密码。 |
| `nickname` | 用户昵称。 |
| `role` | 用户角色，如普通用户、管理员、运营人员。 |
| `avatar_url` | 当前头像地址。 |
| `status` | 账号状态：正常、禁用、锁定。 |
| `last_login_at` | 最近一次登录时间。 |
| `password_updated_at` | 最近一次修改密码时间。 |
| `login_failed_count` | 连续登录失败次数。 |
| `locked_until` | 账号锁定截止时间。 |
| 公共字段 | 见“公共字段说明”。 |

关键约束与索引：

- `user_no` 全局唯一。
- `username`、`email` 在未删除数据中唯一。
- `role`、`status` 使用 CHECK 约束限定合法值。

## 10. 为什么第一版这样建表

### 10.1 先覆盖核心链路

第一版表结构覆盖了：

- 用户与认证。
- 用户画像和头像。
- 对话会话与消息。
- Agent 运行与工具调用。
- 饮食记录、分析报告、膳食计划、购物清单。
- 用户记忆与会话摘要。
- 知识库文档和切片。
- SQL Agent 数据源、Schema 白名单、查询审计。
- 工具注册和工具 schema 版本。
- 模型用量和模型路由。
- 操作审计。

这样 B4 后续可以优先实现用户、认证、个人资料、会话，再逐步接 Agent 主链路，不需要频繁推翻底层表结构。

### 10.2 使用 JSONB 的理由

Agent、RAG、模型调用、营养分析结果都存在大量半结构化字段。如果一开始把所有 payload 都拆成极细的关系表，会导致早期 schema 变更非常频繁。

所以第一版采用：

- 关键查询字段结构化。
- 变化快的内容放入 `JSONB`。
- 等业务稳定后再把高频查询字段抽成独立列。

### 10.3 使用逻辑删除的理由

饮食记录、会话、审计、模型调用、工具调用都可能需要追溯。物理删除会让问题排查和用户恢复变难。

所以第一版所有核心表统一带：

- `is_deleted`
- `deleted_at`
- `deleted_by`

正常业务删除走逻辑删除，恢复必须显式处理。

### 10.4 不使用数据库 enum 的理由

状态字段当前使用 `VARCHAR + CHECK`。

理由：

- 比 PostgreSQL enum 更容易演进。
- 后续新增状态时迁移成本更低。
- 仍然能通过 CHECK 保持基本数据约束。

### 10.5 先不建 `tenants` 表的理由

当前部分表保留 `tenant_id`，但没有建立 `tenants` 表。

理由：

- 当前产品第一阶段主要是单用户/单应用闭环。
- 先预留多租户字段，避免未来大规模改表。
- 等真正进入组织、团队、商户或企业租户能力时，再补 `tenants`、成员、角色授权等表。

## 11. 当前风险与待补项

| 风险/待补项 | 说明 | 建议 |
| --- | --- | --- |
| 未真实执行 Flyway | 本机 Docker daemon 和 `psql` 不可用 | 启动 Docker Desktop 或安装 PostgreSQL 后执行真实迁移验证。 |
| 还没有 PO / Mapper | 当前只有建表脚本和基础 PO 能力 | 下一步做 B4-2，先覆盖用户、资料、token、头像、会话、消息。 |
| 还没有认证 API | 注册、登录、刷新 token 尚未实现 | B4-3/B4-4 开始补认证应用服务和 Controller。 |
| JSONB 字段未细化 schema | 当前只固定存储位置，没有定义完整 JSON 内部结构 | 后续结合接口文档补充 payload schema。 |
| 审计字段自动填充未完全接入业务 | 基础字段已有，真实写入还没开始 | Mapper/Service 落地时补 MetaObjectHandler 或应用层填充。 |

## 12. 下一步建议

建议按正常进度进入 B4-2：

1. 建立 `users`、`user_profiles`、`auth_refresh_tokens`、`user_avatar_assets`、`sessions`、`messages` 的 PO。
2. 建立对应 MyBatis-Plus Mapper，且 Mapper 只放在 `foodmate-infra`。
3. 建立最小 Repository/Adapter，让 application 层不直接接触 Mapper。
4. 加入 H2 或 Testcontainers/PostgreSQL 集成测试方案。
5. PostgreSQL 可用后执行 Flyway 真库验证。
6. 再进入注册、登录、刷新 token、获取当前用户资料等 API。
