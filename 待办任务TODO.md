## Phase 1：静态 UI 原型

### P1-1. 创建前端项目骨架

目标：

- 创建第一版用于静态 FoodMate 页面展示的前端应用。

允许改动：

- 新增前端应用目录。
- 新增所选前端栈需要的 package 和配置文件。
- 更新 README，加入前端启动命令。

验收标准：

- 应用可以本地启动。
- 不依赖后端。
- 首页无控制台错误。

### P1-2. 实现首页静态原型

目标：

- 按 `docs/UI设计文档.md` 实现首页。

允许改动：

- 前端页面和组件文件。
- 会话和快捷任务的静态 mock 数据。

验收标准：

- 展示顶部栏、左侧栏、快捷任务卡、推荐问题、输入区和状态行。
- 通过 mock 开关或组件状态覆盖空态、加载中、错误态。
- 桌面端和移动端布局可用。

### P1-3. 实现会话页静态原型

目标：

- 实现主 Agent 执行工作区。

允许改动：

- 前端页面和组件文件。
- mock 消息、工具轨迹、引用、追问卡和确认卡。

验收标准：

- 展示消息流、运行状态、工具轨迹、引用、结果卡和输入区。
- 覆盖执行中、追问中、完成、失败、取消状态。
- mock 流式状态下输入区布局不跳动。

### P1-4. 抽取共享 UI 组件

目标：

- 抽取首页和会话页需要复用的组件。

允许改动：

- UI 组件和本地样式。

验收标准：

- 存在会话列表、任务卡、输入区、Agent 状态条、工具轨迹项、结果卡、引用块、追问卡、确认卡、空态、错误态组件。
- 组件接收 mock props，不和单个页面强绑定。

## Phase 2：Mock 交互

### P2-1. 添加假 Agent 事件回放

目标：

- 在没有后端的情况下模拟 Agent 执行。

允许改动：

- 前端 mock 事件工具。
- 会话页状态处理。

验收标准：

- 点击快捷任务或发送消息后，按 routing、planning、retrieving、executing、validating、composing、completed 顺序播放状态。
- 工具轨迹和答案内容逐步更新。
- 停止操作会把运行状态切换为 cancelled。

### P2-2. 添加 mock 追问和确认流程

目标：

- 在接后端前验证关键交互。

允许改动：

- 前端 mock 状态和 UI 事件处理。

验收标准：

- 缺参数的规划类输入展示追问卡。
- 饮食记录写入类输入在“保存”前展示确认卡。
- 取消和重试路径可见。

## Phase 3：后端最小链路

### P3-1. 创建模块化后端骨架

目标：

- 建立第一版 Java/Spring 工程结构。

允许改动：

- Maven 根工程和选定 MVP 模块。
- 基础 Spring Boot 应用。
- 初始配置文件。

验收标准：

- 项目可以本地构建。
- 模块边界遵循 `docs/架构文档.md`。
- smoke test 不依赖真实模型或数据库。

### P3-2. 实现 Session、Message、AgentRun API

目标：

- 支持最小会话生命周期。

允许改动：

- API、application、domain、persistence 层。
- 根据当前阶段选择内存实现或数据库实现。

验收标准：

- 可以创建会话。
- 可以列出会话。
- 可以发送消息并创建 AgentRun。
- 可以查询消息和运行详情。
- API 响应结构遵循 `FoodMate-接口与数据规范.md`。

### P3-3. 实现 SSE 运行事件

目标：

- 向 UI 流式输出运行生命周期事件。

允许改动：

- API SSE 端点。
- Orchestrator mock 执行。

验收标准：

- 输出已约定的事件名。
- 前端可以用本地后端事件替换假事件回放。
- 支持 failed 和 cancelled 事件。

## Phase 4：核心能力接入

### P4-1. 添加计算器和时间解析工具

目标：

- 实现第一批确定性工具。

允许改动：

- 工具注册、工具契约、calculator、time_parser、测试。

验收标准：

- 工具元数据包含 schema、超时、是否可重试、幂等性。
- 计算和相对时间示例通过测试。
- 工具调用会记录在运行轨迹中。

### P4-2. 添加知识检索链路

目标：

- 打通知识问答的第一条 RAG 流程。

允许改动：

- RAG service 接口。
- mock 或最小检索实现。
- 引用组装。

验收标准：

- 知识问答命中时返回引用。
- 检索为空时不编造引用。
- UI 可以展示来自后端事件或最终结果的引用。

### P4-3. 添加受控饮食记录写入

目标：

- 闭合第一个业务写入链路。

允许改动：

- FoodLog 领域对象、写入 API、writer 工具、确认处理。

验收标准：

- 缺少关键字段时触发追问。
- 写入前需要确认。
- 保存记录可审计，并显示在运行结果中。

## 工作规则

- 一个分支或一次 AI Coding Agent 会话优先只处理一个 TODO 项。
- 实现某个 TODO 前，先读 `README.md`、相关 `docs/` 入口文档，以及入口文档引用的详细来源文档。
- 除非 TODO 明确要求，不修改运行时 API。
- 实现工作流任务时，不删除或重写原有长文档。
- 每个任务完成后，最终回复必须包含简短验证说明。
