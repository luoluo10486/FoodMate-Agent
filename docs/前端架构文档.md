# FoodMate 前端架构文档

版本：v1.0  
状态：Phase 1-2 前端实现入口文档  
详细来源：`docs/UI设计文档.md`、`FoodMate-接口与数据规范.md`、`FoodMate-实现附录.md`

## 1. 定位

本文是 FoodMate 前端实现入口，覆盖 Phase 1 静态 UI 原型和 Phase 2 Mock 交互。第一轮不创建真实后端依赖，不调用真实接口，只通过 mock 数据和 mock Agent 事件回放验证页面结构、组件边界、状态展示和交互体验。

前端第一轮目标：

- 让首页、会话页、分析页、规划页具备可运行的静态原型。
- 让会话页能在无后端情况下播放一轮 Agent mock 任务。
- 让追问卡、确认卡、工具轨迹、引用、失败和取消状态先在前端闭环。
- 为后续接入 `/foodmate/agent-runs/{agent_run_id}/events` SSE 事件保留数据结构和状态映射。

## 2. 技术栈决策

锁定技术栈：

- React
- Vite
- TypeScript
- Arco Design
- CSS Modules

选择 React 而不是 Vue 的原因：

- FoodMate 更像 Agent 工作台，不是普通表单后台；核心复杂度在消息流、工具轨迹、Agent 状态和事件回放。
- React 的组件组合和 TypeScript 契约适合承载 `AgentRun`、`ToolCall`、`SSEEvent` 这类状态驱动 UI。
- 后续接入 SSE、mock event replay、局部状态机和复杂业务卡片时，React 生态和 AI Coding Agent 生成稳定性更好。
- Vue 3 也可实现本项目，但当前文档、系统设计和执行路线已偏向 React，切换会增加文档和实现同步成本。

选择 Arco Design 的原因：

- Arco 提供稳定的 Button、Input、Modal、Drawer、Tooltip、Tabs、Table、Card、Tag、Progress、Skeleton、Message 等基础控件。
- FoodMate 需要比纯手写控件更快地搭出可用界面，同时保持比通用后台更强的产品气质。
- Arco 参与整体视觉优化，但不直接套 Arco Pro 后台模板；FoodMate 的 Agent 工作台布局、色彩、卡片层级和动效仍由项目自定义。

## 3. Phase 边界

Phase 1：静态 UI 原型

- 创建前端应用骨架。
- 实现首页、会话页、分析页、规划页的静态页面。
- 抽取共享组件，组件通过 props 接收 mock 数据。
- 覆盖空态、加载中、错误态、完成态等静态状态。

Phase 2：Mock 交互

- 使用前端本地 mock event sequence 模拟 AgentRun 生命周期。
- 支持消息发送后的状态推进、工具轨迹出现、答案逐步输出。
- 支持 `waiting_user` 追问流程和确认流程。
- 支持 `failed`、`cancelled`、停止按钮和重试入口。

不属于 Phase 1-2：

- 不接真实后端。
- 不创建真实数据库、SSE 服务或认证系统。
- 不实现真实 RAG、Tool、SQL Agent、ModelService。
- 不引入大型全局状态方案，除非 Phase 2 事件回放复杂度证明必要。

## 4. 路由与页面

推荐路由：

| 路由 | 页面 | Phase |
|---|---|---|
| `/` | 首页 | Phase 1 |
| `/login` | 登录入口与鉴权占位 | Phase 1 |
| `/chat/:sessionId?` | 会话页 | Phase 1-2 |
| `/analysis` | 分析页 | Phase 1 |
| `/planning` | 规划页 | Phase 1 |
| `/knowledge` | 知识库检索占位 | Phase 1 |

路由行为：

- Phase 1 可以先使用前端路由，不依赖后端 session。
- 首页快捷任务可跳转到会话页并带入 mock prompt。
- 会话页如果没有 `sessionId`，使用 mock 当前会话。
- 分析页和规划页第一版使用静态 mock 数据。
- `/login` 第一版只提供 mock 登录入口和鉴权状态说明，不写入真实 token。
- `/knowledge` 第一版只展示知识库检索与 ACL 过滤占位，不接真实 RAG 接口。

```text
frontend/

    layouts/

    components/

      feedback/

    features/
      agent/
      session/
      analysis/
      planning/
    mock/
      sessions.ts
      agentEvents.ts
      analysis.ts
      mealPlans.ts
    styles/
      tokens.css
      arco-theme.css
      motion.css
    types/
      agent.ts
      tool.ts
      session.ts
      ui.ts
```

分层规则：

- `pages/` 只负责页面组合，不直接写复杂业务状态。
- `components/` 放通用 UI 组件，不绑定具体业务数据来源。
- `features/agent/` 放 AgentRun、ToolCall、mock event replay、状态映射。
- `features/session/` 放会话列表、消息、Composer 相关逻辑。
- `mock/` 放静态数据和事件序列，Phase 2 不写死在组件里。
- `styles/` 放 FoodMate design tokens、Arco 主题覆盖和轻量动效。

## 6. Arco Design 使用策略

Arco 参与整体视觉风格，但必须服务 FoodMate 的餐饮 Agent 工作台定位。

使用 Arco 的场景：

| Arco 组件 | FoodMate 用法 |
|---|---|
| `Button` | 发送、停止、确认、重试、快捷任务操作 |
| `Input` / `Input.TextArea` | Composer 输入、搜索会话、追问回复 |
| `Modal` | 高风险确认、保存计划、删除或恢复确认 |
| `Drawer` | 工具轨迹详情、引用详情、移动端侧栏 |
| `Tooltip` | 工具按钮、状态图标、不熟悉的操作 |
| `Tabs` | Tools / Agents 展开面板或管理视图占位 |
| `Table` | 餐食计划、分析明细、后续管理列表 |
| `Card` | 指标卡、结果卡、推荐任务卡的基础容器 |
| `Tag` | Agent 状态、Tool 状态、风险等级 |
| `Progress` | Agent 阶段推进、计划校验进度 |
| `Skeleton` | 页面加载和局部加载 |
| `Message` | 操作反馈、mock 保存成功、错误提示 |

必须自定义的场景：

- 工作台整体布局。
- 左侧会话栏。
- 底部 Composer。
- Tools / Agents 状态区。
- 消息流和 Agent 运行轨迹。
- ResultCard、CitationBlock、ClarificationCard、ConfirmationCard。
- FoodMate 的背景、卡片层级、餐饮相关色彩和轻量动效。

禁止事项：

- 不直接套 Arco Pro 后台模板。
- 不把所有内容都做成默认 Arco Card。
- 不使用默认蓝色作为唯一主视觉。
- 不让组件库默认间距破坏 `1440 x 760` 单屏工作台结构。

## 7. 视觉与动效策略

视觉方向：

- Arco 的控件清晰度 + FoodMate 的温暖餐饮任务感。
- 浅色中性背景，搭配绿色、陶土橙、信息蓝和少量危险红。
- 页面应像一个每天可用的 Agent 工作台，而不是普通后台管理系统。

动效允许范围：

- 页面进入时的轻量 fade/slide。
- 首页任务卡 hover/focus 的轻微抬升和阴影变化。
- Agent 状态推进时的进度条和状态 Tag 过渡。
- 工具轨迹展开/收起时的高度或透明度过渡。
- 追问卡、确认卡出现时的轻量弹入。
- answer stream mock 输出时的文本逐步出现。

性能边界：

- Phase 1-2 优先使用 CSS transition / animation。
- 不引入 Framer Motion 等额外动画库，除非后续明确需要。
- 动效不阻塞首屏渲染。
- 底部输入区高度必须稳定。
- 消息流、工具轨迹和表格只允许局部滚动。
- 必须支持 `prefers-reduced-motion`，用户减少动态时关闭非必要动效。

## 8. Mock 数据与事件

Phase 1 mock 数据：

- sessions：会话列表、置顶、最近会话、当前会话。
- tasks：快捷任务、推荐问题。
- messages：用户消息、助手消息、结果卡。
- analysis：指标、趋势、目标、异常提示。
- mealPlans：约束、多日菜单、购物清单、预算、校验结果。

Phase 2 mock event sequence 必须模拟：

- `run.created`
- `run.routed`
- `run.clarification_requested`
- `run.planned`
- `run.retrieval_started`
- `run.retrieval_finished`
- `run.tool_started`
- `run.tool_finished`
- `run.answer_stream`
- `run.completed`
- `run.failed`
- `run.cancelled`

状态规则：

- AgentRun 状态和 UI 状态必须遵循 `docs/UI设计文档.md` 的映射表。
- ToolCall 状态至少覆盖 `pending`、`running`、`success`、`failed`、`timeout`、`cancelled`。
- 停止按钮触发后进入 `cancelled`，停止追加 mock 工具和答案事件。
- 追问流程进入 `waiting_user`，用户补充后回到 `planning`。
- 确认流程未确认前不能展示“已保存”。

## 9. 状态管理

Phase 1：

- 页面使用静态 mock 数据和组件 props。
- 不引入全局状态库。

Phase 2：

- 优先使用 React state / reducer 管理 mock event replay。
- 建议把事件回放封装为 `useMockAgentRun` 或等价 hook。
- 如果后续 mock 交互跨页面共享明显变复杂，再评估 Zustand 等轻量状态库。
- 不提前引入 Redux、MobX 或复杂状态框架。

## 10. 验收标准

Phase 1 完成时：

- 首页、会话页、分析页、规划页可本地访问。
- 页面结构符合 SVG 原型的信息层级。
- Arco 组件和 FoodMate 自定义样式融合，不呈现通用后台模板感。
- 桌面端首屏能看到侧栏、主内容、Tools/Agents、Composer。
- 移动端不横向溢出。

Phase 2 完成时：

- 无后端时可演示一轮完整 Agent mock 任务。
- Agent 状态、工具轨迹、引用、答案流同步变化。
- 追问、确认、失败、取消都有明确 UI。
- mock 事件名与 `FoodMate-接口与数据规范.md` 的 `run.*` SSE 事件一致。
- 后续接真实 SSE 时，不需要重写核心 UI 组件。

## 11. 当前前端对照审计

已覆盖：

- 首页、会话页、分析页、规划页、知识库占位页和登录入口可访问。
- 首页快捷任务和推荐任务可带入 mock prompt 并进入会话页。
- 会话页覆盖计算、记录、分析、规划、知识问答、追问、确认、失败和取消状态。
- 工具轨迹支持状态、耗时、错误和输入输出摘要展开。
- 分析页蛋白质目标采用 `体重 kg × 1.5-2.0` 的 mock 配置，并展示推荐区间。
- 规划页保留约束摘要、菜单、购物清单、预算和保存确认。
- 移动端已避免横向溢出，内部滚动条默认隐藏。

仍待后端或后续 Phase 完成：

- `/login` 目前是 mock 登录入口，真实登录、刷新、退出和路由守卫需等待认证接口定稿。
- `/knowledge` 目前是知识库占位页，真实文档导入、检索、ACL 过滤和引用详情需等待 RAG/知识库接口。
- Tools / Agents 底部状态目前显示计数，完整展开面板可在真实工具注册接口接入后完善。
- 会话搜索、归档、恢复、删除和历史分页目前仍是静态 mock。
