# FoodMate 架构文档

版本：v1.0  
状态：工程工作流入口文档  
详细来源：`FoodMate-系统设计与技术方案.md`、`FoodMate-接口与数据规范.md`、`FoodMate-Java工程骨架与模块设计.md`、`FoodMate-智能体行为与工具协议.md`、`FoodMate-数据库设计与接口工具清单.md`

## 1. 系统形态

FoodMate 第一阶段采用模块化单体，提前定义未来可拆服务边界。系统核心形态是 Agentic RAG：Agent 编排是主控层，RAG、工具、SQL Agent 和模型调用都是受控能力。

主运行链路：

```text
UI -> API -> Application Service -> Agent Orchestrator
   -> Router / Planner / Executor / Validator / Composer
   -> RAG / Tool Service / SQL Agent / ModelService / Data Service
   -> PostgreSQL / Redis / Milvus / MinIO / RocketMQ
```

## 2. 模块职责

| 模块 | 职责 | 不允许做 |
|---|---|---|
| UI | 输入、渲染、状态订阅、确认操作 | 决定业务真值或绕过后端校验 |
| API | 鉴权、请求校验、REST/SSE 协议、响应封装 | 直接访问数据库、Milvus 或模型供应商 SDK |
| Application | 用例协调、事务、DTO 组装 | 承担多步 Agent 推理 |
| Orchestrator | 意图路由、任务规划、工具和 RAG 协调、最终组装 | 实现底层工具或直接写数据库 |
| RAG | 查询理解、混合检索、重排、引用组装、ACL 过滤 | 生成最终用户答案或写业务数据 |
| Tool Service | 工具注册、schema 校验、确定性执行 | 自行承担任意用户意图路由 |
| SQL Agent | Schema Catalog、只读 SQL 生成、SQL Guard、MCP 执行、审计 | 执行写 SQL 或跳过校验 |
| ModelService | 统一模型调用、结构化输出、Embedding、Rerank、用量日志 | 把供应商 SDK 直接暴露给业务模块 |
| Data Service | 会话、消息、饮食记录、计划、报告、审计记录 | 隐藏软删除或权限行为 |
| Worker | 异步解析、索引、摘要、长任务 | 直接承接同步用户请求 |

## 3. 核心数据约定

详细 schema 以 `FoodMate-接口与数据规范.md` 和 `FoodMate-数据库设计与接口工具清单.md` 为准。

MVP 稳定实体：

- `Session`：用户任务工作区。
- `Message`：用户和助手消息。
- `AgentRun`：一次由用户消息触发的执行。
- `ToolCall`：挂在 AgentRun 下的一次工具调用。
- `RetrievedReference`：检索引用和命中结果。
- `UserMemory`：长期用户偏好或约束。
- `FoodLog`：持久化饮食记录。
- `MealPlan`：生成并可保存的餐食计划。
- `AnalysisReport`：结构化摄入分析结果。

ID 和删除约定：

- 数据库主键使用应用层生成的 Snowflake `BIGINT`。
- API 和 SSE 对外把 ID 序列化为字符串。
- 删除默认是软删除。
- 默认查询排除 `is_deleted = true`。
- 恢复操作使用显式 `POST /restore` 路由。

## 4. 请求与 Agent 流程

标准用户消息流程：

1. UI 向会话消息接口发送用户输入。
2. API 校验鉴权、请求结构和会话访问权限。
3. Application 持久化用户消息并创建 `AgentRun`。
4. Orchestrator 路由意图，判断是否需要追问。
5. Planner 为复杂任务生成最小可执行计划。
6. Executor 按需调用 RAG、工具、SQL Agent 或 ModelService。
7. Validator 校验数值、计划、写入或策略敏感结果。
8. Composer 生成最终结构化回答。
9. API 通过 SSE 输出状态和答案事件。
10. Data 层持久化运行轨迹、工具调用、引用和最终结果。

SSE 事件沿用现有接口规范：

- `run.created`
- `run.routed`
- `run.clarification_requested`
- `run.planned`
- `run.retrieval_started`
- `run.retrieval_finished`
- `run.tool_started`
- `run.tool_finished`
- `run.answer_stream`
- `run.completed`
- `run.failed`

## 5. 工具与能力边界

P0 工具：

| 工具 | 用途 |
|---|---|
| `calculator` | 确定性数学和单位计算 |
| `time_parser` | 相对时间归一化 |
| `knowledge_search` | 知识检索入口 |
| `database_query` | 通过 SQL Agent/MCP 访问结构化数据 |
| `food_log_writer` | 受控饮食记录写入 |
| `plan_validator` | 餐食计划、营养和预算校验 |

MVP 主链路稳定后再增加高频 P1 工具：

- `nutrition_lookup`
- `meal_plan_generator`
- `shopping_list_generator`
- `summary_generator`
- `data_exporter`

工具规则：

- 工具必须定义输入 schema、输出 schema、超时、重试策略、幂等性、权限和审计级别。
- 写工具默认需要用户确认，除非已明确安全且授权。
- 工具失败必须返回结构化错误，并在 Agent 轨迹中可见。

## 6. RAG 与 SQL Agent 边界

RAG 处理非结构化或半结构化知识：

- 烹饪知识。
- 食材保存。
- 营养参考。
- SOP 和产品规则。
- 内部知识文档。

SQL Agent 处理结构化数据：

- 饮食日志。
- 分析报告。
- 用户偏好。
- 报表和管理表。

SQL Agent 硬规则：

- 只允许 `SELECT` 或 `WITH ... SELECT`。
- 必须通过 SQL Guard。
- 必须注入用户或租户过滤。
- 必须限制行数和超时。
- 必须写入查询审计。
- 不能把数据库凭证或 JDBC 连接直接暴露给模型。

## 7. 模型边界

业务模块只调用 `ModelService`，不能直接调用供应商 SDK。

`ModelService` 至少支持：

- `chat`
- `chatStream`
- `structuredOutput`
- `toolCall`
- `embed`
- `rerank`

每次模型调用必须记录：

- request ID 和 trace ID
- 用户或租户信息
- 供应商和模型
- token 用量
- 耗时
- 成本
- 状态和错误

模型网关可以先以内嵌模块实现；当模型供应商路由、租户策略、配额或成本控制复杂后，再拆成独立服务。

## 8. 第一版实现顺序

工作流文档完成后，推荐按以下顺序实现：

1. 使用 mock 数据创建前端静态首页和会话页。
2. 添加假 SSE 事件回放，模拟 Agent 状态和工具轨迹。
3. 构建后端最小会话、消息、运行 API。
4. 添加真实 SSE 运行事件。
5. 添加 P0 计算器和时间解析工具。
6. 添加知识检索链路。
7. 添加受控饮食记录写入。
8. 添加分析和规划流程。
