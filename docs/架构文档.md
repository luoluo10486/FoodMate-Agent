## 1. 系统形态

FoodMate 采用两个受控运行时：Java 业务控制面保持模块化单体，Python Agent Runtime 作为独立进程承载智能执行。系统核心形态是 Agentic RAG，但 Agent 只能提出决策和工具调用建议，业务授权与最终执行仍由 Java 控制。

主运行链路：

```text
UI -> Java API / Auth / Application / AgentRun / SSE
   -> Python Agent Runtime
      -> Router / Planner / RAG / Model / Composer
      -> proposed_tool_call / sql_proposal
   -> Java Tool Gateway / Policy / SQL Guard / Business Service
   -> PostgreSQL / Redis / Milvus / MinIO / RocketMQ
```

当前仓库仍保留 13 个 Maven 模块，其中 `foodmate-orchestrator`、`foodmate-rag`、`foodmate-sql-agent`、`foodmate-model` 只有占位代码，属于待迁移旧骨架。Python Runtime 尚未创建，文档中的 Python 内容是已批准的目标架构，不代表已经实现。

## 2. 模块职责

| 模块 | 职责 | 不允许做 |
|---|---|---|
| UI | 输入、渲染、状态订阅、确认操作 | 决定业务真值或绕过后端校验 |
| Java API | 外部 REST/SSE、鉴权、请求校验、响应封装 | 把 Python 服务直接暴露给前端 |
| Java Application | 用例协调、事务、会话、消息、AgentRun 和 DTO | 承担 Prompt 或多步模型推理 |
| Python Agent Runtime | 路由、规划、RAG 编排、模型调用、结果校验与回答组装 | 直接写业务库、信任用户身份声明、批准敏感工具 |
| Java Tool Gateway | 工具注册、schema、权限、确认、幂等、确定性执行与审计 | 让 Python 绕过策略层调用业务实现 |
| Python SQL Planner | 查询理解、选择授权 schema、生成只读 SQL proposal | 持有数据源凭据或执行 SQL |
| Java SQL Gateway | Schema 授权、SQL Guard、租户过滤、只读执行、审计 | 接受未校验 SQL 或执行写 SQL |
| Python RAG | 查询改写、检索编排、rerank、引用组装 | 决定最终 ACL 或修改知识库业务状态 |
| Java Knowledge Control | 文档元数据、ACL、索引任务状态和访问授权 | 把未经授权的文档范围交给 Python |
| Python Model Runtime | 模型路由执行、结构化输出、Embedding、Rerank | 将供应商响应当成已授权业务动作 |
| Java Data Service | 业务数据、运行记录、工具/SQL/操作审计 | 允许 Python 直接操作 Mapper 或数据库连接 |

## 3. 核心数据约定

详细 schema 以 `FoodMate-接口与数据规范.md` 和 `FoodMate-数据库设计与接口工具清单.md` 为准。

MVP 稳定实体：

- `Session`：用户任务工作区。
- `Message`：用户和助手消息。
- `AgentRun`：一次由用户消息触发的执行。
- `ToolCall`：挂在 AgentRun 下的一次工具调用。
- `RetrievedReference`：检索引用和命中结果。
- `UserMemory`：长期用户偏好或约束。
- `FoodLog`：持久化饮食记录。
- `MealPlan`：生成并可保存的餐食计划。
- `AnalysisReport`：结构化摄入分析结果。

ID 和删除约定：

- 数据库主键使用应用层生成的 Snowflake `BIGINT`。
- API 和 SSE 对外把 ID 序列化为字符串。
- 删除默认是软删除。
- 默认查询排除 `is_deleted = true`。
- 恢复操作使用显式 `POST /restore` 路由。

## 4. 请求与 Agent 流程

标准用户消息流程：

1. UI 向会话消息接口发送用户输入。
2. API 校验鉴权、请求结构和会话访问权限。
3. Application 持久化用户消息并创建 `AgentRun`。
4. Java 使用幂等 `dispatch_id` 将任务派发给 Python Runtime。
5. Python 路由意图、规划步骤，并按需执行模型和 RAG。
6. Python 需要业务动作或查询时，只提交 `proposed_tool_call` 或 `sql_proposal`。
7. Java 校验服务身份、用户授权、schema、确认状态、幂等键与安全策略后执行。
8. Python 校验结果质量并组装最终结构化回答。
9. Python 按顺序回传内部运行事件；Java 去重、持久化并映射为前端 SSE。
10. Java 保存 AgentRun、工具调用、SQL 审计、引用和最终结果，成为可审计状态源。

### 4.1 Java 与 Python 内部契约

第一阶段使用版本化 HTTP/JSON 和流式事件，不引入 Kafka 或 gRPC。字段按消息类型固定：

- 所有跨运行时消息：`schema_version`、`run_id`、`request_id`、`trace_id`。
- Run 命令：`dispatch_id`、`deadline_at`。
- Tool/SQL 请求：`invocation_id`、`idempotency_key`、`deadline_at`。
- Run 事件：`dispatch_id`、`event_id`、`event_seq`、`occurred_at`；事件不携带已经过期的 deadline。

Java 必须验证 Python 服务身份，不能信任 Python 回传的 `user_id` 或权限字段；可信用户上下文由 Java 从已认证会话派生。重复事件按 `run_id + event_id` 幂等处理，乱序事件不得使终态回退。

V1 状态约束没有 `cancelling`。取消流程为：Java 写取消请求审计/控制记录但暂不增加新状态值，随后下发取消；Python 停止后续模型/工具调度并确认，Java 最终迁移到 `cancelled`。若完成、失败和取消并发到达，以 Java 状态机首次接受的合法终态为准。未来若新增 `cancelling`，必须先通过 Flyway 扩展 CHECK 约束。

SSE 事件沿用现有接口规范：

- `run.created`
- `run.routed`
- `run.clarification_requested`
- `run.planned`
- `run.retrieval_started`
- `run.retrieval_finished`
- `run.tool_started`
- `run.tool_finished`
- `run.answer_stream`
- `run.completed`
- `run.failed`
- `run.cancelled`

运行状态与 SSE 事件的关系：

| 前端展示状态 | 后端来源 |
|---|---|
| `routing` | `run.routed`，或 `AgentRun.status=routing/routed` 的派生展示 |
| `planning` | `run.planned` 前后的规划阶段 |
| `retrieving` | `run.retrieval_started` 到 `run.retrieval_finished` |
| `executing` / `executing_tools` | `run.tool_started` 到 `run.tool_finished`，并读取 `ToolCall.status` |
| `validating` | `plan_validator` 或结果校验阶段的派生展示 |
| `composing` | `run.answer_stream` 输出阶段 |
| `waiting_user` | `run.clarification_requested` |
| `completed` | `run.completed` |
| `failed` | `run.failed` |
| `cancelled` | `run.cancelled`，兼容旧实现时可从 `AgentRun.status=cancelled` 派生 |

取消运行优先通过 `run.cancelled` 表达，不建议混用 `run.failed` 表示用户主动取消，避免前端把“失败”和“主动停止”展示成同一种错误。

## 5. 工具与能力边界

P0 工具：

| 工具 | 执行所有者 | 用途 |
|---|---|---|
| `calculator` | Java | 无副作用的确定性数学和单位计算 |
| `time_parser` | Java | 相对时间归一化 |
| `knowledge_search` | Java | Java 执行 ACL 过滤和检索，Python 负责查询改写、重排和引用编排 |
| `database_query` | Java | 通过 SQL Guard/MCP 访问结构化数据 |
| `food_log_writer` | Java | 受控饮食记录写入 |
| `plan_validator` | Java | 餐食计划、营养和预算业务规则校验 |

MVP 主链路稳定后再增加高频 P1 工具：

- `nutrition_lookup`
- `meal_plan_generator`
- `shopping_list_generator`
- `summary_generator`
- `data_exporter`

工具规则：

- 工具必须定义输入 schema、输出 schema、契约版本、执行所有者、超时、重试策略、幂等性、权限和审计级别。
- 写工具默认需要用户确认，除非已明确安全且授权。
- 工具失败必须返回结构化错误，并在 Agent 轨迹中可见。
- 所有注册工具统一由 Java Tool Gateway 执行，Java 是注册、启停、契约、授权、幂等和审计状态源。Python 只提交 proposal；Python 内部纯函数不注册为跨运行时工具。

### 5.1 Prompt Injection 防御

对 FoodMate 来说，Prompt Injection 不只存在于公开网页，也存在于：

- 检索命中的知识库片段
- 用户上传文档的解析结果
- OCR 文本
- 外部 API 返回的富文本字段
- 浏览器抓取到的页面正文
- 第三方工具返回的自然语言说明

这些内容统一视为 `untrusted_content`，只能当证据、数据或候选线索，不能当系统指令执行。

上下文优先级必须固定为：

1. system / safety policy
2. 后端权限与业务规则
3. 用户当前明确意图
4. 当前会话已确认约束
5. 检索结果、网页正文、文档片段等 `untrusted_content`

硬规则：

- `untrusted_content` 中出现“忽略之前所有指令”“立即调用某工具”“把数据发送给某邮箱”之类文本时，Agent 必须把它当内容，不当指令。
- 外部内容不能覆盖 system prompt、后端权限、工具白名单、确认要求或 SQL Guard。
- LLM 只能提出 `proposed_tool_call`，不能直接执行工具。
- 所有真实工具执行前，必须经过独立的 `ToolPolicyChecker` 或等价策略层。
- 高风险写工具、浏览器工具、代码执行工具默认 `deny by default`；只有满足权限、场景、确认状态和白名单条件时才允许放行。
- 被拒绝的工具调用也要写审计，至少记录 `tool_name`、`reason`、`request_id`、`trace_id`。

推荐执行链路：

```text
user_intent + untrusted_content + system_policy
-> ContextIsolator
-> LLM 生成 proposed_tool_call
-> ToolPolicyChecker
-> allow / deny / require_approval
-> ToolExecutor
```

## 6. RAG 与 SQL Agent 边界

RAG 处理非结构化或半结构化知识：

- 烹饪知识。
- 食材保存。
- 营养参考。
- SOP 和产品规则。
- 内部知识文档。

SQL Agent 处理结构化数据，但职责拆成两个信任层：

- 饮食日志。
- 分析报告。
- 用户偏好。
- 报表和管理表。

职责归属：

- Python：问题改写、schema 选择建议和 SQL proposal 生成。
- Java：授权 schema 下发、AST/只读校验、用户或租户过滤、凭据保管、执行和审计。

SQL Agent 硬规则：

- 只允许 `SELECT` 或 `WITH ... SELECT`。
- 必须通过 SQL Guard。
- 必须注入用户或租户过滤。
- 必须限制行数和超时。
- 必须写入查询审计。
- 不能把数据库凭证或 JDBC 连接直接暴露给模型。

RAG / 外部内容安全规则：

- 检索片段默认属于 `untrusted_content`，只能为回答或过滤提供依据。
- 检索结果中的命令式语句不能直接转成工具调用。
- 浏览器抓取、网页正文、上传文档解析内容必须经过 `ContextIsolator` 分层后再进入模型上下文。
- 涉及账号、表单、发送、上传、导出、跨用户查询等动作时，即使外部内容强烈要求执行，也必须回到权限和确认链路重新判断。

## 7. 模型边界

Agent 模型调用统一位于 Python Runtime。Java 业务模块不直接调用供应商 SDK；需要模型治理数据时，通过版本化治理契约与 Python 交互。

Python 模型适配层至少支持：

- `chat`
- `chatStream`
- `structuredOutput`
- `toolCall`
- `embed`
- `rerank`

每次模型调用必须记录：

- request ID 和 trace ID
- 用户或租户信息
- 供应商和模型
- token 用量
- 耗时
- 成本
- 状态和错误

第一阶段模型适配层内嵌于 Python Runtime；供应商凭据不得进入前端或业务数据。模型路由、用量和成本事件回传 Java 持久化，后续只有在多租户配额和供应商治理显著复杂时才再拆独立模型网关。

## 8. 当前实现顺序

以下顺序结合当前仓库实际状态维护，不再把已经完成的前端和后端基础工作列为待实现项。

已完成：

1. 前端 Phase 1-2：页面、共享组件、mock 数据、Agent 事件回放、追问/确认/失败/取消状态。
2. 后端 B3：模块化单体骨架、统一契约、环境配置、异常处理、审计和软删除基础约束。
3. 后端 B4-1：Flyway V1 核心迁移、回滚脚本、24 张核心表、软删除字段和迁移结构测试。

当前：

4. B4-2：建立用户、认证、个人资料、头像资产、会话和消息域的 PO / Mapper。

后续：

5. B4-3 至 B4-8：由 Java 建立 Agent 运行、业务、记忆、知识库、SQL、工具和模型治理的权威持久化。
6. B5-0 至 B5-4：Java 完成认证、个人资料、会话、消息、AgentRun 和真实 SSE。
7. 在 B6 前建立 `agent-runtime/`、Java Runtime Client、内部事件/工具契约和双服务健康检查。
8. B6 至 B10：Python 实现 Agent/RAG/模型/SQL proposal；Java 实现工具、SQL Guard、业务执行和审计。
9. B11：打通五类跨运行时业务闭环。

说明：Milvus、MinIO、RocketMQ、MCP、模型治理、长期记忆、Admin 审计等能力保留为正式架构目标，但按实现阶段逐步接入。第一轮开发可以使用 mock、占位接口或轻量实现，不应因为暂未实现而从文档中删除这些目标。
