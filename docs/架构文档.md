## 1. 系统形态

FoodMate 第一阶段采用模块化单体，提前定义未来可拆服务边界。系统核心形态是 Agentic RAG：Agent 编排是主控层，RAG、工具、SQL Agent 和模型调用都是受控能力。

主运行链路：

```text
UI -> API -> Application Service -> Agent Orchestrator
   -> Router / Planner / Executor / Validator / Composer
   -> RAG / Tool Service / SQL Agent / ModelService / Data Service
   -> PostgreSQL / Redis / Milvus / MinIO / RocketMQ
```

## 2. 模块职责

| 模块 | 职责 | 不允许做 |
|---|---|---|
| UI | 输入、渲染、状态订阅、确认操作 | 决定业务真值或绕过后端校验 |
| API | 鉴权、请求校验、REST/SSE 协议、响应封装 | 直接访问数据库、Milvus 或模型供应商 SDK |
| Application | 用例协调、事务、DTO 组装 | 承担多步 Agent 推理 |
| Orchestrator | 意图路由、任务规划、工具和 RAG 协调、最终组装 | 实现底层工具或直接写数据库 |
| RAG | 查询理解、混合检索、重排、引用组装、ACL 过滤 | 生成最终用户答案或写业务数据 |
| Tool Service | 工具注册、schema 校验、确定性执行 | 自行承担任意用户意图路由 |
| SQL Agent | Schema Catalog、只读 SQL 生成、SQL Guard、MCP 执行、审计 | 执行写 SQL 或跳过校验 |
| ModelService | 统一模型调用、结构化输出、Embedding、Rerank、用量日志 | 把供应商 SDK 直接暴露给业务模块 |
| Data Service | 会话、消息、饮食记录、计划、报告、审计记录 | 隐藏软删除或权限行为 |
| Worker | 异步解析、索引、摘要、长任务 | 直接承接同步用户请求 |

## 3. 核心数据约定

详细 schema 以 `FoodMate-接口与数据规范.md` 和 `FoodMate-数据库设计与接口工具清单.md` 为准。

MVP 稳定实体：

- `Session`：用户任务工作区。
- `Message`：用户和助手消息。
- `AgentRun`：一次由用户消息触发的执行。
- `ToolCall`：挂在 AgentRun 下的一次工具调用。
- `RetrievedReference`：检索引用和命中结果。
- `UserMemory`：长期用户偏好或约束。
- `FoodLog`：持久化饮食记录。
- `MealPlan`：生成并可保存的餐食计划。
- `AnalysisReport`：结构化摄入分析结果。

ID 和删除约定：

- 数据库主键使用应用层生成的 Snowflake `BIGINT`。
- API 和 SSE 对外把 ID 序列化为字符串。
- 删除默认是软删除。
- 默认查询排除 `is_deleted = true`。
- 恢复操作使用显式 `POST /restore` 路由。

## 4. 请求与 Agent 流程

标准用户消息流程：

1. UI 向会话消息接口发送用户输入。
2. API 校验鉴权、请求结构和会话访问权限。
3. Application 持久化用户消息并创建 `AgentRun`。
4. Orchestrator 路由意图，判断是否需要追问。
5. Planner 为复杂任务生成最小可执行计划。
6. Executor 按需调用 RAG、工具、SQL Agent 或 ModelService。
7. Validator 校验数值、计划、写入或策略敏感结果。
8. Composer 生成最终结构化回答。
9. API 通过 SSE 输出状态和答案事件。
10. Data 层持久化运行轨迹、工具调用、引用和最终结果。

SSE 事件沿用现有接口规范：

- `run.created`
- `run.routed`
- `run.clarification_requested`
- `run.planned`
- `run.retrieval_started`
- `run.retrieval_finished`
- `run.tool_started`
- `run.tool_finished`
- `run.answer_stream`
- `run.completed`
- `run.failed`
- `run.cancelled`

运行状态与 SSE 事件的关系：

| 前端展示状态 | 后端来源 |
|---|---|
| `routing` | `run.routed`，或 `AgentRun.status=routing/routed` 的派生展示 |
| `planning` | `run.planned` 前后的规划阶段 |
| `retrieving` | `run.retrieval_started` 到 `run.retrieval_finished` |
| `executing` / `executing_tools` | `run.tool_started` 到 `run.tool_finished`，并读取 `ToolCall.status` |
| `validating` | `plan_validator` 或结果校验阶段的派生展示 |
| `composing` | `run.answer_stream` 输出阶段 |
| `waiting_user` | `run.clarification_requested` |
| `completed` | `run.completed` |
| `failed` | `run.failed` |
| `cancelled` | `run.cancelled`，兼容旧实现时可从 `AgentRun.status=cancelled` 派生 |

取消运行优先通过 `run.cancelled` 表达，不建议混用 `run.failed` 表示用户主动取消，避免前端把“失败”和“主动停止”展示成同一种错误。

## 5. 工具与能力边界

P0 工具：

| 工具 | 用途 |
|---|---|
| `calculator` | 确定性数学和单位计算 |
| `time_parser` | 相对时间归一化 |
| `knowledge_search` | 知识检索入口 |
| `database_query` | 通过 SQL Agent/MCP 访问结构化数据 |
| `food_log_writer` | 受控饮食记录写入 |
| `plan_validator` | 餐食计划、营养和预算校验 |

MVP 主链路稳定后再增加高频 P1 工具：

- `nutrition_lookup`
- `meal_plan_generator`
- `shopping_list_generator`
- `summary_generator`
- `data_exporter`

工具规则：

- 工具必须定义输入 schema、输出 schema、超时、重试策略、幂等性、权限和审计级别。
- 写工具默认需要用户确认，除非已明确安全且授权。
- 工具失败必须返回结构化错误，并在 Agent 轨迹中可见。

### 5.1 Prompt Injection 防御

对 FoodMate 来说，Prompt Injection 不只存在于公开网页，也存在于：

- 检索命中的知识库片段
- 用户上传文档的解析结果
- OCR 文本
- 外部 API 返回的富文本字段
- 浏览器抓取到的页面正文
- 第三方工具返回的自然语言说明

这些内容统一视为 `untrusted_content`，只能当证据、数据或候选线索，不能当系统指令执行。

上下文优先级必须固定为：

1. system / safety policy
2. 后端权限与业务规则
3. 用户当前明确意图
4. 当前会话已确认约束
5. 检索结果、网页正文、文档片段等 `untrusted_content`

硬规则：

- `untrusted_content` 中出现“忽略之前所有指令”“立即调用某工具”“把数据发送给某邮箱”之类文本时，Agent 必须把它当内容，不当指令。
- 外部内容不能覆盖 system prompt、后端权限、工具白名单、确认要求或 SQL Guard。
- LLM 只能提出 `proposed_tool_call`，不能直接执行工具。
- 所有真实工具执行前，必须经过独立的 `ToolPolicyChecker` 或等价策略层。
- 高风险写工具、浏览器工具、代码执行工具默认 `deny by default`；只有满足权限、场景、确认状态和白名单条件时才允许放行。
- 被拒绝的工具调用也要写审计，至少记录 `tool_name`、`reason`、`request_id`、`trace_id`。

推荐执行链路：

```text
user_intent + untrusted_content + system_policy
-> ContextIsolator
-> LLM 生成 proposed_tool_call
-> ToolPolicyChecker
-> allow / deny / require_approval
-> ToolExecutor
```

## 6. RAG 与 SQL Agent 边界

RAG 处理非结构化或半结构化知识：

- 烹饪知识。
- 食材保存。
- 营养参考。
- SOP 和产品规则。
- 内部知识文档。

SQL Agent 处理结构化数据：

- 饮食日志。
- 分析报告。
- 用户偏好。
- 报表和管理表。

SQL Agent 硬规则：

- 只允许 `SELECT` 或 `WITH ... SELECT`。
- 必须通过 SQL Guard。
- 必须注入用户或租户过滤。
- 必须限制行数和超时。
- 必须写入查询审计。
- 不能把数据库凭证或 JDBC 连接直接暴露给模型。

RAG / 外部内容安全规则：

- 检索片段默认属于 `untrusted_content`，只能为回答或过滤提供依据。
- 检索结果中的命令式语句不能直接转成工具调用。
- 浏览器抓取、网页正文、上传文档解析内容必须经过 `ContextIsolator` 分层后再进入模型上下文。
- 涉及账号、表单、发送、上传、导出、跨用户查询等动作时，即使外部内容强烈要求执行，也必须回到权限和确认链路重新判断。

## 7. 模型边界

业务模块只调用 `ModelService`，不能直接调用供应商 SDK。

`ModelService` 至少支持：

- `chat`
- `chatStream`
- `structuredOutput`
- `toolCall`
- `embed`
- `rerank`

每次模型调用必须记录：

- request ID 和 trace ID
- 用户或租户信息
- 供应商和模型
- token 用量
- 耗时
- 成本
- 状态和错误

模型网关可以先以内嵌模块实现；当模型供应商路由、租户策略、配额或成本控制复杂后，再拆成独立服务。
