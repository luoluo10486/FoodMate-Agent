# CookHero 产品需求文档

版本：v1.0  
对应总设计：[CookHero-系统设计与技术方案.md](./CookHero-系统设计与技术方案.md)  
文档目标：把“CookHero Agent 要做什么”定义清楚，确保后续研发、测试、设计、运维都能据此拆分任务和验收。

---

## 1. 背景与目标

### 1.1 背景

CookHero 的目标不是做一个普通聊天机器人，而是做一个面向餐饮、营养、菜单、备餐、饮食记录和分析的任务型智能助手。用户输入的不是单纯问题，而是“要完成的事情”。

### 1.2 产品目标

产品要帮助用户完成以下类型任务：

1. 快速计算
   - 热量换算
   - 食材克重换算
   - 营养估算

2. 记录与查询
   - 记录当天饮食
   - 查询历史饮食
   - 查看某段时间的摄入情况

3. 分析与建议
   - 蛋白质摄入分析
   - 热量趋势分析
   - 营养结构分析

4. 规划与生成
   - 周备餐计划
   - 多人餐食规划
   - 购物清单生成

5. 知识问答
   - 食材保存
   - 烹饪与备餐知识
   - 营养和食材替代建议

### 1.3 产品定义

本产品属于：

- Agent 主导的任务型应用
- Agent + RAG 混合系统
- 以餐饮垂直场景为核心的智能工作台

它不是：

- 纯聊天机器人
- 纯检索问答系统
- 预设流程固定死的工作流机器人

## 2. 用户与场景

### 2.1 目标用户

| 用户类型 | 典型需求 | 关注点 |
|---|---|---|
| 普通用户 | 记录饮食、计算热量、看建议 | 简单、快、少输入 |
| 健身用户 | 控制蛋白质、热量、碳水 | 精准、可量化、可追踪 |
| 备餐用户 | 一周餐单、购物清单、食材复用 | 可执行、低成本、低浪费 |
| 家庭用户 | 多人餐食计划 | 兼顾口味、预算、健康 |
| 运营/管理用户 | 数据统计、趋势分析 | 结构化、可导出、可复盘 |

### 2.2 核心场景

#### 场景 A：快速计算
用户输入“20 克鸡胸肉多少卡路里”，系统完成单位换算和营养估算。

#### 场景 B：饮食记录
用户输入“帮我记录今天午餐”，系统追问缺失信息或直接写入日志。

#### 场景 C：摄入分析
用户输入“分析我最近一周的蛋白质摄入”，系统查询日志并输出趋势分析。

#### 场景 D：计划生成
用户输入“给 2 个人制定一周备餐计划”，系统生成菜单、购物清单和执行建议。

#### 场景 E：知识问答
用户输入“西兰花焯水多久比较合适”，系统从知识库检索并回答。

## 3. 设计原则

### 3.1 工程化原则

1. 优先可落地
   - 每个能力都必须有明确数据来源和执行方式。

2. 优先可观测
   - 每次检索、工具调用、追问、失败都要可追踪。

3. 优先可扩展
   - 工具、知识库、Agent 类型都要可插拔。

4. 优先可校验
   - 关键数值和结构化结果尽量可用工具验证。

5. 优先可回放
   - 每次任务执行路径可复盘，便于调试和评估。

### 3.2 产品原则

1. 不确定就追问，不臆测
2. 复杂任务必须拆解
3. 有依据的回答优先
4. 能用工具就不用模型硬猜
5. 输出要能执行，不只会“建议”

## 4. 范围定义

### 4.1 MVP 范围

MVP 版本优先实现以下能力：

- 会话管理
- Agent 模式对话
- 意图识别
- 基础 RAG 检索
- 单步工具调用
- 食材/营养计算
- 饮食记录写入
- 饮食记录查询
- 基础分析结果
- 结果引用展示

### 4.2 二期范围

- 多步规划
- 复杂任务分解
- 购物清单生成
- 周计划生成
- 用户偏好记忆
- 结果导出
- 多 Agent 协作

### 4.3 暂不做

- 通用开放域搜索引擎
- 任意第三方平台插件市场
- 复杂多人协作编辑
- 无审计的自动写入
- 纯自由聊天娱乐功能

## 5. 功能需求

### 5.1 会话与入口

#### 功能描述
用户进入页面后可以：

- 新建 Agent Session
- 查看最近会话
- 继续历史任务
- 选择不同能力入口

#### 需求点

- 支持新建会话
- 支持会话列表分页加载
- 支持会话命名
- 支持会话置顶和搜索

#### 验收标准

- 能创建新会话并写入数据库
- 刷新页面后会话仍可恢复
- 可以查看最近历史记录

### 5.2 意图识别

#### 功能描述
系统收到用户输入后，需判断任务类型并决定处理路径。

#### 意图类别

| 意图 | 示例 | 处理方式 |
|---|---|---|
| 计算 | 20 克鸡胸肉多少卡 | 工具 + 规则 |
| 记录 | 帮我记录午餐 | 写入流程 |
| 分析 | 最近一周蛋白质摄入 | 查询 + 分析 |
| 规划 | 做一周备餐计划 | 多步规划 |
| 问答 | 西兰花怎么焯水 | RAG 检索 |
| 混合任务 | 先分析再规划 | Agent 编排 |

#### 验收标准

- 识别结果有置信度
- 能识别混合任务
- 低置信度时会追问

### 5.3 追问与澄清

#### 功能描述
当任务缺少关键参数时，系统不能直接猜测，必须追问。

#### 常见缺参

- 时间范围
- 食材名称
- 份量
- 人数
- 预算
- 忌口
- 目标类型

#### 验收标准

- 关键参数缺失时，系统只追问必要问题
- 追问问题必须短、明确、可回答

### 5.4 RAG 知识检索

#### 功能描述
系统可从知识库中检索餐饮、营养、操作类知识，并在回答中引用。

#### 需求点

- 支持文档切分
- 支持向量检索
- 支持关键词召回
- 支持重排序
- 支持引用片段展示

#### 验收标准

- 回答可附来源
- 检索失败时不伪造引用
- 检索结果可回放

### 5.5 工具调用

#### 功能描述
Agent 能调用确定性工具完成数值计算、营养查询、日志写入、计划生成等任务。

#### 必备工具

| 工具 | 作用 |
|---|---|
| calculator | 精确数值计算 |
| unit_converter | 单位换算 |
| nutrition_lookup | 营养信息查询 |
| food_log_writer | 饮食记录写入 |
| food_log_query | 饮食记录查询 |
| meal_plan_generator | 备餐计划生成 |
| shopping_list_generator | 购物清单生成 |

#### 验收标准

- 工具调用参数符合 schema
- 工具失败可重试
- 工具结果可审计

### 5.6 数据分析

#### 功能描述
系统可对用户饮食数据进行统计分析并输出建议。

#### 分析维度

- 热量趋势
- 蛋白质摄入趋势
- 碳水/脂肪占比
- 连续性与稳定性
- 目标达成率

#### 验收标准

- 结果包含统计口径
- 结果能说明数据来源
- 结果可导出

### 5.7 复杂规划

#### 功能描述
系统可基于约束生成可执行方案，例如一周菜单、购物清单、备餐步骤。

#### 需要支持的约束

- 人数
- 预算
- 目标
- 忌口
- 厨房设备
- 时间成本
- 食材复用

#### 验收标准

- 方案分步骤
- 方案可执行
- 方案包含约束说明

### 5.8 用户偏好记忆

#### 功能描述
系统可记住用户长期偏好，但必须可控、可查看、可修改。

#### 记忆项

- 饮食偏好
- 忌口
- 常用单位
- 常见目标
- 常用餐型

#### 验收标准

- 用户可查看并修改记忆
- 记忆不会污染当前会话

## 6. 页面与信息架构

### 6.1 页面组成

| 区域 | 作用 |
|---|---|
| 左侧栏 | 会话列表、历史入口、新建按钮 |
| 顶部栏 | 模块切换、用户入口、全局功能 |
| 主区域 | Agent 说明、推荐能力、快捷任务 |

### 6.2 首页应展示

- 产品 Logo
- 当前模式
- 常用能力卡片
- 示例任务按钮
- 输入框
- 工具状态
- Agent 状态

### 6.3 结果展示方式

输出建议分层：

1. 结论摘要
2. 过程说明
3. 数据来源
4. 可执行动作
5. 异常提示

## 7. 交互流程

### 7.1 标准流程

1. 用户输入任务
2. 系统识别意图
3. 判断是否缺参
4. 判断是否需要 RAG
5. 判断是否需要工具
6. 执行检索/工具
7. 校验结果
8. 输出回答

### 7.2 追问流程

1. 识别缺失信息
2. 生成一个或两个短问题
3. 等待用户补充
4. 回到原任务上下文继续执行

### 7.3 失败流程

1. 工具报错或检索为空
2. 系统记录失败原因
3. 尝试重试或降级
4. 无法继续时向用户透明说明

## 8. 质量与非功能需求

### 8.1 性能指标

| 指标 | 建议目标 |
|---|---|
| 首屏可交互 | <= 2s |
| 普通回答首 token | <= 3s |
| 单次工具执行 | 可控在 1-5s |
| 常规任务完成 | <= 15s |
| 长任务完成 | 支持异步回传 |

### 8.2 可靠性

- 单工具失败不影响全局服务
- 任务状态可恢复
- 结果可重试
- 异常可降级

### 8.3 安全性

- 用户数据写入前要确认
- 不确定数据不能编造
- 敏感操作要审计
- 关键接口要鉴权

### 8.4 可观测性

必须记录：

- 用户输入
- 意图识别结果
- 检索命中
- 工具调用
- 执行耗时
- 回答结构
- 失败原因

## 9. 分期计划

### 9.1 第一阶段

目标：让 Agent 跑起来。

- 会话
- 输入输出
- 意图识别
- 基础工具
- 基础检索

### 9.2 第二阶段

目标：让业务可用。

- 记录与查询
- 分析能力
- 计划生成
- 引用展示

### 9.3 第三阶段

目标：让 Agent 更聪明。

- 多步规划
- 记忆
- 复杂任务编排
- 结果校验

### 9.4 第四阶段

目标：平台化。

- 多 Agent
- 工具扩展
- 评测体系
- 监控与回放

## 10. 验收标准

### 10.1 功能验收

- 核心场景可完成
- 工具调用正常
- 检索结果可解释
- 追问合理
- 结果结构化

### 10.2 体验验收

- 用户理解当前状态
- 用户知道下一步
- 用户知道结果依据
- 页面布局清晰

### 10.3 工程验收

- 接口有版本控制
- 数据结构稳定
- 日志可追踪
- 异常可定位

## 11. 风险与对策

### 11.1 风险：Agent 太泛
对策：先聚焦餐饮垂直场景，避免通用化过早。

### 11.2 风险：LLM 直接胡猜
对策：数值和写入类任务强制走工具与校验。

### 11.3 风险：上下文过长
对策：会话摘要、分层记忆、短期上下文窗口化。

### 11.4 风险：无法评估
对策：建立任务集、打点、回放、离线评测。
