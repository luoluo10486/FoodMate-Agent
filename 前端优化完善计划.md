## 任务清单（按执行顺序）

### 🥇 第 1 步：修复重复路由（App.tsx + ChatPage）

**问题**: App.tsx 中 `/chat/:session_id?` 和 `/chat/:sessionId?` 两条路由指向同一页面。ChatPage 内部用 `params.session_id ?? params.sessionId` 兼容。

**修改文件**：
- `frontend/src/App.tsx` → 删除 `/chat/:sessionId?` 路由行
- `frontend/src/pages/ChatPage/ChatPage.tsx` → 第 18 行简化为 `params.session_id`

---

### 🥇 第 2 步：消除死代码

**问题**: 4 个导出符号从未被导入使用。

**修改文件**：
- `frontend/src/mock/auth.ts` → 删除 `publicAuthCapabilities`
- `frontend/src/mock/sessions.ts` → 删除 `chatMessages`、`planningRun`
- `frontend/src/types/ui.ts` → 删除 `PageState` 类型

---

### 🥇 第 3 步：GSAP 插件注册统一

**问题**: `gsap.registerPlugin(useGSAP)` 在 3 个组件中重复调用，main.tsx 无注册。

**修改文件**：
- `frontend/src/main.tsx` → 添加 `gsap.registerPlugin(useGSAP)` import + 调用
- `frontend/src/components/brand/BrandLogo.tsx` → 删除注册
- `frontend/src/pages/LoginPage/LoginPage.tsx` → 删除注册
- `frontend/src/pages/AnalysisPage/AnalysisPage.tsx` → 删除注册

---

### 🥈 第 4 步：拆分 agentReplay.ts（857 行 → 3 文件）

**问题**: 状态机逻辑 + mock 数据混在一个文件。

**方案**：
- `mock/agentReplay.ts` — hook 入口 + 类型导出（~200 行）
- `mock/agentReplayData.ts` — 各场景 mock 事件序列数据（~300 行）
- `mock/agentReplayMachine.ts` — 事件调度状态机逻辑（~300 行）

---

### 🥈 第 5 步：拆分 AdminPage.tsx（930 行 → 主文件 + 8 tabs）

**问题**: 包含 23 个内联组件/函数，单文件过大。

**方案**：
- `AdminPage.tsx` — 主页面骨架 + 菜单（~100 行）
- `tabs/OverviewTab.tsx` — 概览
- `tabs/UsersTab.tsx` — 用户管理
- `tabs/RunsTab.tsx` — Agent 运行
- `tabs/ToolsTab.tsx` — 工具管理
- `tabs/UsageTab.tsx` — 模型用量
- `tabs/KnowledgeTab.tsx` — 知识库
- `tabs/DeletedResourcesTab.tsx` — 软删除资源

---

### 🥈 第 6 步：建立路由常量

**问题**: 代码中多处硬编码 `/chat/`、`/planning`、`/admin` 等路径字符串。

**方案**：新建 `frontend/src/constants/routes.ts`，定义 ROUTES 常量和 `buildChatPath()`，替换所有硬编码路径。

---

### 🥉 第 7 步：建立前端服务层（src/services/）

**问题**: 所有数据来自 mock，无 API 抽象层，后端接入时每个组件都要改。

**方案**：新建 `frontend/src/services/`，按业务域拆分 7+ 个文件。当前对接 mock，接口签名与未来后端 API 一致。后端就绪后只需替换 `apiClient` 实现。

---

### 🥉 第 8 步：添加开发工具链

- ESLint + Prettier + Vitest
- 配置基础 lint 规则
- `package.json` 添加 `lint`、`format`、`test` 脚本

---

### 📝 第 9 步：更新 TODO 清单

- `待办任务TODO.md` 顶部加入 "第一版 MVP 最短路径" 指引小节

## 执行顺序总结

```
前 3 步（低风险快速修复）→ 中间 2 步（文件拆分）→ 后 3 步（架构改进）→ 文档
```

- 前 3 步改动量极小，5 分钟内搞定
- 第 4-5 步是主力工作，需要仔细检查 import 路径
- 第 7 步服务层需要先确认接口设计，建议做到那一步时再讨论
